## 2. 权威签名贴合（Contract #2）

- **§5.6 `SaveManager` interface**：1:1 实现 `saveRun / savePlayer / saveDungeon / saveEnemy / saveSeed / loadRun`，无增删方法。
- **§5.7 `ReplayRecorder` + `ReplayPlayer`**：`header` 字段 + `record(frame, cmd)` / `play(header, frames)` 形状一致；`ReplayPlayer.play` 确定性重演。
- **§5.5 DebugPanel**：计划仅给面板布局（无 interface 签名），按布局建模 `DebugSnapshot` 12 类；不臆造新接口。

## 3. 关键设计决策

1. **令牌复用**：`IDebugService / ISaveManager / IReplayRecorder` 已在 `GameContext.ts` 声明（§5.x 服务清单），直接复用，**未新增令牌**（Contract #2）。
2. **Logger 缓冲转发**：`Logger`（Demo0）无缓冲，故在 GameBootstrap 把 `Logger` 的 `sink` 改为 `console.log` + `debugPanel.pushRaw(line)`，`DebugPanel` 用正则解析 `[time][channel][level] msg` 喂入 Events 环缓冲——即「ILogger 缓冲」数据源（§5.5），无需改 Logger。
3. **相机面板鸭子探测**：§5.5 列 `ICameraBrain.getDebugState()` 为数据源，但 Demo2 的 `CameraBrain` 未实现该方法。按「不擅自扩大修改范围」铁律，**不修改 CameraBrain**；`DebugPanel.initialize` 用 `try/get` + `typeof getDebugState === 'function'` 探测，未来 CameraBrain 实现后自动接入。
4. **SaveBackend 注入**：`SaveManagerImpl` 构造注入 `SaveBackend`（纯 TS 抽象）；测试用 `MemorySaveBackend`，引擎层在 GameBootstrap 可换 `localStorage` 后端。保持纯 TS 可 node 单测（红线相关）。
5. **Replay 确定性**：`computeConfigHash` 用 FNV-1a（无 `Math.random`，红线5）；`ReplayPlayer.play` 按序重发帧流，单测以「同种子 + 同输入 → 两次重演回调序列完全相等」直接证明确定性。
6. **环形缓冲**：`ReplayRecorder` 保留最近 5 局（`_maxRuns`），单局帧上限 100 万；`endRun` 入环、溢出丢弃最旧。
