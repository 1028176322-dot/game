## 0. 总纲与决策

### 0.1 执行顺序（硬性，不可跳步）

```
Demo0 基础设施地基
  → Demo1 模型加载 → Demo2 相机 → Demo3 碰撞
  → Demo4 技能 → Demo5 房间 → Demo6 100怪性能
  → 垂直切片(VS)
  → Phase1-6 正式开发
```

每个 Demo 全部通过后才进入下一个；禁止跨 Demo 预实现。

### 0.2 推荐起点：Demo0 地基（首周）

只做 `core/` 下 4 个文件 + 接入 `GameBootstrap` + 建立单测基座（详见 §2 索引与 `docs/ai-tasks/demo0/`）。

### 0.3 架构红线（写入 CI 门禁，任何任务都不得违反）

1. 业务禁止 `import PhysicsSystem`，只允许依赖 `ICollisionService`
2. 技能禁止 `switch(skillId)`，必须走 `SkillGraph`
3. 运行时 System 必须 `implements ILifecycle` 并由 `LifecycleManager` 管理
4. 业务禁止 `new` 服务，必须 `ctx.get(Token)`
5. 业务禁止 `Math.random()`，必须走 `RunRng`
6. 仅 `SceneFlowService` 可 `director.loadScene()`

> 若实现中发现某条红线与"输入"章节冲突，**停止并报告**（见 §1.10 铁律 5）。

### 0.4 文档引用约定

- 章节：`§5.2` = `docs/2D转3D全面升级方案.md` 第 5.2 节
- 图示：`图6.3` = 同文件第 6.3 节生命周期状态图
- 接口：`§5.1 接口 ILifecycle` = 该节给出的 TS 签名，为**唯一权威签名**
- 冲突时以 v3 设计文档为准；但 AI 发现冲突须**停止报告**，不得自行修正计划或设计。

## 1. AI 执行协议（所有 Demo 通用，必须遵守）

### 1.1 固定 8 步 Pipeline（每个任务都按此执行）

```
Step0   扫描工程   : 扫描 package.json / tsconfig / 目录结构 / 现有 core / Config / Service / 接口
                     → 生成或刷新 docs/ai-tasks/_architecture_report.md（含"既有实现清单"）
Step1   阅读+查重  : 读任务卡"输入"章节；并先全局搜索是否已存在同名/同职责实现（禁止重复实现）
Step2   修改前Diff : 列出 新增/修改/删除 文件与符号；确认 ⊆ 允许范围；输出影响统计
Step2.5 执行前确认 : 展示 Execution Plan，等待 "Plan Approved" 后才写代码（见 §1.4）
Step3   生成代码   : 按"输出"签名实现；单任务 ≤3~5 文件（Token Budget，见 §1.5）
Step4   运行测试   : npm run test
Step5   修复错误   : 仅修本任务错误（≤3 轮）；设计冲突 → 停止（见 §1.7）
Step6   验证       : npm run validate:all，9 门禁全过
Step7   提交       : [Demo0][D0-x] 格式提交
Step8   Checkpoint : 生成 REPORT_<task>.md（完成/测试/Validate/新增/修改/风险，见 §1.6）
```

### 1.2 任务卡字段规范（每个任务必须包含）

| 字段 | 含义 | 示例 |
|---|---|---|
| **输入** | 参考的文档/接口/章节 + 查重对象 | `§5.2` + 搜索是否已有 ServiceLocator |
| **输出** | 新增/修改哪些文件、必须导出哪些符号 | `core/GameContext.ts` 导出 `GameContext` |
| **严格约束** | 禁止自由发挥：不得新增接口/改名/加职责/重复实现 | 严格按 §5.2，且不得重写现有 ConfigManager |
| **允许修改范围** | 本任务可动的目录/文件 | 允许新增 `core/GameContext.ts` |
| **禁止修改范围** | 本任务绝不可动的目录/文件 | 禁止修改 `battle/**` `config/**` |
| **完成定义(DoD)** | 何时算结束，必须可验证 | ✓ `npm run test` ✓ `validate:all` ✓ 空场景打印序列 |

### 1.3 禁止重复实现（Anti-Duplication）★v3 新增

- 写任何模块前，先全局搜索同名/同职责实现：`Database` / `Repository` / `DataCenter` / `ConfigManager` / `ServiceLocator` / `Logger` / `Lifecycle` 等。
- 若已存在**同职责**实现 → **停止并报告**，复用而非重写。
- 关键实例（见 `_architecture_report.md`）：
  - `ConfigDatabase` 必须复用现有 `ConfigService` + `ConfigManager`（config 加载已存在），**不得再写一套加载器**。
  - `GameContext` 是新增 ServiceLocator，但不得重实现 `GameManager`/`ConfigManager` 的既有单例管理能力——只做服务注册/获取。
- 查重命令参考：`grep -rn "class XxxDatabase\|interface IXxxService\|function useXxx" assets/scripts`。

### 1.4 修改前 Diff 分析 + 执行前确认（Execution Plan）★v3 新增

- **Step2 修改前 Diff**：输出将要做的变更清单，AI 自确认影响范围：
  ```
  准备新增: GameContext.ts
  准备修改: 无
  准备删除: 无
  影响: 1 file | 新增 1 | 修改 0 | 删除 0
  ```
- **Step2.5 执行前确认（Execution Plan）**：写代码前先输出计划并**等待 `Plan Approved`**：
  ```
  Execution Plan — 本次任务: D0-2
  将新增: assets/scripts/core/GameContext.ts
  将修改: 无
  预计影响: 1 file
  不会修改: battle/ ui/ scene/ config/ assets/(服务) app/ run/ utils/ (禁止范围)
  预计步骤:
    1. 读 §5.2，确认签名
    2. 搜索确认无既有 ServiceLocator（禁止重复实现）
    3. 写 GameContext.ts（导出 GameContext + 13 令牌）
    4. 写 tests/core/gamecontext.test.ts
    5. npm run test → npm run validate:all
  等待: Plan Approved
  ```
- 人工参与模式：输出 Plan 后**停止**，等待用户回 `Plan Approved` 再继续。
- 自动模式：设 `AUTO_APPROVE=false` 默认等待；仅在显式开启时跳过等待。

### 1.5 Token Budget ★v3 新增

- 单任务 ≤ **3~5 个文件**（Claude Code 官方建议）。超出 → 拆为子任务，不得一次吞下整个 Demo。
- 每个 Demo 拆为独立任务文件（见 §2），每个文件 = **一个 Prompt + 一个输出 + 一个 DoD**，AI 逐文件读取执行，上下文不会越滚越长。
- Demo0 已拆 6 文件（D0-0~D0-5）；Demo1-6 各一个骨架文件，启动前按 §1.2 补全详细卡。

### 1.6 Checkpoint 报告（REPORT.md）★v3 新增

- 每个任务完成（Step8）后生成 `docs/ai-tasks/<demo>/REPORT_<task>.md`：
  ```
  # REPORT D0-2
  完成: ✓
  测试: ✓ (vitest 全绿)
  Validate: ✓ (9 门禁)
  新增文件: 1 (GameContext.ts)
  修改文件: 0
  风险: 无
  ```
- 价值：断点恢复。下次会话直接读最新 `REPORT_*.md` 即可续作，无需重读全部任务卡。

### 1.7 失败处理规则（红线级）

- 类型错误 / 单测失败（本任务代码引起）：自动修复，最多 3 轮；仍失败 → 停止并输出 diff 与错误原文。
- `validate:all` 失败：逐门禁修复直到全过；若暴露**设计冲突**（如 architecture 红线）→ 停止，不绕过门禁、不注释检查。
- 依赖冲突（需改范围外文件 / 改现有接口签名）：**立即停止**，输出原因，等待人工确认。严禁自行改设计或扩大范围。
- **严禁**：注释掉测试、跳过 validate、改 `.gitignore` 绕过、降级门禁、为通过而删断言、关闭 Lint。

### 1.8 范围约定（写法）

- `允许新增: path` / `允许修改: path（仅限 …）` / `禁止修改: path/**`（含递归子目录）。未列出的目录默认**禁止修改**。

### 1.9 提交与门禁规则

- 每个 Demo 一个 feature 分支：`feature/3d-demo0` …；commit message `[Demo0][D0-2] implement GameContext (§5.2)`。
- 禁止多 Demo 混 commit、禁止夹带范围外修改；门禁全过后 PR 合回 `master`（人审）。

### 1.10 Agent Contract（10 条铁律）

AI 每次执行前**必须加载并遵守** `docs/ai-tasks/_agent_contract.md`：

1. 禁止修改计划（本规范与任务卡）
2. 禁止修改接口（签名/命名/职责，以 §5.x 为准）
3. 禁止新增职责（不得给模块加超出接口的事）
4. 禁止猜测需求（缺信息就问/停，不脑补）
5. 发现冲突立即停止（设计/红线/范围冲突 → 停）
6. 任何超出范围必须报告（不私自扩大修改面）
7. 不能为了通过测试修改测试（改实现，不改断言）
8. 不能关闭 Lint
9. 不能关闭 Validate
10. 不能降低架构约束（红线 1–6 不可放宽）

## 4. 垂直切片（VS · 最关键一条）

Demo0-6 全绿后做「最小可玩闭环」：**先做到可玩再复制扩张**（一个角色+一种怪+一个技能+一个 Boss+一个房间+完整 UI+完整流程+调试/存档/重放）。切片暴露的坑修复成本 << 全量后才发现。VS 同样按 §1 协议，允许范围扩大到纵向串联，但**禁止改动 Demo0-6 已固化的 `core/` 接口签名**。

## 6. 版本迁移与门禁

### 6.1 分支策略

- `master` 长稳，始终可运行 2D 版本
- 功能按 `feature/3d-*` 分支（foundation / render / animation / combat / dungeon / ai）
- 冻结 2D：垂直切片通过、全量复制开始时标 `@deprecated`
- 删除 2D：Phase 6 末，门禁全绿后删 `Sprite`/`SpriteAnimationService`/旧 `BattleManager`

### 6.2 验收门禁（每阶段）

- `npm run validate:all`（9 gates）
- 3D 资产门禁（面数/贴图/骨骼/动画/原点朝向，与 §5.9 共用）
- 架构门禁（§8.3 红线 1–6，静态检查；AI 触红线即失败，见 §1.7/§1.10）
- 性能门禁（Demo6 起真机：DrawCall/内存/帧率）
- 测试门禁（§5.11 单测 + §5.12 分支门禁，PR 必过）
