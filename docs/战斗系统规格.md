## 0. 设计师笔记

### 0.1 玩家体验目标

```
进入房间 → 观察: 敌人位置? 地形? 我有什么资源?
   ↓
分配 AP: 我是进攻? 周旋? 找回复?
   ↓
执行行动 → 观察到结果
   ↓
等待怪物行动 → 判断是否需要调整策略
   ↓
循环直到房间清空 → 获得奖励
```

**核心感受**：每回合都是一个"资源分配谜题"，而不是"点到最厉害的技能然后跳过回合"。

### 0.2 好玩假设

> **"AP 分配 + 地形 + 元素反应 = 每回合都有值得决策的选择"**

这个假设必须在纯文字原型中得到验证。如果失败，回退到简化版本（2AP 固定+攻击+防御）。

### 0.3 工程师提示

本文档中的"实现方案"部分为建议实现方式。设计需求以"规格"为准，实现方式可以有更优解。如有疑问优先问"这个规格澄清了什么体验"而不是"这个实现是不是最优"。

## 2. AP 机制 (Action Point System)

### 2.1 规格

**Purpose**: 让每回合的"我能做什么"成为一个资源分配决策，而不是"移动然后攻击"的固定流程。

**Player Fantasy**: 我是战术指挥官——我决定这回合是全力输出、稳扎稳打、还是战略性等待。

**Input**: 每回合开始时自动获得 3 AP。玩家点击行动按钮消耗 AP。

**Output**: AP 减少，对应行动执行。

**Success Condition**: 玩家总能完整用完或保留 AP，不会因为系统原因丢失 AP。

**Failure State**: 玩家有 AP 但无法执行任何有意义的行动（被困住/所有敌人免疫当前攻击方式）。

**Edge Cases**:
- 什么都不敢做也不想等待 → 系统自动等待(0AP)，跳过该回合
- 在移动过程中触发地形效果 → 先处理地形，再扣除移动 AP
- 玩家只有 1 AP，想用 2 AP 的技能 → 提示"AP 不足"，保留当前 AP

**Tuning Levers**:
- AP 基础值 (当前: 3) → 影响整体战斗节奏
- 保留 AP 上限 (当前: 1) → 影响蓄力策略的价值
- 各行动的 AP 消耗 → 调整行动的相对价值

### 2.2 实现方案

```
类: ActionPointSystem

属性:
- currentAP: int = 3    // 当前可用 AP
- maxAP: int = 3        // 永久上限
- retainedAP: int = 0   // 从上一回合保留的 AP（不可再保留）

行为:
- onTurnStart(): 
    currentAP = maxAP + retainedAP
    retainedAP = 0
- onActionCost(cost):
    if currentAP < cost: return false  // AP 不足
    currentAP -= cost
    return true
- onTurnEnd():
    retainedAP = min(currentAP, 1)      // 最多保留 1 点
    currentAP = 0
- onPermanentUpgrade():
    maxAP += 1
```

## 3. 行动规格 (Actions)

### 3.1 行动总表

| 行动 | AP | 范围 | 效果 | 冷却 |
|------|----|------|------|------|
| 移动 (Move) | 1/格 | Move 上限内 | 角色位移 1 格 | 无 |
| 普攻 (Basic Attack) | 1 | Range | 对目标造成基础伤害 | 无 |
| 技能 (Skill Attack) | 2 | Range 或特殊 | 附带元素/特效的攻击 | 无 |
| 防御 (Defend) | 1 | 自身 | 本回合 DEF +3 | 无 |
| 道具 (Use Item) | 1 | 取决于道具 | 使用消耗品 | 道具本身限制 |
| 等待 (Wait) | 0 | 无 | 跳过，保留 AP | 无 |

### 3.2 Mechanic: 移动 (Movement)

```
Mechanic: 移动

Purpose: 玩家在网格上重新定位，获取战术优势
Player Fantasy: "我站在这里是因为我想站在这里"
Input: 点击/滑动到目标格子
Output: 玩家角色移动到目标格子
Success Condition: 角色移动到指定格子，AP 减少
Failure State: 目标格子被占据或不可通行 → 显示红色提示，不消耗 AP

Edge Cases:
  - 目标格子距离 > Move 上限 → 不可到达，提示"超出移动范围"
  - 目标格子被敌人占据 → 不可通过（除非拥有"穿影"能力）
  - 移动路径经过荆棘地板 → 每经过 1 格荆棘扣 1 HP
  - 经过高台 → 需要 2 AP（爬上去），不是按格子计费
  
Tuning Levers:
  - Move 上限 (当前: 3)
  - 移动 AP 消耗 (当前: 1/格)
  - 地形移动修正 (高台: 2/格)

Dependencies: 网格系统, 地形系统
```

### 3.3 Mechanic: 普攻 (Basic Attack)

```
Mechanic: 普攻

Purpose: 基础伤害输出，没有花哨效果，稳定可靠
Player Fantasy: "稳定的一刀"
Input: 点击在 Range 内的目标敌人
Output: 对目标造成 `ATK + d6 - 目标DEF×0.5` 伤害
Success Condition: 目标 HP 减少, 扣 1 AP
Failure State: 目标不在 Range 内 → 提示"攻击范围外"

Edge Cases:
  - 骰子结果 = 6 → 暴击，伤害 ×1.5
  - 骰子结果 = 1 → 普通命中（不额外惩罚）
  - 伤害 < 1 → 取 1 (最小伤害保底)
  - 武器上有元素附魔 → 攻击同时附加元素状态
  - 目标 HP 降为 0 → 目标死亡，触发击杀事件

Tuning Levers:
  - 基础 ATK
  - 骰子范围 (当前: d6)
  - 暴击条件 (当前: 骰子=6)
  - 暴击倍率 (当前: 1.5×)
  - DEF 减伤系数 (当前: 0.5)

Dependencies: 伤害公式, 元素系统（如果武器有附魔）
```

### 3.4 Mechanic: 技能攻击 (Skill Attack)

```
Mechanic: 技能攻击

Purpose: 消耗更多 AP 换取更强的攻击效果（元素/控制/范围）
Player Fantasy: "这一击会有所不同"
Input: 选择技能 → 选择目标
Output: 对目标造成伤害 + 技能附带效果
Success Condition: 技能效果正常触发
Failure State: 目标不适合该技能（如范围技能打单体目标）→ 但仍可释放,浪费 AP

Edge Cases:
  - 技能附带元素 → 目标身上已有元素 → 触发元素反应
  - 当前没有可用技能 → 按钮置灰，提示"没有技能"
  - 技能目标是范围 AoE → 选择中心格子，而非单体目标聚焦
  - 技能被闪避/免疫 → 消耗 AP, 无效果（特殊怪物属性）

Tuning Levers:
  - 技能 AP 消耗 (当前: 2)
  - 技能效果数值
  - 元素附带概率 (当前: 100%)

Dependencies: 核心能力系统（技能来源）, 元素系统
```

### 3.5 Mechanic: 防御 (Defend)

```
Mechanic: 防御

Purpose: 牺牲攻击力换取生存力，在危险回合保全自己
Player Fantasy: "放马过来吧"
Input: 点击防御按钮
Output: 本回合 DEF +3，效果持续到本回合结束
Success Condition: 防御姿态下受到攻击时，伤害显著降低
Failure State: 防御姿态期间没有受到攻击 → AP 浪费，但策略上可能不想浪费

Edge Cases:
  - 防御姿态下被暴击 → 暴击伤害在减伤后计算
  - 防御叠加 → 连续两回合防御，效果不叠加，重新计时
  - 拥有"盾反"核心能力 → 防御姿态受击时反弹 50% 伤害
  - 拥有"怒吼"核心能力 → 防御姿态变为攻击姿态，ATK+2, DEF-1

Tuning Levers:
  - DEF 加成值 (当前: +3)
  - 持续时间 (当前: 本回合)
  - 是否可取消 (建议: 不可取消，选择即锁定)

Dependencies: 核心能力系统（盾反/怒吼等修改防御行为）
```

### 3.6 Mechanic: 道具 (Use Item)

```
Mechanic: 道具

Purpose: 消耗资源换取战斗优势——药水回血、临时强化、改变局势
Player Fantasy: "是时候用掉它了"
Input: 打开道具栏 → 选择道具 → 使用（或选择目标）
Output: 道具效果生效，道具从背包中移除，消耗 1 AP
Success Condition: 道具效果正常触发
Failure State: 道具栏为空 → 按钮置灰

Edge Cases:
  - 回血道具在满 HP 时使用 → 提示"HP 已满"，不消耗道具和 AP
  - 道具使用后背包清空 → 道具栏变为空
  - 道具选择后反悔 → 提供取消按钮（不消耗 AP）
  - 道具需要选择目标（如投掷药水）→ 目标选择流程同攻击

Tuning Levers:
  - 道具价格
  - 道具效果数值
  - 道具使用 AP 消耗 (当前: 1)

Dependencies: 背包系统, 商店系统
```

### 3.7 Mechanic: 等待 (Wait)

```
Mechanic: 等待

Purpose: 不消耗 AP 跳过一个回合，保留 AP 到下一轮实现蓄力
Player Fantasy: "隐藏实力"
Input: 点击结束回合（未消耗 AP）
Output: 当前回合直接结束，本回合所有 AP 保留（最多 1 点）
Success Condition: 回合结束，下回合获得保留 AP
Failure State: 不适用（总是成功）

Edge Cases:
  - 连续等待 → 每次保留 1 点，但不能叠加（下回合只有额外 1 点）
  - 等待后下回合直接用 4 AP 爆发 → 有效策略
  - 玩家什么都不做时间耗尽 → 系统自动执行等待

Tuning Levers:
  - 保留 AP 上限 (当前: 1)

Dependencies: AP 系统
```

## 4. 伤害公式规格

### 4.1 公式

```
原始伤害 = ATK(攻击者) + 骰子(1~6)
减伤值  = floor(DEF(目标) × 0.5)
实际伤害 = max(原始伤害 - 减伤值, 1)

// 暴击判定
if (骰子结果 == 6):
    实际伤害 = floor(实际伤害 × 1.5)
```

### 4.2 验证数据

| 场景 | ATK | DEF | 骰子 | 原始伤害 | 减伤 | 实际伤害 | 暴击? |
|------|-----|-----|------|---------|------|---------|-------|
| 玩家打 1 层小怪 | 5 | 0 | 3 | 8 | 0 | 8 | 否 |
| 玩家打 1 层小怪 (暴击) | 5 | 0 | 6 | 11 | 0 | 16 | 是 |
| 玩家打 2 层骷髅 | 7 | 1 | 4 | 11 | 0 | 11 | 否 |
| 玩家打 3 层 BOSS | 9 | 2 | 5 | 14 | 1 | 13 | 否 |
| 小怪打 2 层玩家 | 4 | 2 | 3 | 7 | 1 | 6 | 否 |
| BOSS 打 3 层玩家 | 7 | 3 | 4 | 11 | 1 | 10 | 否 |
| 高 DEF 减伤极限 | 3 | 10 | 3 | 6 | 5 | 1 (保底) | 否 |

### 4.3 最小伤害保底的意义

最小伤害为 1 确保：**永远不会出现"完全打不动"的情况**。即使玩家 ATK=1, 怪物 DEF=10, 骰子=1，依然能造成 1 点伤害。

这在心理上很重要——**每一刀都在削血**。一个永远不掉血的敌人会让玩家感到绝望。

### 4.4 [HYPOTHESIS] 验证要点

| 假设 | 验证方法 | 通过标准 |
|------|----------|----------|
| 暴击 1.5× 足够刺激 | 统计玩家暴击时的情绪反应 | 暴击时有明显的正面反馈 |
| DEF 半值减伤不鸡肋 | 高 DEF 玩家 vs 低 ATK 怪物 | 减伤效果可感知但不绝对 |
| 保底 1 伤害不破坏平衡 | 高 DEF 怪物 vs 低 ATK 玩家 | 需要约 15~20 次攻击才能击杀 |

## 5. 回合流程规格

### 5.1 完整回合时序

```
================= 开始战斗 =================

1. 进入房间
   a. 生成怪物（调用怪物装配系统）
   b. 放置地形（房间预设布局）
   c. 玩家进入入口格子
   d. 显示敌人信息（HP 条/类型图标）

2. 战斗第一回合开始
   a. 玩家回合开始
      - 检查地形效果 (治疗符文回血/荆棘扣血)
      - 分配 AP 给玩家
      - 游戏等待玩家输入 (不限时)

   b. 玩家行动阶段
      - 玩家执行任意行动组合 (移动/攻击/技能/防御/道具/等待)
      - 每次 AP 消耗后更新状态
      - 玩家可随时点击"结束回合"

   c. 玩家回合结束
      - 处理剩余 AP 保留
      - 清除临时状态 (防御加成等)

   d. 怪物回合开始
      - 按 SPD 降序排列怪物行动顺序
      - 同 SPD 时按生成顺序

   e. 怪物行动阶段
      - 每个怪物依次执行其 AI:
        1. 检测自身状态 (血量/异常)
        2. 选择目标 (总是玩家)
        3. 执行 AI 行为 (移动/攻击/技能)
        4. 攻击判定 (伤害计算)
        5. 状态更新 (元素/增益/减益持续时间 -1)

   f. 怪物回合结束
      - 处理所有"回合结束"效果 (毒伤/灼烧扣血)
      - 检查是否触发元素反应 (两种元素同时存在)

   g. 第三回合开始前
      - 检查玩家是否死亡 → 是则进入觉悟战
      - 检查是否全部怪物死亡 → 是则房间清空
      - 否则回到 a 开始新一回合

3. 房间清空
   a. 停止回合循环
   b. 结算奖励 (金币掉落)
   c. 显示选择: 继续前进 / 查看跑马灯
   d. 退出战斗状态

================= 战斗结束 =================
```

### 5.2 状态的顺序与时点

```
回合内状态变化顺序（严格执行）:

1. [回合开始] 地形效果判定
2. [行动阶段] 玩家行动
3. [怪物回合] 怪物 AI 行动
4. [行动后] 元素状态持续时间 -1
5. [回合结束] 持续伤害结算 (毒/灼烧)
6. [回合结束] 元素反应检测
7. [回合结束] 临时增益/减益持续时间 -1
8. [下回合开始] 回到 1
```

### 5.3 战斗 Rage (超时狂暴)

| 指标 | 值 | 说明 |
|------|----|------|
| 超时阈值 | 20 回合 | 超过后触发 |
| 狂暴效果 | 所有敌人 ATK×2, DEF 归零 | 攻守双杀 |
| 环境伤害 | 每回合额外 2 点 AoE | 全房受伤 |
| 解除条件 | 全清敌人 | 狂暴不能取消 |

**设计意图**：超时狂暴不是"时间到了你输"，而是"时间到了，看谁先死"。给龟缩流一个合理的底线。

## 6. 地形系统规格

### 6.1 地形总表

| 地形 | 通行 | 效果 | 视觉暗示 | 出现概率 |
|------|------|------|----------|----------|
| 空地 | 可 | 无 | 普通地板 | 60% |
| 障碍物 | 不可 | 阻挡移动和视线 | 石柱/木箱 | 15% |
| 荆棘地板 | 可 | 经过扣 1 HP | 红色尖刺 | 8% |
| 治疗符文 | 可 | 站在上面每回合回复 2 HP | 蓝色发光圆 | 5% |
| 高台 | 可(2AP) | 站在上面 Range +1 | 灰色阶梯 | 7% |
| 暗区 | 可 | 进入后隐身 1 回合 | 紫色烟雾 | 5% |

### 6.2 Mechanic: 地形效果

```
Mechanic: 地面地形效果

Purpose: 让"站在哪里"和"走哪条路"成为战略决策
Player Fantasy: "这个位置很好利用"

地形触发时点:
  - 荆棘: 玩家或怪物**经过**该格时立即触发 (非"进入"后每回合)
  - 治疗符文: **站在**该格上的角色在回合开始时触发
  - 暗区: 角色**进入**该格时触发隐身效果, 离开时消失

Tuning Levers:
  - 各地形出现概率 (总和 100%)
  - 荆棘伤害值 (当前: 1)
  - 治疗符文回复值 (当前: 2/回合)
  - 高台额外 AP 消耗 (当前: 2/格)
  - 隐身持续时间 (当前: 隐身在暗区时持续)

Dependencies: 网格系统, 回合流程
```

### 6.3 房间地形布局规则

```
每个房间的地形布局由预设模板 + 随机偏移生成:

1. 房间大小: 6×6
2. 预设模板池: 10 种模板
   - 开放厅 (50%): 中央空地, 四周少量障碍
   - 走廊房 (15%): 两排障碍形成窄通道
   - 环形房 (10%): 中央凸起高台, 周围一圈荆棘
   - 治愈室 (8%): 中央治疗符文, 四周障碍保护
   - 暗影室 (7%): 大量暗区覆盖, 少量空地
   - 迷宫房 (5%): 密集障碍形成迷宫通道
   - 高台房 (3%): 四角高台, 中央空地
   - 荆棘房 (1%): 大量荆棘覆盖, 走错路就扣血
   - 复合房 (1%): 两种地形组合, 随机
3. 玩家入口固定在地图左侧, 怪物生成在右侧
```

## 7. 怪物 AI 系统规格

### 7.1 AI 主循环

```
每个怪物在回合中执行:

function executeTurn(monster, player):
    // 1. 检查状态
    if monster.isStunned: skipTurn() return
    if monster.isFrozen: 行动力减半 return
    
    // 2. 选择行为
    let behavior = selectBehavior(monster, player)
    // behavior 返回 (action, target, path)
    
    // 3. 执行行为
    switch behavior.action:
        case MOVE: moveAlong(path)
        case ATTACK: attack(target)
        case SKILL: useSkill(behavior.skill, target)
        case DEFEND: activateDefense()
        case WAIT: doNothing()
    
    // 4. 攻击后
    if attackHit:
        applyElement(monster.element)
        checkElementReaction(target)

// 怪物也可以在回合内移动+行动, 但不需要 AP 系统
// 怪物移动: 1 回合内最多 Move 格
// 怪物行动: 攻击/技能后, 该怪物结束
```

### 7.2 6 种 AI 行为规格

#### AI 类型 1: 冲锋型 (Charger)

```
Behavior: 冲锋型
Purpose: 快速接近玩家, 施加近战压力
Input: 检测玩家位置
Output: 沿 A* 路径向玩家移动, 进入 Range 后攻击
Success: 玩家受到伤害, 被追着跑
Failure: 被障碍物卡住 → 绕路, 如果完全无法到达则等待

varaints:
  - 遇到高台 -> 绕路 (不爬高台, 浪费时间)
  - 目标在 Range 内 -> 不移动, 直接攻击
  - 自身 HP < 30% -> 撤退到障碍物后 (求生本能)

Tuning: Move=3, ATK=中, DEF=低
```

#### AI 类型 2: 远程型 (Ranger)

```
Behavior: 远程型
Purpose: 保持距离, 持续远程消耗
Input: 检测玩家距离
Output: 
  - 玩家在 Range 外 -> 移动靠近到 Range 内
  - 玩家在 Range 内 -> 攻击
  - 玩家在 Range 内且过于接近(<=2格) -> 后退到 3 格距离
Success: 持续消耗玩家, 玩家需要近身
Failure: 被堵在角落 -> 被迫近身战斗

Tuning: Range=2~3, ATK=高, 近战DEF=低, Move=2
```

#### AI 类型 3: 防御型 (Defender)

```
Behavior: 防御型
Purpose: 吸收伤害, 保护后排 (或在没有后排时成为沙包)
Input: 检测自身 HP 和玩家位置
Output:
  - HP > 50% -> 架盾 (DEF×2), 若有攻击机会则攻击
  - HP <= 50% -> 取消架盾, 全力攻击
  - 玩家在背后 -> 转身面对玩家
Success: 玩家需要绕后或使用技能破防
Failure: 被 AoE 技能绕过防御

Tuning: DEF=高, ATK=低, Move=1, 架盾时不可移动
```

#### AI 类型 4: 召唤型 (Summoner)

```
Behavior: 召唤型
Purpose: 拖延战斗, 制造数量优势
Input: 检测已存在于场的召唤物数量
Output:
  - 当前回合数能被 2 整除 -> 在空地上召唤 1 个小怪
  - 召唤 CD 中 -> 后退远离玩家
  - 玩家贴近 <= 2 格 -> 放弃召唤, 攻击
Success: 玩家被小怪包围, 召唤者躲在后方
Failure: 没有空地可召唤 -> 只能攻击

Tuning: HP=中, ATK=低, 召唤小怪=迷你版自身(HP/ATK减半), 召唤CD=2回合
```

#### AI 类型 5: 自爆型 (Suicider)

```
Behavior: 自爆型
Purpose: 给玩家明确的时间压力——"3回合内解决我, 不然你炸"
Input: 自身计数器 (从进入战场开始)
Output:
  - 回合 1: 向玩家移动, 身体闪烁红光
  - 回合 2: 继续接近, 红光更亮
  - 回合 3: 锁定自爆! 范围 2 格 AoE, 伤害 = 剩余HP×2
  - 如果在 3 回合内死亡 -> 普通死亡, 不自爆
Success: 玩家需要优先击杀它或远离
Failure: 玩家没躲开 -> 受到大范围高伤害

Tuning: HP=8(低), 自爆伤害=剩余HP×2, 自爆范围=2格
Visual: 每回合红光更亮, 第 3 回合全身闪烁
```

#### AI 类型 6: 精英型 (Elite) — 混合行为

```
Behavior: 精英型 (BOSS 和精英怪物专用)
Purpose: 给玩家真正的挑战——需要策略, 不是蛮力
Input: 自身状态 + 玩家状态 + 战场状态
Output: 从 2 种行为模式中选择当前最合适的行为

精英 AI 需要单独为每个 Boss 编写行为树, 不通用。
示例: "暗影领主" BOSS:
  Phase 1 (HP > 50%): 远程型 + 召唤型混合, 每 3 回合召唤 1 个暗影小怪
  Phase 2 (HP <= 50%): 冲锋型 + 防御型混合, 召唤 CD 缩短为 2 回合
  Phase 3 (HP <= 20%): 疯狂模式, 每回合双动, 受击回血

Tuning: 每个 Boss 单独设计
```

### 7.3 怪物 AI 决策表速查

| 条件 | 冲锋型 | 远程型 | 防御型 | 召唤型 | 自爆型 |
|------|--------|--------|--------|--------|--------|
| 玩家在攻击范围外 | 靠近 | 靠近到射程 | 原地架盾 | 后退 | 靠近 |
| 玩家在攻击范围内 | 攻击 | 攻击 | 架盾+攻击 | 后退 | 靠近 |
| 玩家贴脸(<=2格) | 攻击 | 后退 | 架盾 | 攻击(放弃召唤) | 不动(准备自爆)|
| HP < 30% | 撤退 | 撤退 | 取消架盾,全力攻击 | 后退,召唤 | (不适用) |

## 8. 战斗生命周期

### 8.1 战斗状态机

```
    [探索中] ──进入房间──→ [战斗开始]
                                │
                                ├─→ [玩家回合] ←──────────────┐
                                │       │                      │
                                │       ↓                      │
                                │  [玩家行动]                  │
                                │       │                      │
                                │       ↓                      │
                                │  [怪物回合]                   │
                                │       │                      │
                                │       ↓                      │
                                │  [回合结算] ──存活+有怪──→ ──┘
                                │       │
                                │       ├──全清──→ [房间清空]
                                │       │              │
                                │       └──死亡──→ [觉悟战]
                                │                       │
                                │                   复活?──是──→ [战斗开始]
                                │                       │
                                │                       └─否──→ [觉悟结算]
                                │                                  │
                                └──────────────────────────────────┘
```

### 8.2 Mechanic: 觉悟战 (Death & Revival)

```
Mechanic: 觉悟战

Purpose: 死亡不是终点, 是选择——让"垃圾时间"也有价值
Player Fantasy: "我还没输, 我选择怎么死"
Input: 玩家 HP 归零
Output: 
  选项 A: 看 30s 广告 → 以 50% HP 在当前房间复活
  选项 B: 不接受 → 进入觉悟结算, 按照当前层数获得魂石

Success Condition: 
  - 复活: 玩家回到战斗中, 继续挑战
  - 结算: 玩家获得合理的魂石回报, 不会觉得"白打了"

Failure State: 
  - 复活后再死 → 再次弹出觉悟战 (不重复播放广告, 直接结算)
  - 复活广告播放失败 → 自动进入结算

Edge Cases:
  - HP=0 时还有怪物没行动 → 觉悟战在怪物行动完当前回合后弹出
  - 玩家在觉悟战中有金币 → 金币保留 (不清零)
  - 玩家在觉悟战中连续触发事件 → 仅触发一次

Tuning Levers:
  - 复活 HP 比例 (当前: 50%)
  - 复活是否影响魂石结算 (建议: 不影响, 该多少就多少)
  - 每局复活次数限制 (建议: 无限制, 但每次广告更贵——二次复活看 60s)

Dependencies: 广告系统, 经济系统
```

### 8.3 房间清空结算

```
战斗结束 → 房间清空时执行:

1. 固定奖励:
   - 金币: 3~5 (随机, 房间等级越高越多)
   - 有 20% 概率掉落 1 把钥匙
   - 有 5% 概率掉落一个道具

2. 特殊条件奖励:
   - 无伤通关: 额外 2 金币
   - 触发 3 次以上元素反应: 额外 1 金币
   - 3 回合内全清: 额外 1 金币

3. 跑马灯进度:
   - 战斗结束显示跑马灯进度条
   - 每场战斗 +1 格进度
   - 玩家可选择看广告点亮该格
```

## 9. 战斗平衡数据

### 9.1 各层战斗预期

| 层数 | 房间数 | 战斗房数 | 预期总回合 | 预期 HP 消耗 | 预期金币获得 |
|------|--------|---------|-----------|-------------|-------------|
| 1 | 5 | 3 | 9~15 | 8~15 | 12~20 |
| 2 | 6 | 3~4 | 12~24 | 15~25 | 15~25 |
| 3 | 7 | 4 | 16~32 | 20~35 | 18~30 |
| Boss | 1 | 1 | 5~10 | 10~20 | 6~10 (Boss奖励) |

### 9.2 战斗难度目标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 普通战斗获胜率 | > 90% | playtest 统计 |
| Boss 战获胜率 | 50~60% | playtest 统计 |
| 无伤通关率 | < 20% | 避免过于简单 |
| 回合超时率 | < 5% | 超时 = 设计失败 |
| 玩家死亡体验 | "就差一点"而非"完全打不过" | 问卷反馈 |

### 9.3 [HYPOTHESIS] 战斗平衡假设

```
1. 1 层的小怪玩家应该 2 次攻击内击杀
   → 验证: 玩家 ATK=5, 骰子平均 3.5, 伤害 ≈ 7~8, 小怪 HP=6 → 1~2 次攻击
   → 结论: [HYPOTHESIS] 合理

2. 3 层 Boss 战应该在 5~10 回合内结束
   → 验证: 玩家 ATK=9, 骰子平均 3.5, Boss DEF=2 → 伤害 ≈ 11/回合
   → Boss HP=35 → 约 4 回合击杀 (加上暴击和技能)
   → 结论: [HYPOTHESIS] 偏快, 可能需要调整

3. 玩家一场战斗的 HP 消耗应控制在 HP 的 30~60%
   → 验证: 1 层玩家 HP=20, 战斗房 2 怪 → 受到约 6~12 伤害 → 30~60%
   → 结论: [HYPOTHESIS] 合理

4. 防御姿态的价值感知
   → 验证: 本回合 DEF+3 vs 普攻可能造成的伤害 → +3DEF ≈ 减少 1.5 伤害
   → 1 AP 换 1.5 减伤 ≈ 值得吗?
   → 结论: [HYPOTHESIS] 需要 feel test — 如果玩家几乎不用防御, 说明价值不足
```

## 10. 战斗系统的"破解"标准

在 playtest 前, 先定义什么情况下战斗系统"坏了":

| 破坏模式 | 判定标准 | 严重等级 |
|----------|----------|----------|
| **无限回血** | 玩家可以在战斗中无限回复, 不会死 | ❌ 灾难 |
| **无敌龟缩** | 玩家躲在角落防御, 怪物永远打不中/打不动 | ❌ 灾难 |
| **一刀秒 Boss** | 玩家在第一层就获得能力, Boss 被秒杀 | ⚠️ 严重 |
| **AP 无限循环** | 某套连招可以恢复 AP, 形成无限行动 | ❌ 灾难 |
| **怪物完全打不动** | 高 DEF 怪物, 玩家造成 1 点伤害要 20+ 回合 | ⚠️ 严重 |
| **怪物自己打死自己** | 反伤/荆棘让怪物比自己伤害还高 | ✅ 可接受(玩家乐趣)|
| **Boss 两回合击杀玩家** | Boss 行动让玩家无反应时间直接死亡 | ❌ 灾难 |
| **战斗超过 30 回合** | 玩家或怪物持续低伤害无解 | ⚠️ 严重 |

## 11. Playtest 计划

### 11.1 纯文字原型测试

```
目标: 验证"AP 分配"是否有趣, 地形是否带来决策

准备:
  - 6×6 网格的打印纸或 Excel 表格
  - 3 种颜色的棋子 (玩家/怪物1/怪物2)
  - 骰子 1 个
  - 计时器

测试脚本:
  1. 摆好预设房间模板 (3 种)
  2. 玩家每回合:
     a. 在纸上标记 AP 消耗
     b. 写下"为什么这么选"
  3. 测试者扮演怪物 (按 AI 规则)
  4. 记录每回合的决定时间和选择
  5. 测试后问卷: "你最喜欢哪个选择? 最纠结哪个?"
```

### 11.2 首次代码原型验收标准

```
[ ] 玩家能在 6×6 网格上自由移动
[ ] 3 AP 的分配有感受——不够用? 刚好? 太多?
[ ] 6 种地形都有玩家使用的情况
[ ] 6 种怪物 AI 表现出可辨识的行为差异
[ ] 战斗在 5~15 回合内结束
[ ] 玩家至少使用过防御姿态
[ ] 元素反应在战斗中至少触发过 1 次
[ ] 觉悟战的"选择感"强于"挫败感"
```

## 12. 与其他系统的接口定义
