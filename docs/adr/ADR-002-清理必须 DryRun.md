# ADR-002：为什么清理必须 Dry-Run（默认不删除）

- **状态**: Accepted
- **日期**: 2026-07-11
- **决策人**: 项目治理（多 AI 协作）

## 背景
早期方案曾直接产出"可删除清单"并隐含可执行，导致风险：① 门禁"9/9"被误读；② 误以为 MD5 相同即可直接删除。多个 AI 对话场景下，任一 AI 擅自 `rm` 都可能造成不可逆破坏。

## 决策
所有清理/删除/移动操作强制 **三阶段流程**：
1. **计划（Dry-Run）**：工具只生成清单，绝不修改文件系统。
2. **确认**：人工审阅清单，并创建 Git 安全点（`git tag` / 分支）。
3. **执行（Apply）**：仅在显式二次确认（`--i-have-reviewed-the-plan`）后，且仅删除通过四步安全门的项。

执行后须**重扫验证**，确认无新断链/新孤儿；若引用关系变化则终止并提示人工。

## 后果
- 正面：删除性操作零默认执行；即便 AI 遗忘规则，工具内置安全门（grep 引用 + package.json + git 状态）仍阻止误删核心文件。
- 负面：清理流程变长，需人工介入；但这是有意为之的安全成本。

## 反选项
- 直接删除：效率最高但风险不可接受，已被 v1 经验否定。
