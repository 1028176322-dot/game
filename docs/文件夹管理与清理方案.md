### 5.2 文件命名规范（扩展既有 `design/文件管理规范.md`）

### 5.8 删除前必须创建安全点（Git 可恢复 ≠ 零风险）

> **更正上一版"零风险"说法**：git 可回滚，但 `git clean -fdx` / `git gc` / 强制 `push --force` / `rebase` 会显著抬高恢复成本。因此**任何删除前必须创建 Tag 或临时分支**：

```bash
# 方式一：打 tag（推荐，轻量）
git tag cleanup_before_20260711

# 方式二：开临时分支（便于整体回退）
git checkout -b backup_cleanup

# 之后如需回退：
git checkout main && git tag -d cleanup_before_20260711   # 或 git checkout backup_cleanup
```

### 5.9 删除前安全门（四步复核，针对"完全相同即可删"误区）

无论规划器是否标记 SAFE_DELETE，**对每一个待删除文件 / 重复组副本，人工或 AI 执行前必须依次确认**：

1. **① hash 一致**：确认待删文件与"保留方"内容相同（规划器重复组已验证 MD5）。
2. **② grep 全文引用扫描**：在仓库内全文搜索待删文件的**相对路径与 basename**，确认引用数 = 0。
   ```bash
   git grep -n "tools/gen_scenes.py"        # 查根层副本是否被引用
   git grep -n "gen_scenes.py"              # 查 basename 是否被引用
   ```
   若引用数 > 0 → **终止删除该副本**，改删另一份或保留。
3. **③ package.json 检查**：确认 `package.json` / `回到地面/package.json` 的 `scripts` 未引用该文件。
4. **④ 删除**：仅当 ①②③ 全部通过，才执行 `git rm`（已跟踪）或 `rm`（未跟踪 / 已忽略）。

> 规划器已内置此门：`SAFE_DELETE` 仅当（无未提交修改 ∧ 引用数=0 ∧ 未出现在 package.json ∧ 非 MUST_KEEP）才成立；`weak` 标记文件（如 `tmp_*`）即便无引用也**永不自动删除**，强制人工复核。

```
扫描 → 生成 inventory → 重复检测 → 引用检测 → Git 状态检测 → 生成删除计划 → 等待确认 → Apply → 重新扫描 → 生成报告
   │           │             │           │           │              │              │          │         │            │
   └───────────┴─────────────┴───────────┴───────────┴──────────────┴──────────────┴──────────┴─────────┴────────────┘
                                         （全程默认 dry-run，不删除）
```

| 阶段 | 动作 | AI 是否可自动 |
|---|---|---|
| 1 扫描 | `cleanup_scan.py` 遍历 + 哈希 | ✅ 自动 |
| 2 inventory | 输出 `inventory.json` | ✅ 自动 |
| 3 重复检测 | MD5 分组 | ✅ 自动 |
| 4 引用检测 | `git grep` basename 计数 | ✅ 自动 |
| 5 Git 状态 | tracked / ignored / Modified | ✅ 自动 |

## 8. AI 执行原则（新增章节，对应反馈 §七）

> 无论未来由哪个 AI / 哪次会话执行清理、移动、重命名操作，均应遵守以下统一原则：

1. **默认 Dry Run**：任何扫描 / 规划工具默认不删除、不移动、不重命名；破坏性动作需显式开关。
2. **所有删除操作必须提供待删除清单**：执行前打印 / 输出完整清单（路径 + 判定 + 依据），供人工审阅。
3. **删除前重新进行引用分析**：对每一待删项，删除前再次 `git grep` 全文引用，确认引用数 = 0（应对"扫描后到删除前"窗口内的新引用）。
4. **删除前确认 Git 已提交 / 已建安全点**：执行删除前必须 `git commit` 当前进度，并打 `git tag cleanup_before_YYYYMMDD` 或开 `backup_cleanup` 分支。
5. **删除后重新扫描验证**：删除后再次运行规划器，确认无引用关系变化、无新产生的孤儿 / 断链。
6. **若发现引用关系变化，则终止执行并提示人工确认**：任何阶段若发现"原判定 SAFE 的项现在被引用"或"删后产生新断链"，立即停止，回报人工。
7. **不靠单一信号判定**：删除 / 保留判定必须联合 `git tracked` + `gitignore` + 引用分析 + `package.json` 检查，禁止仅凭文件名（如 `_tmp_` / `debug`）下结论。

| **P2** | **AI 可自动执行**（通过安全门后） | 命中风 `_tmp_`/`_debug_`/`.backup_` 且四步门全过的备份/草稿、空目录 |

> 示例：`DebugPanel.ts` 在 `must_keep` → 任何涉及它的操作均为 **P0**；`prompts.backup_*.json` 通过四步门 → **P2**，可在人工确认后由 AI 自动清理。

### 11.7 规则演进与决策记录（避免规则漂移）

- **演进日志**：`docs/CHANGELOG_AI_RULES.md` 记录规则从 v1（无安全门）→ v2（DryRun）→ v3（must_keep）→ v4（优先级 / AST / 退出码）的完整演进，便于长期维护与多 AI 协作审计。

  - `ADR-002` 为什么清理必须 DryRun
