# 资源批量生成优先级计划

> 基于 `visual_review_status.md` 进度基准，全量 499 个资源中 10 个已入库，1 个待修复，488 个未启动。

---

## 优先级判定依据

| 维度 | 说明 |
|------|------|
| **游戏可用性** | 缺少该资源后玩家能否进入/完成核心玩法 |
| **UI 依赖** | 是否被已有场景/面板直接引用 |
| **单批可控性** | 批量大小是否适合一次 API 调用 + 一次视觉验收 |
| **管线成熟度** | 该类管线是否已通过端到端验证 |

---

## 批次计划

### 第 1 批 — Monsters（36 张）

- **估计 API 调用**: 36 次（36 张 × 1 轮）
- **预估耗时**: ~30-45 分钟
- **生成方式**: Agnes AI + 管线后处理
- **优先级理由**: 游戏基础怪物资产，数量适中，适合作为管线首轮全流程验证
- **验收方式**: 生成后出 contact sheet → 用户视觉确认 → 批量 import

### 第 2 批 — Tiles（24 张）

- **估计 API 调用**: 0（程序化生成）
- **预估耗时**: ~2 分钟
- **生成方式**: 管线内置程序化生成
- **优先级理由**: 无 API 成本，生成快；地图 tiles 是关卡基础
- **验收方式**: 生成后自动验证 + 用户 spot check

### 第 3 批 — Icons（67 张）

- **估计 API 调用**: ~67 次（含 icon_item_key 已验证的 prompt 模板）
- **预估耗时**: ~60-90 分钟
- **生成方式**: Agnes AI（复用 icon_item_key 的 prompt 模板经验）
- **优先级理由**: icon 被 UI 面板广泛引用；已有已验证的 prompt 模板可供复用
- **验收方式**: 分 2-3 个子批次生成 → contact sheet → 视觉确认 → import

### 第 4 批 — Backgrounds（17 张）

- **估计 API 调用**: 17 次（含 splash_bg 修复）
- **预估耗时**: ~20-30 分钟
- **生成方式**: Agnes AI
- **优先级理由**: 部分场景已引用背景资源，需先修复 splash_bg 体积问题
- **验收方式**: contact sheet → 视觉确认 → import

### 第 5 批 — Characters（35 张）

- **估计 API 调用**: 35 次（复用 character_card 经验）
- **预估耗时**: ~30-45 分钟
- **生成方式**: Agnes AI
- **优先级理由**: 选角色面板已有 UI Kit 卡片，但角色图标/立绘仍需 AI

### 第 6 批 — UI 其余（~97 张）

- **估计 API 调用**: ~97 次
- **预估耗时**: ~90-120 分钟
- **生成方式**: Agnes AI（非 procedural_ui 的 UI 资源，如 login_panel、event_panel 等）
- **优先级理由**: 大数量，且涉及 login/settings/event 等核心面板

### 第 7 批 — Bosses（120 张）

- **估计 API 调用**: 120 次（体量最大的一批）
- **预估耗时**: ~2-3 小时（建议拆为 3 个子批次）
- **生成方式**: Agnes AI（含 asset_kind 分类检查）
- **优先级理由**: 数量最多，每张需要更高预算（256KB hard limit）

### 第 8 批 — Effects（27 张）

- **估计 API 调用**: 27 次
- **预估耗时**: ~30 分钟
- **生成方式**: Agnes AI（160KB hard limit）
- **优先级理由**: 视觉特效，对游戏核心玩法影响最小

## 执行规则

1. **每批启动前**：运行 `encoding_audit.py --ci`，确认代码无编码问题
2. **每批生成后**：运行 `generate --self-check` 自动跑技术门禁
3. **视觉验收前**：生成 contact sheet 供用户一次审查
4. **import 前**：手动确认 `visual_review_status.md` 中状态
5. **import 后**：运行 `npm.cmd run validate:all`

## 进度文件维护

- 每批完成后，`visual_review_status.md` 新增该类的 section
- 资源按状态（已入库 / 待重跑 / 待生成）分组
- 未启动的类保留在"未启动类别"汇总表中计数递减
