## 0. 设计背景与决策记录

### 0.1 参考游戏定位（经截图核实）
《酋长你别跑》= 暗黑风格 ARPG + Roguelike + 地牢 + 养成，核心是"部落伙伴辅助下的肉鸽地牢探索"。
**本重做只借用其「肉鸽地图/场景的生成与呈现方式」，不复制其暗黑美术，也不引入其没有的群体/追逐机制。**

### 0.2 用户决策（Q&A 收敛结果）
| 维度 | 决策 | 来源 |
|------|------|------|
| 肉鸽地图结构 | 节点路线图（Slay the Spire 式）| 用户确认 |
| 场景呈现 | 3D 俯视 | 用户确认（原 Q&A "上3D俯视"）|
| 战斗形态 | 保留单人 ARPG 手感 | 用户确认（原 Q&A "保留ARPG"）|
| 群体随从 | 不引入 | 用户确认（参考游戏无，非本重做范围）|
| 被追击追逐 | 不引入 | 用户确认（参考游戏无，非本重做范围）|

### 0.3 关键架构转变（务必理解）
现有地牢是**物理连续穿越**：玩家在 6×6 网格房间里左右走动，清怪→开门→走下一间。
节点路线图改为**选路 + 独立场景（instanced encounter）**：地图 UI 上选节点 → 进入该节点专属 3D 场景 → 打完/交互完 → 回到地图选下一节点。**玩家不再在地图里物理行走，地图是决策层，场景是执行层。**

## 1. 设计支柱 (Design Pillars)

1. **战略选路 (Strategic Routing)** — 进层即见整图，每条路都有取舍（安全/风险回报/资源），玩家为"下一节点去哪"做 meaningful choice。
2. **肉鸽不可预测 (Roguelike Uncertainty)** — 节点类型与连接每次随机，但同种子可复现；"这一层长什么样"是每次新鲜的盲盒。
3. **ARPG 手感不丢 (Keep ARPG Feel)** — 战斗仍是手动 ARPG；节点只是"去哪打/打什么"的决策，不稀释操作爽感。
4. **3D 俯视的立体场景 (3D Top-Down Scenes)** — 每个节点进入后是 3D 俯视战斗/事件场景，规模感与打击感升级（复用 CombatEffectService 的闪白/震屏/弹道）。
5. **清晰进度反馈 (Clear Cadence)** — 节点地图提供"从层底走到层顶 Boss"的明确目标感与逐节点推进反馈。

## 2. 核心循环 (Core Loop)

### Moment-to-Moment (0–30 秒)
- **Action**: 在节点地图上点击一个**当前可达**的节点（如战斗/事件/宝藏）。
- **Feedback**: 地图高亮选中 → 场景过场（淡入/传送）→ 进入 3D 俯视场景。
- **Reward**: 立刻知道"我要打什么/拿什么"，决策感满足。

### Session Loop (5–30 分钟)
- **Goal**: 从层底节点一路选路抵达层顶 Boss 节点并击败它。
- **Tension**: 路径资源有限（血/金币/强化次数），选"精英高风险高回报"还是"绕路保平安"。
- **Resolution**: 击败 Boss → 进入下一层（地图重新生成，结构更深/更难）→ 或失败结算。

### Long-Term Loop (小时–周)
- **Progression**: 多层推进解锁 meta（角色/装备/元素Build）；种子挑战与每日/周常。
- **Retention Hook**: 每层地图不同带来的重开新鲜感 + Build 成长 + 深层 Boss 目标。

## 3. 关键机制

### 3.1 Mechanic: 节点路线图生成 (Node-Route Map Generation)

**Purpose**: 生成一层"可见全图、可战略选路"的肉鸽节点地图，替代现有浅层 DAG。
**Player Fantasy**: "这一层有两条路——左边绕精英拿好货，右边直接冲 Boss，我选哪条？"
**Input**: `seed`(子种子) + `floorNum` + `difficulty`
**Output**: 一层节点图 = `rows[]`，每行 `nodes[]`，边 `edges[]`（仅连相邻行）。

**生成算法（基于现有 地牢生成系统规格.md 第1章 DAG 升级）**:

```
1. 确定行数 (rows)
   - 公式: rows = R_BASE + floorNum        [PLACEHOLDER] R_BASE=10
   - 理由: 每层比上层长 1 行，制造"越深越长"的压迫感

2. 每行节点数
   - 中间行: 2~4 个节点（randInt）
   - 第 0 行(层底入口): 1 个固定起点节点
   - 末行(层顶): 1 个固定 Boss 节点
   - 理由: 入口/Boss 唯一保证路径收束

3. 生成边（相邻行连接）
   - 对每个 row r 的节点，连接到 row r+1 的 1~3 个节点
   - 约束: row r+1 每个节点至少被 1 个 row r 节点连接（无死点）
   - 约束: 至少有 2 条从起点到 Boss 的不同路径（战略选择的前提）
   - 约束: 禁止跨行连接（只连 r→r+1，保证无环 DAG）

4. 节点类型分配（见 3.1.1 表）
   - 第 0 行起点: 固定"出发"节点（非战斗）
   - 末行 Boss: 固定 Boss 节点
   - 其余按权重抽取，但保证:
       · 每层至少 1 个 商店 / 1 个 休息（保命经济）
       · 每层至少 1 个 事件（叙事/变量）
       · 战斗 + 精英 占比 ≥ 55%（保证战斗密度）

5. 验证（失败重试 3 次，否则用预设模板）
   - ✅ 所有节点可达（从起点 BFS）
   - ✅ Boss 可达
   - ✅ 无环、无死点
   - ✅ ≥2 条不同路径

6. 预设模板 fallback（同现有 地牢生成系统规格.md 第1.3章思路，改为多行）
```

**3.1.1 节点类型表**

| 节点类型 | 基础权重 | 进入后场景 | 奖励/交互 |
|----------|----------|------------|-----------|
| 战斗 Combat | 40% | 3D 战斗场景（普通怪）| 经验 + 装备掉落 |
| 精英 Elite | 12% | 3D 战斗场景（强化怪）| 高质掉落 + 金币 |
| 事件 Event | 10% | 3D/2D 事件场景（2选1）| 变量奖励（参考现有事件房）|
| 宝藏 Treasure | 10% | 宝箱场景 | 金币/道具（参考现有宝箱房）|
| 商店 Shop | 8% | 商店场景（参考现有商店房）| 购买 |
| 休息 Rest | 10% | 营地场景（参考现有回复房/强化房）| 回血 或 强化3选1 |
| 出发 Start | 固定1 | — | 层入口 |
| Boss | 固定1 | 3D Boss 场景（参考现有 Boss 房）| 通关 + 进深层 |

**Edge Cases**:
- 某行只有 1 个节点 → 该行成为"瓶颈"，玩家无选择（可接受，但避免连续 2 行单节点——视觉疲劳 + 失去选路）。
- 边生成导致某节点 0 入度 → 验证失败，重连。
- 5 层时 rows 达 15 → 地图 UI 需可滚动/缩放（见 3.2）。

**Tuning Levers**: 行数公式、每行节点数范围、类型权重、精英占比下限、最小路径数。
**Dependencies**: 种子系统（子种子链，消费顺序：行数→每行节点数→边→类型→偏移）。

### 3.2 Mechanic: 节点地图 UI（决策层）

**Purpose**: 让玩家"看全图、做选路"，是节点路线图区别于物理穿越的核心。
**Player Fantasy**: "整层在我眼前，我规划路线。"
**Input**: 生成的 `rows[]/nodes[]/edges[]`
**Output**: 屏幕上的可交互节点图（覆盖层，2D HUD，非 3D 场景内）。

**呈现规范**:
- **视角**: 地图 UI 为 2D 覆盖层（类 Slay the Spire 竖向节点图），叠加在场景之上或独立全屏面板。**不强制 3D**（节省成本，且节点图本质是信息 UI）。
- **状态样式**:

| 状态 | 显示 | 交互 |
|------|------|------|
| 未解锁(无路径) | 暗色 + 类型图标 | 不可点 |
| 当前可达 | 高亮边框 + 类型图标 | 可点 → 进入 |
| 已通过 | 灰色勾选 | 可回顾 |
| 当前所在 | 全彩 + 脉冲高亮 | 当前位置 |
| Boss | 红色图标（禁血/骷髅，用酋长徽记替代）| 始终可见 |

- **布局**: 竖向滚动，行从上(层顶/Boss)到下(层底/起点)或反之，**保持"从底走到顶"的方向直觉**。
- **缩放/滚动**: 行数 >10 时支持拖拽缩放（触屏友好，节点间距 ≥ 50px）。

**Success Condition**: 玩家进入新层 3 秒内理解"我在哪、能去哪、Boss 在哪"。
**Failure State**: 玩家困惑"下一步去哪" → 地图信息层级错误。
**Dependencies**: 节点路线图生成（3.1）、UITree（现有 UI 框架）。

---

### 3.3 Mechanic: 节点进入与 3D 场景呈现（执行层）

**Purpose**: 每个节点类型对应一个 3D 俯视场景；玩家在场景里执行战斗/交互，完后返回地图。
**Player Fantasy**: "点下去，瞬间置身战场。"
**Input**: 玩家选中节点 → `enterNode(nodeType)`
**Output**: 加载对应 3D 场景 → 执行 → 结算 → 返回地图 UI。

**场景呈现规范（3D 俯视）**:
- **相机**: 正交或微倾俯视（orthographic top-down），保证群体/规模可读。
- **场景即"房间"的 3D 化**: 现有 6×6 地形模板（开放厅/走廊/荆棘…）升级为 3D 地形块（tile mesh），障碍/荆棘/高台为 3D 实体。
- **战斗内**: 复用现有 ARPG（虚拟摇杆 + 攻击动画 + 受击闪白 + 震屏 + 弹道）+ CombatEffectService；元素反应在 3D 下用粒子/着色增强。
- **场景生命周期**: `enterNode` → 实例化场景 → `onRoomCleared`/`onEventResolved` → 卸载 → 回地图。
- **参考**: 2D转3D全面升级方案.md v3（AssetCache / ICollisionService 3D / 渲染组替代 RuntimeLayerService 5 层）。

**Success Condition**: 节点进入 <1.5s 无卡顿，3D 场景辨识度高于原 2D。
**Failure State**: 场景加载慢/3D 辨识混乱 → 退回 2D 该节点验证手感。
**Edge Cases**:
- 连续进入战斗节点 → 场景资源需 AssetCache 复用，避免重复加载卡顿。
- Boss 节点 → 场景需更大（参考现有空旷房/复合房）。
**Dependencies**: 2D转3D方案、RuntimeLayerService(升级)、SpriteAnimationService(升级为3D动画)、CombatEffectService。

---

### 3.4 Mechanic: 单人 ARPG 战斗（不变，仅承载于 3D 场景）

**Purpose**: 提供核心操作爽感；本重做**不改战斗规则**，只把战斗搬到 3D 节点场景里。
**Player Fantasy**: "我手动砍怪，打击感拉满。"
**保留项**: 虚拟摇杆移动、近身攻击动画、受击反馈、技能按钮、元素反应、Boss HP 条。
**移除项（相对原设想）**: 无群体随从（本重做不含），战斗对象仍为"玩家 1 + 怪物群"。
**Dependencies**: 现有 BattleManager / MonsterController / CombatEffectService / 技能系统设计.md / 元素反应 Build。

---

### 3.5 Mechanic: 路径奖励与 Build 积累

**Purpose**: 让"选路"有长期意义——不同节点喂不同成长资源。
**节点→奖励映射**（复用现有系统）:

| 节点 | 奖励系统 | 复用现有 |
|------|----------|----------|
| 战斗/精英 | 经验 + 装备掉落 | 装备系统设计.md |
| 宝藏 | 金币/道具 | 道具系统设计.md |
| 商店 | 购买 | 现有商店房 |
| 休息 | 回血 或 强化3选1 | 现有强化房 |
| 事件 | 2选1 变量 | 现有事件房 |

**Dependencies**: 经济系统模型.md、装备/道具/技能系统。

## 5. 新手引导 (Onboarding Checklist)

- [ ] 进入首层即显示**全图节点 UI**，起点高亮，Boss 可见（目标感）
- [ ] 首节点必为**战斗**且**保证可胜**（参考现有"首层至少2战斗房"保护）
- [ ] 第一个战斗节点前 30 秒引入核心动词（移动+攻击），无失败可能
- [ ] 第二个节点引导玩家**主动选路**（提示"可走两条路"），而非自动前进
- [ ] 首层含 1 个 休息/事件 节点，教学"节点类型差异"
- [ ] 首 session 结束于"击败首层 Boss"的钩子（解锁第 2 层，想再来一局）

## 6. 技术影响与风险

### 6.1 最大风险：2D → 3D 场景转换
- 必须遵循 `docs/2D转3D全面升级方案.md` (v3) 的分阶段路线（Demo→Vertical Slice→Phase1-6）。
- `RuntimeLayerService` 5 层 → 3D 渲染组 / 深度排序（遵循 ARCH_FOUNDATION）。
- `SpriteAnimationService` 竖排 sprite sheet → 3D 骨骼/状态动画（用户已具备 3D 资源生成能力）。
- `ICollisionService` 需支持 3D 重叠检测（避免直接 `PhysicsSystem`）。
- 战斗反馈 `CombatEffectService` / 事件总线（`attack:performed` 等）**复用**，不重写。

### 6.2 架构原则（不可违反）
- 遵循 `ILifecycle`+`LifecycleManager`、`GameContext` DI、`SkillGraph`（禁 `switch(id)`）、`IAIController`、`AssetCache`。
- 仅 `SceneFlowService` 可调 `director.loadScene()`。

### 6.3 受影响文件（实施时需同步更新，供 `validate:all` 门禁）
| 文件 | 改动 |
|------|------|
| `docs/地牢生成系统规格.md` | DAG(3层) → 节点路线图(多行) 重写 |

## 7. 分阶段实施计划（垂直切片优先）

| 阶段 | 内容 | 验收 | 依赖 |
|------|------|------|------|
| **P0 逻辑** | 节点路线图生成算法（扩展现有 DAG，纯逻辑+单测）| 同种子复现、≥2路径、Boss可达 | 种子系统 |

| **P5 Meta** | 多层推进 + 种子挑战 + 每日/周常 | 重开有新鲜感 | P4 + 经济系统 |
