# 美术资源生成与入库规范

> 适用范围：后续所有新增、重做、替换、补齐、压缩、入库的角色、怪物、Boss、背景、tile、特效、图标、UI、按钮、面板、展示区资源。  
> 目标平台：TapTap（安卓原生游戏）。  
> 主风格：高清卡通动物冒险风。不是像素风，不是暗黑风，不是写实恐怖风。  
> 权威提示词来源：`assets/resources/config/prompts.json`。  
> 正式运行目录：`assets/resources/textures/`。  
> 母版目录：`art_source/textures_review/master/`。  
> 运行时候选目录：`art_source/textures_review/runtime_candidates/`。

## 0. 文档关联与适用范围

本文是 **2D 贴图 + 3D 模型** 双轨资源的「生成与入库操作 SOP」。

- **2D 贴图**（icons / ui / backgrounds / 2D tiles 等）：本章正文即完整规范，出图走 Agnes / 程序化。
- **3D 模型**（角色 / 怪物 / Boss / 特效 / 3D tile，Blender→.glb→Prefab）：
  - 制作参数（为什么 / 原则 / 预算 / LOD / 依赖 / 生命周期）见 **Layer 1** `./美术资源制作参数总表_3D.md`；
  - 机器可读逐资源规格（target/min/max、Sockets 数组、Fail 条件、Input/Output）见 **Layer 2** `./AI资源制作规范_MachineSpec.yaml` 与 `.json`；
  - AI 执行流水线（9 步、每步门禁）见 **Layer 3** `./AI执行流程_Workflow.md`；
  - **3D 唯一权威规则源**：`../../.workbuddy/memory/topics/ART_RESOURCE_RULES.md` §16（命名 / 预算 / 目录 / 生命周期），数值预算在 `assets/resources/config/art_quality_budget.json → rules3d`。
  - **3D 技术门禁**：`python tools/asset_validate.py <manifest> [--budget]`，未全绿禁止入库。

> ⚠️ 3D 的三角面 / 骨骼 / 纹理 / LOD 等**数值一律以 §16 / rules3d / Layer 2 为准**，本文不重复硬编码。

## 1. 绝对规则

任何资源不满足以下规则，都不能进入 `assets/resources/textures/`。

| 项 | 要求 |
|---|---|
| 风格 | 明亮、友好、圆润、高清卡通动物游戏美术 |
| 文本 | 图片内禁止任何中文、英文、数字、Logo、水印、伪文字 |
| 安全 | 禁止血液、骷髅、骨头、器官、尸体、恐怖表情、惊悚宗教符号、写实伤口 |
| 格式 | 透明小图用 PNG RGBA；全屏不透明背景优先 JPG；禁止 PNG P 模式/索引色 |
| 清晰度 | 母版足够大，运行图按实际显示尺寸导出，禁止小图强拉伸 |
| 命名 | 小写英文、数字、下划线；禁止中文、空格、特殊符号 |
| 配置 | 正式资源必须同步 `assets.json`，并按类型同步 `ui_assets.json` 或 `game_assets.json` |
| 审核 | 技术门禁通过后，还要人工确认无文字、无血腥、无暗黑恐怖、无侵权联想 |

UI 文本必须走：

```text
assets/resources/config/text.json
Label / LocalizedLabel
```

图片只负责图形，不承载文本。

## 2. 目录规范

正式运行资源只允许放在：

```text
assets/resources/textures/
├── backgrounds/
├── bosses/
├── characters/
├── effects/
├── icons/
├── monsters/
├── tiles/
└── ui/
```

母版和中间产物放在：

```text
art_source/textures_review/
├── master/                 # 高质量母版，允许较大
├── runtime_candidates/     # 压缩后的运行时候选
├── rejected/               # 失败、违规、糊、带字、带血资源
└── contact_sheets/         # 批量验收图
```

### 2.1 3D 资源目录（Blender→glb→Prefab）

3D 资产不落在 `textures/`，独立目录（详见 `ART_RESOURCE_RULES.md §16.4`）：

```text
assets/resources/
├── models/          # 导出 .glb（角色 / 怪物 / Boss / 特效 / 3D tile）
├── prefabs/         # Cocos 导入 .glb 产出的 .prefab
├── textures/        # 3D 贴图走 ASTC 压缩（非 2D RGBA 运行时图）
└── config/          # 每个 3D 资产一个 .assetmeta.json（manifest）
```

母版与中间产物建议放在：

```text
art_source/models_review/
├── master/                 # 高模 / 源工程
├── runtime_candidates/     # 导出 .glb 候选
└── rejected/               # 未过 asset_validate 的资产
```

禁止事项：

| 禁止 | 原因 |
|---|---|
| 把 raw/master/rejected 放进 `assets/resources/textures` | 会进包体，污染运行时资源 |
| 从外部复制 `.meta` 到 textures | 可能破坏 Cocos UUID 和 SpriteFrame |
| 直接替换正式目录且不跑校验 | 容易缺文件、路径错、P 模式、尺寸错 |
| 让资源图自带文字 | 文本必须走 `text.json` + Label |

## 3. 命名规范

统一使用：

```text
{category}_{subject}_{action_or_state}.png
```

或：

```text
{prefix}_{semantic_name}.png
```

允许：

```text
warrior_attack.png
bg_combat_forest.jpg
icon_skill_dash.png
fx_reaction_burn.png
tile_forest_floor.png
btn_default.png
```

禁止：

```text
Warrior Attack.png
按钮.png
icon-skill-dash.png
new image (1).png
```

常用路径：

| 类型 | 路径 | 命名 |
|---|---|---|
| 角色 | `textures/characters/warrior/warrior_attack.png` | `{character}_{action}` |
| 怪物 | `textures/monsters/forest/slime_idle.png` | `{monster}_{action}` |
| Boss | `textures/bosses/main/forest/forestguardian_idle.png` | `{boss}_{action}` |
| 背景 | `textures/backgrounds/bg_combat_forest.jpg` | `bg_{roomType}_{zone}` |
| Tile | `textures/tiles/forest/tile_forest_floor.png` | `tile_{zone}_{terrain}` |
| 特效 | `textures/effects/combat/fx_hit_normal.png` | `fx_{effectName}` |
| 图标 | `textures/icons/items/icon_item_key.png` | `icon_{group}_{name}` |
| UI | `textures/ui/common/btn_default.png` | `{component}_{state}` |
| UI 展示区 | `textures/ui/create/create_model_stage.png` | `{panel}_{purpose}` |

> **2D 角色部件化（2026-07-11 起生效）**：`characters/*` 不再按 `{character}_{action}.png` sprite sheet 生成。改为每个职业 1 张母版（`art_source/textures_review/master/characters/{character}/reference/{character}_rig_master.png`），由 `tools/cut_character_parts.py` 从同一张母版切割部件到 `assets/resources/textures/characters/{character}/parts/*.png`。手武器第一版合并（archer=`arm_r_weapon`，不再拆 `arm_r`+`bow`）。完整规格见 `docs/角色部件化母版标准姿势规范.md` 与 `docs/角色部件化母版切割与精确拼装方案.md`。旧 sprite sheet 行（如 `character.warrior.attack` 注册为 `sprite_sheet`）仅作为历史参考，新流程用 `character_parts.json` / `character_rigs.json` / `character_part_animations.json` 驱动。

### 3.1 3D 资源命名（双轨）

3D 资源使用 **大写前缀 + PascalCase**，与 2D 小写蛇形不同：

```text
{PREFIX}_{PascalName}[_{Variant}]
CHR_  角色    MON_  怪物    BOSS_ Boss
FX_   特效    TILE_ 3D地块
```

命名正则（asset_validate 强制）：

```text
^(CHR|MON|BOSS|FX|TILE)_[A-Za-z0-9]+(_[A-Za-z0-9]+)?$
```

示例：`CHR_Warrior_A.glb`、`BOSS_AbyssLord_FINAL.glb`、`FX_Fireball_Boss.glb`、`TILE_Forest_Floor.glb`。每个 3D 资产在 `config/` 下配套同名 `.assetmeta.json`（manifest），登记 tri/bones/sockets/colliders/depends/lifecycle 等，字段模板见 Layer 2 的 `manifest`。

## 4. 格式与体积预算

| 类型 | 运行时格式 | 模式 | 推荐运行尺寸 | 硬上限 | 目标体积 |
|---|---|---|---:|---:|---:|
| 全屏背景 | JPG | RGB | 1280x720 或 960x540 | 1280x720 | 80-180KB |
| 主界面背景 | PNG/JPG | RGB/RGBA | 1280x720 | 1280x720 | PNG <= 300KB / JPG <= 180KB |
| 面板底板 | PNG | RGBA | 64x64 / 128x128 九宫格 | 256x256 | <= 24KB |
| 按钮 | PNG | RGBA | 192x64 / 256x80 | 384x128 | <= 24KB |
| UI 小图标 | PNG | RGBA | 64x64 / 128x128 | 256x256 | <= 32KB |
| 创建角色展示区 | PNG/JPG | RGBA/RGB | 512x288 / 640x360 | 640x360 | PNG <= 120KB / JPG <= 100KB |
| 角色单帧 | PNG | RGBA | 192x192 / 256x256 | 256x256 | <= 80KB |
| 角色序列帧 | PNG | RGBA | 192xN / 256xN | 单帧 256x256 | <= 300KB |
| 普通怪物 | PNG | RGBA | 128x128 | 192x192 | <= 80KB |
| 精英怪 | PNG | RGBA | 192x192 | 256x256 | <= 120KB |
| Boss 单帧 | PNG | RGBA | 256x256 | 384x384 | <= 180KB |
| Boss 序列帧 | PNG | RGBA | 256xN | 单帧 256x256 | <= 500KB |
| Tile | PNG | RGBA/RGB | 32x32 / 64x64 | 64x64 | <= 8KB |
| 特效序列帧 | PNG | RGBA | 192x768 常见 4 帧 | 单帧 192x192 | <= 120KB |
| 道具/技能/套装图标 | PNG | RGBA | 128x128 | 192x192 | <= 32KB |

说明：

- PNG 必须是 RGBA 或 RGB，禁止 P 模式索引色。
- 透明资源优先 RGBA。
- 背景无透明需求时优先 JPG，对 TapTap 安卓包体更友好。
- 体积硬上限不能通过牺牲清晰度硬压。应先优化尺寸、构图、颜色数量、透明边界，再压缩。

> **3D 预算不在本节定义。** 角色 / 怪物 / Boss / 特效 / 3D tile 的三角面、骨骼数、纹理尺寸、LOD 级数、粒子预算等数值，唯一权威源为 `art_quality_budget.json → rules3d`（对齐 `ART_RESOURCE_RULES.md §16.3`），逐资源明细见 Layer 2（`AI资源制作规范_MachineSpec.yaml/.json`）的 `budget` 字段（target/min/max）。入库前由 `asset_validate.py` 按 rules3d 校验。

## 5. 当前反例：create_model_stage.png

当前资源：

```text
路径: assets/resources/textures/ui/create/create_model_stage.png
尺寸: 1312x736
模式: RGB
大小: 840KB
```

问题：

| 问题 | 影响 |
|---|---|
| 尺寸过大 | 只是模型展示小舞台，不需要全屏级尺寸 |
| RGB 且无透明 | 不利于叠加到面板中，也容易遮挡 UI |
| 840KB | 对 TapTap 安卓 UI 小资源严重超预算 |
| 语义像完整背景 | 容易被误用为主背景或面板背景 |

正式规范：

```text
路径: assets/resources/textures/ui/create/create_model_stage.png
尺寸: 512x288 或 640x360
模式: PNG RGBA；如果完全矩形不透明，可用 JPG RGB
目标大小: PNG <= 80KB，JPG <= 70KB
硬上限: PNG <= 120KB，JPG <= 100KB
用途: 创建角色模型展示区小舞台
配置 key: ui.create.model_area
```

推荐提示词：

```text
PROJECT STYLE: bright high-definition cartoon animal adventure UI, soft forest stage, rounded decorative frame, cheerful and clean.
ASSET PURPOSE: character creation model preview stage background, used behind a selected animated animal hero.
CANVAS: 640x360 runtime image, center area clear for character, decorative corners and soft depth, no text.
SUBJECT: small forest stage or meadow display frame with leaves, stars, soft light, cute adventure mood.
COMPOSITION: center empty area, decorations around edges, no object covering the character.
TECH: PNG RGBA if transparent edges are needed, otherwise opaque JPG; no baked text, no logo, no watermark.
SCRIPT-ONLY RISK FILTER: text, letters, numbers, logo, watermark, unsafe horror or injury terms, blurry, busy center, character already inside. Do not send risk-filter words to the image API prompt; use them only in local lint rules.
```

> 注：§6.1–§6.9 仅覆盖 **2D 贴图** 资源。3D 资源（角色 / 怪物 / Boss / 特效 / 3D tile）走 Blender 建模管线，不在此节，制作入口见 §6.10。

## 6. 分类生成规范

### 6.1 Characters

路径：

```text
assets/resources/textures/characters/{characterId}/{characterId}_{action}.png
```

动作：

```text
idle, walk, attack, skill, dodge, hit, death
```

规格：

| 项 | 规格 |
|---|---|
| 母版 | 每帧 512x512 或更高 |
| 运行时单帧 | 192x192 或 256x256 |
| 序列帧 | 竖排 sprite sheet |
| 背景 | 透明 RGBA |
| 构图 | 角色居中，脚底对齐，四周留 8-12% 安全边距 |
| 风格 | Q 版动物职业，大眼睛，圆润盔甲/服装，清晰轮廓 |

提示词模板：

```text
Create a full-body cute cartoon animal adventurer character sprite for a mobile fantasy adventure game.
Character: {animal} {class}. Action: {action}.
Bright friendly high-definition cartoon animal style, round shapes, big expressive eyes, clean silhouette, readable at small size, transparent background.
For sprite sheet: vertical sheet with {frame_count} frames, each frame exactly {frame_w}x{frame_h}, same character design and same scale in every frame.
No text, no letters, no watermark. Use clean polished gear, soft sparkle effects, toy-like props, warm friendly cartoon adventure mood.
```

验收：

- 每帧角色比例一致。
- 不允许每帧换装、换脸、换动物。
- 攻击动作可以有武器，但不能有血液、穿刺、伤口。
- `death` 动作必须是化光、倒下、星光消散，禁止尸体、血、骷髅。

### 6.2 Monsters

路径：

```text
assets/resources/textures/monsters/{zone}/{monsterId}_idle.png
```

规格：

| 类型 | 运行尺寸 | 体积 |
|---|---:|---:|
| 普通怪物 | 128x128 | <= 80KB |
| 精英怪 | 192x192 | <= 120KB |

提示词模板：

```text
Create a cute fantasy creature enemy sprite for a mobile cartoon animal adventure game.
Zone theme: {zone_theme}. Creature: {monster_description}.
Friendly readable cartoon monster, not scary, not realistic, bright colors, clean silhouette, transparent background, centered full body.
No text, no letters, no watermark. Use rounded friendly creature shapes, bright readable colors, soft magical details, playful non-scary adventure mood.
```

验收：

- 怪物可以有敌意，但必须偏卡通可爱。
- 深渊、墓穴类也不能暗黑恐怖化。
- 禁止骷髅怪、血肉怪、断肢、真实器官。

### 6.3 Bosses

路径：

```text
assets/resources/textures/bosses/main/{zone}/{bossId}_{action}.png
assets/resources/textures/bosses/miniboss/{zone}/{bossId}_idle.png
```

规格：

| 类型 | 运行尺寸 |
|---|---:|
| Main Boss 单帧 | 256x256 |
| Main Boss 序列帧 | 256xN |
| Miniboss | 192x192 |

提示词模板：

```text
Create a large cute cartoon animal fantasy boss sprite for a mobile adventure game.
Boss theme: {boss_theme}. Action: {action}.
Powerful but friendly cartoon design, bright readable colors, rounded forms, expressive animal features, clear silhouette, transparent background.
No text, no logo, no watermark. Use powerful but friendly animal fantasy design, polished clean props, bright magical energy, rounded readable shapes.
```

验收：

- Boss 要大、有辨识度，但不要恐怖。
- phasechange 用光环、星光、能量变化表现，不用血、骨、黑暗仪式。
- 文件名里如果含 death，提示词正向描述改成 `magical sparkle dissolve` 或 `fade into light`。

### 6.4 Backgrounds

路径：

```text
assets/resources/textures/backgrounds/bg_{roomType}_{zone}.jpg
```

roomType：

```text
combat, event, room_rest, room_healing, room_shop, room_treasure, room_upgrade
```

规格：

| 项 | 规格 |
|---|---|
| 母版 | 1536x864 或 1920x1080 |
| 运行时 | 1280x720 JPG，必要时 960x540 |
| 体积 | 80-180KB，复杂背景最大 220KB |
| 模式 | RGB |
| 构图 | 中央留可玩区域，不遮挡角色、怪物、HUD |

提示词模板：

```text
Create a bright cartoon animal fantasy game background for a mobile adventure game.
Scene: {room_type} in {zone_theme}.
Warm friendly lighting, readable foreground/midground/background separation, no characters as main subjects, no text, no signs with letters, no watermark.
Keep the center gameplay area visually calm and not too busy.
Use warm safe exploration mood, clean scenery props, plants, crystals, stones, lamps, clouds, stars, coins, gems, books, tents, tools, and soft magic light only.
```

验收：

- 不允许血迹、心脏、骷髅、墓碑文字、英文招牌。
- 商店、休息、治疗房间必须温暖安全。
- 战斗背景也不能过暗、过恐怖。

### 6.5 Tiles

路径：

```text
assets/resources/textures/tiles/{zone}/tile_{zone}_{terrain}.png
```

terrain：

```text
floor, wall, highground, thorn
```

规格：

| 项 | 规格 |
|---|---|
| 运行时 | 32x32 或 64x64 |
| 体积 | <= 8KB |
| 生成方式 | 优先程序化生成，其次 AI 母版后程序化修边 |
| 拼贴 | 四边必须可无缝拼接，中心不能有强热点 |

提示词模板仅用于母版参考，不建议直接用 AI 输出最终 tile：

```text
Create a seamless top-down cartoon terrain tile texture for a mobile adventure game.
Zone: {zone_theme}. Terrain type: {terrain}.
Simple readable pattern, no central object, no strong hotspot, edges must tile seamlessly.
No text, no letters, no numbers. Use terrain material only, no icons, no center symbols, no object-like decorations.
```

验收：

- 必须通过 tile 可拼贴质量检查。
- 不允许中心出现宝石、火焰、角色、道具等强主体。
- 不允许 32x32 里塞复杂插画。

### 6.6 Effects

路径：

```text
assets/resources/textures/effects/{group}/fx_{effectName}.png
```

group：

```text
combat, reactions, relics, ui
```

规格：

| 项 | 规格 |
|---|---|
| 单帧 | 192x192 |
| 序列帧 | 竖排 192x768，4 帧常用 |
| 模式 | RGBA |
| 背景 | 真透明 |
| 体积目标 | <= 60KB |
| 体积警戒 | <= 90KB |
| 体积硬上限 | <= 120KB |

提示词模板：

```text
Create a clean abstract cartoon visual effect sprite sheet for a mobile animal adventure game.
Effect: {effect_description}.
Vertical sprite sheet with {frame_count} frames, each frame {frame_w}x{frame_h}. Transparent background, crisp edges, readable shape, limited colors.
Abstract energy only; do not create physical props unless the asset name explicitly requires one.
No text, no letters, no watermark. Use pure abstract light, clean sparkle particles, soft elemental colors, and transparent empty surroundings.
```

验收：

- 特效不能糊，边缘要清晰。
- 不允许黑底、绿幕、洋红残留。
- 如果是时间特效，不能生成沙漏道具本体，除非配置明确需要道具图标。

### 6.7 Icons

路径：

```text
assets/resources/textures/icons/{group}/icon_{name}.png
```

group：

```text
items, relics, skills, elements, buffs, sets
```

规格：

| 项 | 规格 |
|---|---|
| 母版 | 512x512 |
| 运行时 | 128x128 |
| 模式 | RGBA |
| 体积目标 | <= 16KB |
| 体积硬上限 | <= 28KB |
| 构图 | 主体占 70-82%，四周透明安全边距 |

提示词模板：

```text
Create a clean RPG item icon for a mobile cartoon animal adventure game.
Icon subject: {item_or_skill_description}.
Single object or simple symbol, centered, transparent background, thick friendly cartoon outline, bright colors, readable at 64x64.
No face unless the item is explicitly a character portrait. No text, no letters, no watermark. Use clean symbolic item shapes, crystals, leaves, stars, ribbons, gems, polished toy-like equipment, and friendly readable silhouettes.
```

验收：

- 道具图标必须是道具，不要生成头像、表情、人脸。
- 吸血、狂暴、生命、死亡相关词要改成安全表达，例如生命水晶、能量环、红橙斗志光。
- frenzy axe 不得有血液；life/link 不得有真实心脏。

## 7. 通用提示词结构

所有新增资源的 prompt 必须使用以下结构：

```text
[STYLE]
Bright friendly high-definition cartoon animal adventure game art.
Round shapes, clean silhouette, readable at mobile size, colorful but not noisy.

[ASSET]
Category: {category}
Resource id: {semantic_key}
File path: {runtime_path}
Purpose: {in_game_usage}

[COMPOSITION]
{specific composition, frame count, transparent/background, safe margin}

[TECHNICAL]
Output target: {width}x{height}
Format: PNG RGBA / JPG RGB
No baked text. No watermark. No logo.

[SAFETY]
WeChat mini-game safe. Use only positive safe visual language: warm camp, flowers, leaves, stars, crystals, gems, soft magic, polished toy-like gear, rounded friendly shapes, and cheerful adventure mood.
```

危险词规则：

- 正向 prompt 中不要出现 blood/skull/bone/heart/death/horror 等会诱导 AI 的词。
- 如果 API 只有单一 prompt 字段，不要把危险词作为 `no xxx` 负面词一起发送；危险词只放在脚本本地 `riskFilter` / `lint-prompts` 中。
- 实际传给 Agnes 的 prompt 只保留正向安全画面描述，以及 `no text/no letters/no words/no numbers` 这类文本约束。
- 如果必须描述功能，例如 `death` 文件名，正向文案改为 `adventure failed settlement`、`fade into light`、`magical sparkle dissolve`。
- 负面 prompt 可以列危险词。

通用负面提示词：

```text
text, letters, words, numbers, logo, watermark, skull, skeleton, bone, blood, gore, organ, heart, corpse, horror, dark fantasy, realistic violence, realistic wound, scary emblem, English text, Chinese text, blurry, low resolution, dirty grunge
```

## 9. 新增资源标准流程

1. 确认资源用途、路径、尺寸、格式、体积预算。
2. 在 `prompts.json` 添加或更新该资源的精确 prompt。
3. 生成母版到 `art_source/textures_review/master/...`。
4. 人工审查母版：风格、文字、血腥、构图、版权风险。
5. 导出运行时候选到 `art_source/textures_review/runtime_candidates/...`。
6. 检查 PNG 模式、尺寸、透明、体积、sprite sheet 帧数、tile 拼贴。
7. 通过后复制到 `assets/resources/textures/...`。
8. 更新 `assets.json`。
9. UI 资源更新 `ui_assets.json`，如需静态绑定再更新 `ui_skin_bindings.json`。
10. 非 UI 资源更新 `game_assets.json` 和对应业务配置。
11. 运行：

```bash
npm.cmd run validate:all
```

12. 打开 Cocos 预览，确认控制台没有：

```text
node not found
apply failed
sprite load failed
asset map not loaded
text config not loaded
```

### 9.0 3D 资源入库分支

3D 资产走独立分支（与 2D 不混）：

1. 在 `prompts_dim.json` 登记资产参数；Blender 建模并导出 `.glb` 到 `assets/resources/models/<cat>/`。
2. Cocos 导入 `.glb` 产出 `.prefab` 到 `assets/resources/prefabs/<cat>/`。
3. 编写 / 生成同名 `.assetmeta.json` 到 `assets/resources/config/`（含 tri/bones/sockets/colliders/depends/lifecycle）。
4. 运行技术门禁（**未全绿禁止入库**）：
   ```bash
   python tools/asset_validate.py assets/resources/models/<cat>/<NAME>.assetmeta.json --budget assets/resources/config/art_quality_budget.json
   ```
5. 扫描整目录并出报告：
   ```bash
   python tools/asset_validate.py --scan assets/resources/models/ --report art_source/models_review/validate_report.csv
   ```
6. 自测工具正确性：
   ```bash
   python tools/asset_validate.py --self-test
   ```
7. 门禁全绿 + 人工视觉确认（toon 材质 / 描边 / 色饱 / 轮廓 / 性能 tier）后，视为可入库。
8. 任何 FAIL 项必须先修复，或将资产 `lifecycle` 显式降级（如 `已弃用`）后再处理。

## 9.1 脚本化低 API 消耗流程

结论：全流程必须脚本化，但不能“一键全量生图”。正确方式是“本地审计优先、分类小批量生成、失败原因缓存、人工只看少量 contact sheet”。这样可以显著减少 Agnes API 调用次数。

### 9.1.1 哪些步骤必须脚本化

| 阶段 | 是否调用 API | 脚本职责 | 说明 |
|---|---:|---|---|
| 资源清单生成 | 否 | 从 prompts.json / assets.json / 磁盘目录生成待处理 manifest | 找出缺失、过期、尺寸错、格式错、未注册资源 |
| prompt 风险扫描 | 否 | 检查 prompt 是否包含高风险正向词 | 出现 blood/skull/heart/death 等正向诱导词时直接阻断，不调用 API |
| 已有资源复用判断 | 否 | 按路径、尺寸、模式、hash、promptHash 判断是否可复用 | 已有合格资源不得重复生成 |
| 程序化生成 | 否 | 仅 tile 与纯几何线条/遮罩本地生成 | UI 按钮/面板/卡片/槽位已统一改 AI 出图（见 §11.3），避免用 AI 仅限 tile 等可控几何资源 |
| AI 母版生成 | 是 | 只对缺失且无法程序化生成的资源调用 API | 必须支持 --category、--limit、--resume-failed |
| 后处理导出 | 否 | RGBA/RGB 转换、裁切、缩放、压缩、JPG 导出 | 不能靠反复 API 生成解决体积问题 |
| 技术门禁 | 否 | 尺寸、模式、透明、体积、sprite sheet、tile 拼贴、注册表 | 失败进入 rejected 或 retry queue |
| 视觉验收 | 否 | 生成 contact sheet，人工只看缩略图和少量重点图 | 人工确认文字、血腥、风格、构图 |
| 正式替换 | 否 | approved 资源复制到 assets/resources/textures/ 并更新配置 | 禁止手工散复制 |

### 9.1.2 API 调用准入规则

只有同时满足以下条件，脚本才允许调用 Agnes API：

1. targetPath 不存在，或现有文件未通过技术门禁。
2. promptHash 与上次成功生成记录不同，或上次结果明确被人工标记为 rejected_regenerate。
3. 资源类型不是可程序化生成类型。
4. prompt 风险扫描通过。
5. 本次命令明确指定了分类，例如 --category ui/create，不能默认全量执行。
6. 没有超过本批次预算，例如 --limit 10 或 --max-api-calls 20。

推荐状态字段：

~~~json
{
  "assetId": "textures/ui/create/create_bg",
  "targetPath": "assets/resources/textures/ui/create/create_bg.jpg",
  "category": "ui/create",
  "promptHash": "sha256:...",
  "fileHash": "sha256:...",
  "status": "approved",
  "source": "ai|procedural|manual|reused",
  "lastError": "",
  "apiCalls": 1,
  "updatedAt": "2026-07-07"
}
~~~

状态流转：

~~~text
planned -> blocked_prompt_risk -> needs_manual_prompt
planned -> reused -> approved -> imported
planned -> procedural_generated -> technical_passed -> visual_review_pending -> approved -> imported
planned -> api_generated -> postprocessed -> technical_passed -> visual_review_pending -> approved -> imported
planned -> api_generated -> rejected_regenerate
planned -> rejected_do_not_generate
~~~

### 9.1.3 分类执行命令规范

后续生成脚本必须支持分类执行，不能默认全量重做。推荐命令形态：

~~~bash
# 只审计，不生成，不消耗 API
python tools/art_pipeline.py audit --category ui/create

# 只检查 prompt 风险，不生成
python tools/art_pipeline.py lint-prompts --category ui

# 生成验收图，不生成新图
python tools/art_pipeline.py contact --category ui/create

# 先试 3 张，最多消耗 3 次 API
python tools/art_pipeline.py generate --category ui/create --limit 3

# 正式生成某一类，但仍限制最大 API 调用次数
python tools/art_pipeline.py generate --category ui/create --max-api-calls 20

# 只重试上次失败且 prompt 已变化的资源
python tools/art_pipeline.py generate --category ui/create --resume-failed

# 技术门禁，不生成
python tools/art_pipeline.py validate --category ui/create

# 人工确认后入库
python tools/art_pipeline.py import --category ui/create --only-approved
~~~

3D 资源校验 / 入库使用 `asset_validate.py` + `art_pipeline import`（路由 models/prefabs），不混用 2D 出图命令：

~~~bash
# 单个 3D 资产 manifest 门禁
python tools/asset_validate.py assets/resources/models/characters/CHR_Warrior_A.assetmeta.json
# 扫描整目录并出报告
python tools/asset_validate.py --scan assets/resources/models/ --report art_source/models_review/validate_report.csv
# 工具自测
python tools/asset_validate.py --self-test
# 入库（路由 models/prefabs）
python tools/art_pipeline.py import --category characters --only-validated
~~~

如果暂时没有统一的 art_pipeline.py，可以先用现有脚本组合，但原则不变：先 audit/validate，再决定是否 generate。

### 9.1.4 资源类型与生成策略

| 类型 | 首选策略 | API 使用建议 |
|---|---|---|
| Tile | 程序化生成 + 拼贴检测 | 默认不调用 API |
| 通用按钮/小面板/卡片/槽位 | AI 母版出图（prompts.json） | 调用 API（UI 已统一改 AI，见 §11.3） |
| 全屏 UI 背景 | AI 母版 + JPG/PNG 运行版导出 | 允许小批量调用 API |
| 角色/怪物/Boss | AI 母版 + sprite sheet 校验 | 允许按角色/区域调用 API |
| 图标 | AI 或人工模板，但必须 prompt 风险扫描 | 小批量调用 API，失败不要无限重试 |
| 特效 | 优先程序化粒子/矢量重绘，其次 AI 母版 | 避免靠 API 反复赌透明和体积 |
| 背景压缩 | JPG/WebP 候选导出，本地压缩 | 不调用 API |

### 9.1.5 减少 API 消耗的硬规则

1. 禁止默认全量生成 418 张。脚本默认只能 audit，生成必须显式传 generate。
2. 禁止覆盖 approved 文件，除非传 --overwrite-approved，并要求额外确认。
3. 每张资源保存 promptHash。prompt 没变且已有 rejected 结果时，不重复调用 API。
4. 同一资源连续失败 2 次后进入 blocked_prompt_risk 或 needs_manual_prompt，先改 prompt，不继续烧 API。
5. 体积超限优先本地后处理：裁切透明边、减少尺寸、JPG 背景、调色板/颜色量控制、PNG 压缩。不能通过反复生成碰运气。
6. 安全违规直接改 prompt，不能重试同一 prompt。
7. 分类执行顺序建议：ui/common、ui/create、ui/main、backgrounds、characters、monsters、bosses、effects。每批先 --limit 3。
8. 生成脚本必须输出 batch_report.csv、manifest.json、contact_sheet.png，方便另一个对话接手。

### 9.1.6 推荐统一脚本结构

~~~text
tools/art_pipeline.py
  audit              # 生成/更新 manifest，不调用 API
  lint-prompts       # prompt 风险扫描，不调用 API
  plan               # 输出本批次会消耗多少 API
  generate           # 只生成缺失/需重做资源，可限量
  postprocess        # 本地导出 runtime candidates
  validate           # 技术门禁
  contact            # 生成验收图
  approve            # 合并人工结果
  import             # 复制到正式目录并更新配置
  # --- 3D 资产（不走 AI，走 asset_validate + import 路由）---
  asset_validate     # 3D 技术门禁：python tools/asset_validate.py <manifest> [--budget] / --scan / --self-test
  import_3d          # 路由 models/prefabs：python tools/art_pipeline.py import --category <cat> --only-validated
~~~

核心伪代码：

~~~python
def should_call_api(asset, state):
    if asset.category in PROCEDURAL_CATEGORIES:
        return False
    if prompt_has_risk(asset.prompt):
        asset.status = "blocked_prompt_risk"
        return False
    if existing_file_passes_gate(asset):
        asset.status = "reused"
        return False
    prev = state.get(asset.id, {})
    if prev.get("promptHash") == hash_prompt(asset.prompt):
        if prev.get("status") in ["approved", "technical_passed", "visual_review_pending"]:
            return False
        if prev.get("failCount", 0) >= 2:
            asset.status = "needs_manual_prompt"
            return False
    return True
~~~

## 12. 常见问题与处理

| 问题 | 原因 | 处理 |
|---|---|---|
| 图片加载后还是灰底 | 资源未绑定、Sprite 禁用、资源表未加载、节点路径错误 | 查 `ui_skin_bindings.json` 和控制台 |
| 图被缩进模型展示区 | 全屏背景误用于局部 UI | 新建局部资源，如 `create_model_stage` |
| PNG Cocos 搜不到 SpriteFrame | P 模式索引色或 meta 未刷新 | 转 RGBA，重新导入 |
| 图片糊 | 母版太小或运行时强拉伸 | 提高母版，运行时按目标尺寸导出 |
| AI 生成文字 | prompt 不够明确 | 加 `no baked text/no letters/no watermark`，失败即 rejected |
| 出现血/心脏/骷髅 | 正向 prompt 诱导了危险词 | 改安全语义，用水晶、星光、能量表达 |
| Tile 不可拼 | AI 直接生成强中心主体 | 程序化生成或程序化修边 |
| 特效有黑底/绿幕残留 | 抠图方案错误 | 使用 RGBA 真透明或程序化重绘 |
| sprite sheet 贴整张 | 未按帧切片 | `game_assets.json` 配 `frameWidth/frameHeight/frameCount/layout` |
| 资源加载失败 | `assets.json` 或语义 key 缺失 | 补齐注册并运行校验 |

## 正式上线资源验收标准

所有进入 `assets/resources/textures/` 的资源都按正式上线标准验收，不允许“临时可用”“后面再换”的灰区。母版可以保留探索稿，但运行时资源必须满足以下要求。

### 1. 通用硬标准

| 项 | 标准 |
|---|---|
| 风格 | 统一为明亮、温暖、卡通动画风、动物幻想冒险风，不使用像素风、暗黑风、写实恐怖风 |
| 文本 | 图片内不允许出现任何文字、字母、数字、伪文字、Logo、水印、签名；所有文本由 Cocos Label/TextManager 渲染 |
| 审核安全 | 只能使用花朵、叶子、星星、水晶、蘑菇、营地、木质、布料、柔和光效等安全元素 |
| 清晰度 | 运行时尺寸必须与实际显示尺寸匹配，不允许小图硬拉伸；sprite sheet 必须按帧尺寸切片显示 |
| 色彩 | 不允许异常色块、色偏、过度压缩导致的脏边、糊边、断层 |
| 入库 | 正式资源必须通过脚本校验和肉眼验收后才能覆盖 `assets/resources/textures/` |

### 2. UI 按钮标准

按钮是承载运行时文字的控件，中心区域必须干净。星星、花朵、水晶等装饰只能放在边角或左右两侧，不能压在文字核心区域。

| 项 | 标准 |
|---|---|
| 中心区域 | 保持低纹理、低对比、无强图案，保证中文 Label 可读 |
| 装饰 | 只允许边框、角花、左右小装饰；不要在按钮正中放大星星、大水晶、大徽章 |
| 状态区分 | default/hover/active 使用亮度、描边、柔和光效区分，不改变按钮语义图案 |
| 透明 | 小按钮必须是真 RGBA 透明，边缘有 alpha 过渡 |
| 透明像素 RGB | alpha=0 的像素不能残留纯青、纯绿、纯品红；需要做 alpha bleed，颜色延展为邻近边缘色 |

### 3. UI 面板标准

承载文字的大面板不能是纯白大块，也不能是假透明棋盘格。正式面板应为柔和浅色内容区或真透明装饰框，二者只能选一种。

| 面板类型 | 标准 |
|---|---|
| 文本承载面板 | 中心使用浅奶油色、羊皮纸、柔光半透明内容区，不允许纯白矩形 |
| 装饰框 | 只有边框和角饰可见，中心应真透明；不得把棋盘格画进图片 |
| 背景一体化面板 | 与整屏背景同光源、同透视、同色调，不能像另贴一张图 |
| alpha | 边缘需要软过渡，不允许只有 0/255 硬切导致锯齿 |

### 4. 背景与一体化界面标准

主界面、创建角色、结算、商店、角色选择等整屏界面，如果追求一体化观感，应优先生成“整屏背景 + 预留功能区”，而不是用多个风格不同的小图叠贴。

| 项 | 标准 |
|---|---|
| 画面 | 1280x720 横屏基准，按安全区预留 UI 区域 |
| 预留区 | 预留标题区、角色展示区、按钮区、列表区，但不能画入文字 |
| 连贯性 | 同一界面的背景、展示台、按钮、面板必须同光源、同材质、同色调 |
| 压缩 | 背景可使用 JPG 候选，但 UI 按钮、图标、透明面板必须使用 PNG RGBA |

### 5. 图标与角色资源标准

| 类型 | 标准 |
|---|---|
| 图标 | 128x128 或配置指定尺寸，主体清晰、无文字、无头像误生成、无高风险符号 |
| 角色 | 动物角色风格统一，不能出现危险伤口或不安全符号；攻击帧必须动作一致，不允许每帧换设计 |
| 怪物/Boss | 可爱动物或幻想生物方向，避免恐怖暗黑方向；同一 sprite sheet 内角色身份必须一致 |
| 特效 | 保持清晰边缘和真透明，不允许糊成光团；序列帧每帧尺寸一致 |
| Tile | 必须程序化或人工保证可拼贴，不能依赖随机 AI 碰运气 |

### 6. 失败即重做规则

以下情况一律不能进入正式资源目录：

- 母版是 RGB 假透明，画面里出现棋盘格。
- 正式资源中心出现纯白大块、异常色块、青色/品红/绿色残留。
- 按钮中心有强图案，影响运行时文字识别。
- 图片内有任何可读或伪造文字。
- 风格变成暗黑、写实、像素、低清、过度压缩。
- sprite sheet 切片后每帧角色不一致、大小跳变、构图漂移。
- alpha 边缘硬切、透明像素 RGB 污染会在 Cocos 缩放时产生描边。

### 7. 入库流程要求

1. 先生成母版到 `art_source/textures_review/master/`。
2. 脚本检查尺寸、模式、alpha、透明像素 RGB、体积、命名、路径、assets.json 注册。
3. 人工看 contact sheet 或实际界面截图。
4. 通过后导出运行时版本到 `art_source/textures_export/runtime_replace/`。
5. 最后再覆盖 `assets/resources/textures/`。
6. 覆盖后重新运行资源注册检查和 Cocos 预览。

正式目录只放最终合格资源，不放试验稿、失败稿、母版稿。

## 3D 资源上线标准

3D 模型的上线标准与 2D 贴图不同，核心在**卡通渲染表现 + 性能 tier**。数值以 `ART_RESOURCE_RULES.md §16` / `rules3d` / Layer 2 为准。

### 1. 卡通渲染标准

| 项 | 标准 |
|---|---|
| 材质 | Toon / 卡通材质，非写实 PBR；明确描边（outline）宽度与阈值 |
| 颜色 | 颜色饱和度、明度符合卡通动物风，不偏灰暗写实 |
| 轮廓 | 角色 / 怪物 / Boss 轮廓清晰可读，小屏不糊 |
| 表现 | Boss 通过体型 / 光环 / 能量表现压迫感，不用血 / 骨 / 暗黑仪式 |

### 2. 性能 tier 标准

| 项 | 标准 |
|---|---|
| 三角面 | ≤ rules3d 上限（final boss 可更高，仍设硬顶） |
| 骨骼 | ≤ 规则上限，动画可复用 |
| 贴图 | 主贴图尺寸符合预算，ASTC 压缩 |
| LOD | 远景降级，级数符合规则 |
| 包体 | 单资产体积符合 TapTap 安卓包体预算 |

### 3. 失败即重做（3D）

- 三角面超 rules3d 硬顶。
- 缺必需 Socket / Collider 导致运行时挂点 / 受击失效。
- `depends` 断裂（prefab 引用缺失贴图 / 动画）。
- toon 材质写实化、描边丢失、颜色灰暗。
- `lifecycle` 非「已批准」即入库。

## 9-Slice 与分辨率自适应 UI 资源规范

> 本节**仅适用于 2D UI 贴图**；3D UI 资源不涉及 9-slice。

长条按钮、列表卡片、信息面板、输入框、奖励条、状态条等 UI 资源必须优先按 9-slice 设计。禁止用 AI 直接生成超长整图后在运行时继续拉伸。9-slice 的目标是：边角和装饰不变形，中间内容区可随分辨率和布局宽度伸缩。

### 1. 适用资源

| 类型 | 必须使用 9-slice 的资源 | 原因 |
|---|---|---|
| 长条卡片 | `ui/area/route_card_default.png`, `ui/area/route_card_locked.png`, `ui/event/choice_card.png`, `ui/log/stat_row.png` | 横向宽度随布局变化，直接整图会拉伸纹理和装饰 |
| 角色卡片 | `ui/character/character_card_default.png`, `ui/character/character_card_selected.png` | 列表宽度可能随屏幕缩放，选中态要保持边框清晰 |
| 小信息条 | `ui/shop/coin_panel.png`, `ui/equipment/set_counter_bg.png`, `ui/login/agreement_strip.png`, `ui/create/input_name.png` | 文本长度和语言可能变化，需要中心可扩展 |
| 常规按钮 | 所有 `btn_*.png`，除纯图标关闭按钮外 | 文字由 Cocos Label 渲染，按钮中心必须可读且不变形 |
| 大面板 | `*_panel.png`, `*_frame.png`, `card_frame_*.png` | 不同分辨率下内容区可能缩放，边框不能拉坏 |
| 状态条 | `hud_hpbar_frame.png`, `loading_bar.png`, `reward_strip.png` | 进度或奖励数量随布局变化 |

### 2. 资源母版尺寸与显示尺寸

9-slice 资源的文件尺寸不等于最终显示尺寸。资源文件只负责提供清晰的边角、边框和可伸缩中心，最终显示尺寸由 Cocos `UITransform` 控制。

| 资源类型 | 建议文件尺寸 | 典型显示尺寸 | 说明 |
|---|---:|---:|---|
| 小按钮 | `180x64` 或 `220x72` | `180-320w x 64-88h` | 左右端帽固定，中间横向伸缩 |
| 主按钮 | `260x88` | `260-420w x 76-96h` | 装饰在边角，不放中心图案 |
| 长条卡片 | `240x92` 或 `300x92` | `480-760w x 84-108h` | 不生成 720px 整图，避免拉伸感 |
| 输入框 | `240x72` 或 `320x72` | `360-640w x 64-88h` | 中心低对比、无强图案 |
| 信息条 | `240x64` 或 `320x64` | `260-620w x 56-80h` | 用于金币、统计、提示 |
| 大面板 | `480x320`, `640x420`, `760x520` | 按安全区缩放 | 大面板可直接做目标尺寸，但边框仍需 slice |
| 卡牌框 | `220x320` 或 `300x420` | `220-360w x 320-520h` | 顶/底/角固定，内容区伸缩 |

### 3. Slice 边距标准

所有 9-slice 资源必须在清单或配置中记录 `slice`，不要靠编辑器口头约定。

| 类型 | left | right | top | bottom | 中心要求 |
|---|---:|---:|---:|---:|---|
| 小按钮 `180/220x64/72` | 32 | 32 | 18 | 18 | 低对比纯净底色 |
| 主按钮 `260x88` | 44 | 44 | 22 | 22 | 中心 70% 留给 Label |
| 长条卡片 `240/300x92` | 44 | 44 | 18 | 18 | 羊皮纸或浅色内容区，不能有椭圆坑位 |
| 输入框 `320x72` | 36 | 36 | 18 | 18 | 中心干净，光效不能压文字 |
| 信息条 `260x64` | 36 | 36 | 16 | 16 | 可承载数字和短文本 |
| 大面板 `760x460` | 64 | 64 | 64 | 64 | 中心柔和内容区，不能纯白 |
| 卡牌框 `300x420` | 44 | 44 | 58 | 58 | 卡面主体可读，不拉伸角花 |
| 进度条框 `520x42` | 28 | 28 | 12 | 12 | 中间只作为 fill 裁剪区 |

如果资源实际尺寸不同，slice 可按比例调整，但端帽最小不得低于 28px，角装饰必须完全落在不可拉伸区域内。

### 4. 生成提示词要求

所有 9-slice 资源提示词必须明确：这是可拉伸 UI 组件，不是完整界面截图；边角固定，中间可伸缩；中心区域干净。

True alpha transparency, clean soft alpha edges, no fake checkerboard, no cyan/green/magenta background, no abnormal colored fringe. No text, no letters, no numbers, no pseudo-writing, no watermark. Not a full UI screenshot.
```

```

### 5. Cocos 使用规范

导入后必须配置为 sliced sprite：

```text
Sprite Type: SLICED
Trim: false
Size Mode: CUSTOM
UITransform size: 由布局系统控制
Label: 单独子节点覆盖在中心，不写进图片
```

代码创建或修复时必须保证：

```ts
import { Sprite } from "cc";

export function applySlicedSprite(sprite: Sprite, left: number, right: number, top: number, bottom: number): void {
    sprite.type = Sprite.Type.SLICED;
    sprite.sizeMode = Sprite.SizeMode.CUSTOM;
    const sf = sprite.spriteFrame;
    if (!sf) return;
    sf.insetLeft = left;
    sf.insetRight = right;
    sf.insetTop = top;
    sf.insetBottom = bottom;
}
```

正式做法是把 slice 数据写入配置，例如：

```json
{
  "ui.area.route_card": {
    "assetId": "textures/ui/area/route_card_default",
    "type": "sliced",
    "slice": { "left": 44, "right": 44, "top": 18, "bottom": 18 },
    "minSize": { "width": 240, "height": 92 },
    "defaultSize": { "width": 720, "height": 92 }
  }
}
```

### 6. 分辨率自适应规则

项目按横屏 `1280x720` 设计，但运行时必须适配不同宽高比。UI 资源不通过图片拉伸适配，而通过布局容器和 9-slice 适配。

| 层级 | 规则 |
|---|---|
| 全屏背景 | 使用 1280x720 JPG/PNG，按 cover 方式铺满，允许边缘裁切，不允许变形拉伸 |
| 安全区容器 | 内容放在 safe area 内，左右保留 48-80px，顶部/底部保留 32-56px |
| 面板 | 宽度使用 `min(maxDesignWidth, screenWidth * ratio)`，高度按内容布局，不硬拉图片 |
| 长条卡片 | 宽度跟随容器，Sprite 使用 SLICED；端帽不变形，中间伸缩 |
| 按钮 | 不同屏幕只改 UITransform 宽度，Label 居中；按钮底图使用 SLICED |
| 图标 | 图标不使用 9-slice，只等比缩放，不能非等比拉伸 |

推荐自适应尺寸公式：

```ts
const designW = 1280;
const designH = 720;
const safeW = Math.min(screenW, designW);
const panelW = Math.min(900, safeW * 0.78);
const routeCardW = Math.min(760, panelW - 48);
const buttonW = Math.max(180, Math.min(320, safeW * 0.18));
```
