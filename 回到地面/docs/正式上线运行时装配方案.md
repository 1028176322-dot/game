# 正式上线运行时装配方案

## 目标

本方案用于替代临时“跑起来即可”的场景兜底逻辑。正式上线标准是：

- 启动顺序唯一：配置、资源映射、全局管理器必须由统一 Bootstrap 完成。
- 场景契约清晰：场景节点由 Installer 统一装配，业务 Controller 不直接堆节点。
- 资源加载可审计：运行时资源通过 `assets/resources/config/assets.json` 的资源 ID 加载，不在业务代码里散落硬编码路径。
- 编辑器依赖最小：入口脚本挂载后能自动补齐必要系统，但缺失关键 UI 节点会明确报错。
- 微信小游戏友好：后续分包、远程资源、资源预加载都从 `AssetBundleService` 统一接入。

## 启动链路

正式链路：

1. `SplashUI.onLoad`
   - 调用 `GameManager.ensure()` 创建全局单例。
   - 调用 `GameBootstrap.ensure()` 确保启动器存在。
2. `GameBootstrap.startup`
   - `ConfigService.instance.loadAll()` 加载所有 JSON 配置。
   - `ConfigManager.getInstance().loadAll()` 同步旧接口兼容数据。
   - `AssetBundleService.instance.loadAssetMapFromResources()` 加载 `config/assets.json`。
3. `SplashUI._goToMain`
   - 等待 `GameBootstrap.ready === true`。
   - 有错误时阻止进主界面。
   - 正常时 `director.loadScene('main')`。
4. `MainSceneController`
   - 直接预览 main 时也会补加载配置和资源映射。
   - 自动绑定 `StartButton` 的点击事件。
   - 点击开始后初始化新局并进入 `dungeon`。
5. `DungeonSceneController`
   - 调用 `DungeonSceneInstaller.install()` 创建正式运行时节点。
   - 等待配置和资源映射完成。
   - 调用 `DungeonSceneInstaller.loadInitialArt()` 加载初始美术。
   - 接线战斗、地牢、UI、奖励、变异等系统。

## 场景节点契约

`DungeonSceneInstaller` 会确保以下节点存在：

```text
Canvas
├─ World
│  ├─ Grid
│  ├─ Player
│  └─ ExitDoor
├─ Systems
│  ├─ BattleManager
│  ├─ DungeonManager
│  ├─ RoomTransition
│  ├─ SkillSystem
│  ├─ UpgradeManager
│  ├─ ElementSystem
│  ├─ EquipmentSystem
│  └─ ItemSystem
└─ UIRoot
   ├─ VirtualJoystick
   ├─ BattleHUD
   ├─ DungeonMapUI
   ├─ SkillUI
   ├─ UpgradeUI
   ├─ DeathUI
   ├─ EquipmentUI
   └─ InventoryUI
```

后续如果你制作正式 Prefab，替换策略是：

- 保留节点名和组件类型不变。
- 在 Installer 中优先实例化 Prefab，再回填组件引用。
- 不要在 `DungeonSceneController` 里直接写 UI 创建逻辑。

## 资源加载规范

资源 ID 使用 `assets.json` 的 key，例如：

```ts
const PLAYER_IDLE = 'textures/characters/warrior/warrior_idle';
const frame = await AssetBundleService.instance.loadSpriteFrame(PLAYER_IDLE);
```

禁止在业务逻辑中这样写：

```ts
resources.load('textures/characters/warrior/warrior_idle/spriteFrame', SpriteFrame, ...);
```

允许使用 `resources.load` 的位置：

- `ConfigService` 加载 JSON 配置。
- `AssetBundleService` 加载 `config/assets.json`。

## 当前保留的开发期占位

正式上线前仍需替换：

- `BattleManager` 在 `monsterPrefab` 缺失时会生成 debug 圆形占位怪物，并在 `CC_DEBUG` 下警告。
- `GridManager` 在 `tilePrefab` 缺失时会用 `Graphics` 生成色块地面。
- `DungeonSceneInstaller` 会生成基础 HUD/摇杆占位控件。

这些占位不会阻塞运行，但上线前建议用正式 Prefab 替换。原则：

- 战斗怪物必须走 `monsterPrefab` 或正式 `MonsterVisualService`。
- 地块必须走 Tile Prefab 或 TileMap/程序化正式渲染。
- HUD/摇杆/按钮必须使用已审核通过的 UI 资源，不能依赖纯色块。

## 验收清单

每次替换资源或调整场景后执行：

```bash
npm run validate:all
```

编辑器内验收：

- 从 `splash.scene` 预览，能自动进入 `main.scene`。
- 点击主界面开始按钮，只触发一次进入地牢。
- `dungeon.scene` 中能看到玩家、网格、HUD、摇杆。
- 控制台没有红色错误。
- 控制台没有 `asset map not loaded`。
- 控制台没有 `unknown resource`。
- 控制台没有 `StartButton is missing`。

微信小游戏验收：

- 构建前执行 `npm run prebuild:wechat`。
- 首包体积和分包体积按 `bundle_budget_report.json` 检查。
- 所有资源加载都通过 `AssetBundleService`，不要新增直接路径加载。

## 后续正式化建议

优先级从高到低：

1. 新增 `MonsterVisualService`：根据怪物配置 ID 加载对应 SpriteFrame/AnimationClip。
2. 新增 `PlayerVisualService`：根据已选角色加载 idle/attack/death 动画。
3. 将 `DungeonSceneInstaller` 的占位 UI 替换为正式 UI Prefab。
4. 将 `GridManager` 的 Graphics fallback 替换为正式 tile 渲染。
5. 给 CI 增加静态扫描：禁止业务目录直接调用 `resources.load('textures/...')`。
