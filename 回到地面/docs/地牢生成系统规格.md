## 0. 设计师笔记

### 0.1 玩家体验目标

```
选择前往第一层:
    ↓
看到地牢地图: "入口 → 左边是战斗房, 右边有商店...
                      ↘ 尽头好像是 Boss 房"
    ↓
选择一个方向前进 → 进入对应房间 → 战斗/探索
    ↓
到达楼层末端 → 强化房(3选1) → 前往深层梯
    ↓
进入下一层 → 地图变换, 结构重新生成
```

**核心感受**：每次进入新楼层都像拆盲盒——"这一层的地图长什么样？什么方向更划算？"

### 0.2 与游戏类型的适配

| 维度 | 说明 |
|------|------|
| 实时自动攻击 | 地牢生成不受战斗模式影响——6×6 房间布局独立于战斗系统 |
| 微信小游戏 | 纯算法生成，无额外包体消耗 |
| 每局 5~15 分钟 | 每层设 5~7 个房间，总房间数约 15~21（3 层）至 25~35（5 层）|

### 0.3 好玩假设

> **"有限选择 + 未知回报 = 玩家愿意探索"**

玩家每层只有 2~3 个分支，选择少但每个都有意义。不是"40 个房间随便走"，而是"两个路口，左边有商店，右边可能是 Boss，你选哪个？"

## 1. 楼层结构（DAG 图）

### 1.1 楼层层次结构

```
每层 = 一个分层有向无环图（Layered DAG）

层数 = 3 层节点，每层包含 1~3 个房间

        layer 0        layer 1        layer 2
       (入口层)       (中间层)       (末端层)
        
          [入口]
         /  |  \
      [房A] [房B] [房C]
        |   / \    |
      [房D] [房E] [房F]
         \   |   /
          [强化房] ← 确保可达
            |
          [Boss房]
```

**核心结构**：
- **3 层节点**（入口层/中间层/末端层），每层 1~3 个房间
- **有向边**：从上层指向下层，不回头（DAG 保证无环）
- **汇聚点**：末端层汇聚至强化房 → Boss 房
- **Boss 房固定**：每层仅 1 个 Boss 房，必经之路

### 1.2 Mechanic: DAG 生成算法

```
Mechanic: DAG 楼层生成

Purpose: 生成一个有分支、有汇聚、无死胡同的地牢楼层地图
Player Fantasy: "前面有两条路，选哪条？"

Input: 种子(seed), 层数(floorNum 1~5), 难度级别
Output: 一个 DAG 结构: 房间列表 + 连接列表

生成步骤（7 步）:

1. 确定房间总数
   - 基础: 5 个房间（第 1 层）
   - 每深一层 +1 个房间
   - 公式: roomCount = 5 + floorNum     (1层=6, 3层=8, 5层=10)

2. 分配到 3 层
   - layer 0 (入口层): 1 个房间（入口）
   - layer 1 (中间层): ceil((roomCount - 2) × 0.6) 个房间
   - layer 2 (末端层): roomCount - 入口层数 - 中间层数
   
   验证: 末端层至少有 1 个房间（Boss 房）

3. 生成连接边
   - 入口层(L0) 的入口房间 → 连接到中间层(L1) 的 2~3 个房间
   - 中间层(L1) 每个房间 → 连接到末端层(L2) 的 1~2 个房间
   - 确保: L2 的每个房间至少被 1 个 L1 房间连接（无可达性死点）
   - 确保: 至少有 2 条不同路径从入口到达 Boss 前

4. 特殊房间替换
   - L2 中选 1 个房间 → 替换为强化房（必选层末节点）
   - L2 中最末节点 → 固定为 Boss 房
   - 剩余房间从类型池中随机分配

5. 房间类型分配（详见第 2 章）
   - 战斗房 55%
   - 宝箱房 12%
   - 回复房 10%
   - 商店房 10%
   - 事件房 5%
   - 强化房 8%（层末固定）

6. 验证
   - ✅ 所有房间可达（从入口 BFS 遍历全部节点）
   - ✅ 强化房可达（至少 1 条路径经过）
   - ✅ Boss 房可达
   - ✅ 无环（每一层只连向下一层，不连回）
   - ✅ 至少 2 条不同路径（入口→L1→L2）

7. 若验证失败 → 重新生成（最多重试 3 次，否则使用预设模板）

Edge Cases:
  - 5 层时房间数达 10 → 中间层(L1) 最多 5 个房间，L2 至少 3 个
  - 强化房如果因为概率分配被覆盖 → 强制替换 L2 中非 Boss 的一个房间
  - Boss 房不参与概率分配——它是固定的

Tuning Levers:
  - 每层房间数量 (当前: 5 + floorNum)
  - 入口分支数 (当前: 2~3)
  - 中间层到末端层的连接数 (当前: 1~2)

Dependencies: 种子系统（提供随机数源）
```

### 1.3 预设模板（当算法失败时的 fallback）

```
// 当随机生成连续 3 次验证失败时，使用预设模板
// 预设模板共 5 种，每层随机选一种

模板 A:  入口 ─→ 战斗 ─→ 战斗 ─→ 强化 ─→ Boss
                 ↘ 商店 → 战斗 ─↗

模板 B:  入口 ─→ 战斗 ─→ 战斗 ─→ 强化 ─→ Boss
                 ↘ 事件 → 回复 ─↗

模板 C:  入口 ─→ 战斗 ─→ 战斗 ─→ 强化 ─→ Boss
                 ↘ 宝箱 → 战斗 ─↗
                         宝箱 ─↗

模板 D:  入口 ─→ 战斗 ─→ 战斗 ─→ 强化 ─→ Boss
               ↘ 商店 ─→ 事件 ─↗
                 ↘ 回复 ─↗

模板 E:  入口 ─→ 战斗 ─→ 战斗 ─→ 强化 ─→ Boss
                 ↘ 宝箱 → 精英 ─↗
                         回复 ─↗
```

## 2. 房间类型分配

### 2.1 类型概率表

| 房间类型 | 基础概率 | 层数修正 | 特殊规则 |
|----------|----------|----------|----------|
| 战斗房 | 55% | 无 | 第 1 层至少 2 个 |
| 宝箱房 | 12% | 无 | 第 1 层至少 1 个（新手教学）|
| 回复房 | 10% | 层数越深概率 +5%（最深 0+5=15%）| Boss 层前至少 1 个 |
| 商店房 | 10% | 无 | 每层最多 1 个 |
| 事件房 | 5% | 无 | 每层最多 1 个（可选存在）|
| 强化房 | 固定 1 个 | 每层固定 | 层末，非战斗房 |

### 2.2 房间类型分配算法

```
分配步骤（在 DAG 生成后执行）:

1. 标记特殊房间
   - Boss 房: layer 2 最后一个节点（固定）
   - 强化房: layer 2 中非 Boss 的 1 个节点（固定）
   - 入口: layer 0（固定，不参与类型分配）

2. 剩余房间从类型池中按权重抽取
   - 权重: 战斗(55) / 宝箱(12) / 回复(10) / 商店(10) / 事件(5)
   - 使用 Fisher-Yates 洗牌打乱后按顺序分配
   - 确保: 战斗房占比 ≥ 50%

3. 验证通过条件
   - 至少 1 个回复房在 Boss 房前一层
   - 第 1 层至少 1 个宝箱房
   - 第 1 层至少 2 个战斗房（教程需要）

4. 如果验证失败 → 强制替换最靠近 Boss 的非战斗房为战斗房
```

### 2.3 层数难度表

| 层数 | 房间数 | 战斗房数 | 入口分支 | 特殊说明 |
|------|--------|---------|----------|----------|
| 1 | 6 | 3~4 | 2 | 宝箱房必出，用于新手教学 |
| 2 | 7 | 3~4 | 2~3 | 难度开始上升 |
| 3 | 8 | 4~5 | 2~3 | 中间层可能出精英房替代普通战斗房 |
| 4 | 9 | 5~6 | 3 | 首现双精英战斗组合 |
| 5 | 10 | 5~6 | 3 | 全层混合+强化版怪物 |

## 3. 房间物理布局（6×6 网格）

> 战斗系统的 6×6 网格战场布局，由地牢生成系统决定。

| 走廊房 | 15% | 6~8 | 0 | 0 | 0 | 0 | 窄通道，远程怪有利 |
| 环形房 | 10% | 3~4 | 4~6 | 0 | 1 | 0 | 中间高台+周围荆棘 |
| 治愈室 | 10% | 4~5 | 2~3 | 1~2 | 0 | 0 | 治疗符文+障碍保护 |
| 暗影室 | 10% | 2~3 | 0 | 0 | 0 | 4~6 | 大量暗区，隐身走位 |

### 3.3 模板选择策略

```
每个房间的地形模板选择规则:

1. 根据房间类型筛选可用模板池
   - 战斗房: 全部 10 种模板可用（默认开放厅概率最高）
   - 宝箱房: 开放厅(70%) / 走廊房(30%)（宝箱放在中央，障碍物旁）
   - 回复房: 治愈室固定（中央治疗符文）
   - 商店房: 开放厅(80%) / 复合房(20%)（NPC 在中央空地）
   - 事件房: 复合房(60%) / 暗影室(40%)（氛围营造）
   - 强化房: 开放厅(50%) / 高台房(50%)（仪式感）
   - Boss 房: 空旷房(70%) / 复合房(30%)（场地要大）

2. 模板选择后，可做小幅度随机偏移
   - 将单个障碍物向外移动 1 格（不超出 6×6 边界）
   - 偏移概率: 30%（为了增加房间间的差异性）
   - 偏移不影响房间类型的功能性（如回复房必有治疗符文）

3. 玩家入口位置: 固定在地图左侧中央 (0,2) 或 (0,3)
4. 怪物生成位置: 地图右侧（具体位置由怪物装配系统决定）
```

## 5. 种子集成

### 5.1 种子使用流程

```
种子码 (如 "AB3K7Z")
    ↓
种子 → 确定性随机数生成器 (如 mulberry32 / xorshift128+)
    ↓
地牢生成系统消费随机数的顺序:

消费 1: 层数 1 的 DAG 结构 (房间数/连接/分支)
消费 2: 层数 1 的房间类型分配
消费 3: 层数 1 每个房间的地形模板选择
消费 4: 层数 1 的偏移量
消费 5: 层数 2 的 DAG 结构
... 以此类推直到所有层生成完成

注意: 怪物装配系统和词缀系统也使用同一个种子，
      但它们是"按房间进入顺序"消费随机数，
      不在地牢生成阶段消费完。
```

### 5.2 种子一致性保证

```
// 保证"同一种子 → 完全相同的地牢布局"

function generateDungeon(seed, maxFloors) {
  const rng = createSeededRNG(seed);  // 确定性
  
  for (let floor = 1; floor <= maxFloors; floor++) {
    // 每次生成前，先"提交"seed+floor 作为新的子种子
    const floorSeed = hashSeed(seed + '_floor_' + floor);
    const floorRNG = createSeededRNG(floorSeed);
    
    // 用子种子生成这一层
    rooms[floor] = generateFloor(floorRNG, floor);
  }
  
  return rooms;
}

// 子种子策略: 玩家消耗固定的随机数顺序
// 怪物装配和词缀使用不同的子种子链
function hashSeed(input) {
  // 简单哈希，保证同一输入→同一输出
  let hash = 0;
  for (let i = 0; i < input.length; i++) {
    hash = ((hash << 5) - hash) + input.charCodeAt(i);
    hash |= 0;  // 转换为 32 位整数
  }
  return Math.abs(hash);
}
```

## 6. 地牢生成的"破解"标准

| 破坏模式 | 判定标准 | 严重等级 |
|----------|----------|----------|
| **可达性死点** | 某个房间从入口无法到达 | ❌ 灾难 |
| **Boss 房不可达** | 所有路径都不经过 Boss 房 | ❌ 灾难 |
| **无强化房** | 强化房被覆盖或未生成 | ❌ 灾难 |
| **地图过大** | 单层 > 12 个房间（超 10 分钟）| ⚠️ 严重 |
| **地图过小** | 单层 < 3 个房间（不够玩）| ⚠️ 严重 |
| **连续同类型房间** | 连续 3 个战斗房 | ✅ 可接受（可手动修正）|
| **模板重复** | 连续 3 个房间用相同地形模板 | ✅ 可接受（视觉疲劳）|
| **种子不唯一** | 两个不同种子产生相同布局 | ❌ 灾难（随机算法问题）|

## 7. 系统接口

### 7.1 输出

| 输出 | 类型 | 说明 |
|------|------|------|
| `floorLayout[floor]` | Room[] | 每层的房间数组（DAG 节点）|
| `room.connections` | int[] | 该房间连接到的下层房间索引 |
| `room.type` | enum | 房间类型 |
| `room.terrain` | int[6][6] | 地形模板矩阵 |
| `room.entrancePos` | (x, y) | 玩家进入该房间的位置 |
| `room.monsterSpawns` | (x, y)[] | 怪物生成位置（留空，由怪物系统填充）|

### 7.2 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| `seed` | 种子系统 | 6 位字符串 |
| `maxFloors` | 游戏配置 | 当前局的总层数 |
| `floorDifficulty` | DDA 系统 | 当前难度调整参数 |

### 7.3 事件输出

| 事件 | 触发时机 | 谁消费 |
|------|----------|--------|
| `onFloorGenerated(floor)` | 每层生成完毕 | 怪物装配、房间变异 |
| `onRoomEntered(room)` | 玩家进入房间 | 战斗系统、事件系统 |
| `onRoomCleared(room)` | 房间清空 | 跑马灯、统计 |
