## Phase 2：可选 — 追加显式 TapTap 合同标签

### 背景

当前的 `normalize_prompts_positive_style.py` 生成的质量描述已经满足 TapTap 标准：
- `QUALITY_CORE` → "Release quality target: crisp outlines, clean alpha-ready edges, high-resolution details..."
- `SAFE_COPY_CORE` → "Runtime copy rule: all readable copy is provided by Cocos labels..."
- 各分类都有关键合同语义的描述

唯一缺失的是显式的 `TAPTAP ANDROID QUALITY CONTRACT:` 标识，如果要求：
- 便于后续审计时一眼看出"该资源已按 TapTap 标准更新"
- 便于自动化脚本按标签做不同处理

### 追加策略（如需要）

在 `normalize_prompts_positive_style.py` 的每个 `CORE` 段落后追加对应合同标识，或者用一个单独的轻量脚本：

```python
# postfix_taptap_label.py
# 在 prompts.json 的每个 prompt 末尾追加 TapTap 标签
TAPTAP_LABEL = " TAPTAP ANDROID QUALITY CONTRACT: this prompt targets TapTap Android release quality."
```

### 决策

**建议：不追加。** 原因是：
1. 合同内容已内化到 prompt 各段中，并不存在待填补的缺口
2. 追加显式标签只是字符串冗余，AI 不会因此生成不同质量的结果
3. 资源合规验证应由 `validate_taptap_quality.py` 按内容标准判断，而不是按标签字符串存在判断

## Phase 3：更新生成脚本体积策略（必要）

### 3.1 审查硬编码体积上限

方案文档第 7 章指出生成脚本可能包含硬编码 size 上限（`failed_size → 删除文件`）。

搜索范围：
- `tools/p1_batch_generate.py`
- `tools/p1_gen_v2.py`
- `tools/p2_batch_generate.py`
- `tools/gen_game_assets.py`

### 3.2 更新策略

| 旧规则 | 新规则 |
|---|---|
| `size > hard_limit → failed_size → 删除` | 先查安全/尺寸/透明/帧数 → 再查体积 |
| | `size > manual_review_kb → manual_review_size` |
| | `size > warning_kb → size_warning` |
| | **不自动删除**体积大但视觉合格的资源 |

`batch_report.csv` 同时增加 `recommend_kb` / `warning_kb` / `manual_review_kb` 字段。

### 3.3 验证

小批量试生成 2-3 个资源，确认：
- 超过推荐体积 → warning
- 极大数据 → manual_review_size
- 不会因体积自动删除
