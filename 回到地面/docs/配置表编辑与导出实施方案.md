# 配置表编辑与导出实施方案

> 项目：回到地面  
> 当前配置：`assets/resources/config/*.json`  
> 目标：让配置变得适合日常编辑、批量修改、校验、导出和版本管理，避免长期手写 JSON。

---

## 1. 总体结论

当前项目已经有一套 JSON 配置：

```text
assets/resources/config/
├─ battle.json
├─ economy.json
├─ elements.json
├─ equipment.json
├─ items.json
├─ monsters.json
├─ player.json
├─ skills.json
├─ text.json
└─ zones.json
```

JSON 适合程序读取，但不适合长期人工配置。后续最推荐的工作流是：

```text
人编辑：Excel / WPS / 在线表格
        ↓
导出脚本：校验、转换、格式化
        ↓
运行时：JSON
        ↓
Cocos：ConfigManager 读取
```

也就是：

```text
表格是配置源
JSON 是运行产物
脚本是质量门禁
```

不要把 JSON 当成主要编辑界面。长期手写 JSON 会遇到这些问题：

- 少逗号、少引号导致整个配置不可读。
- 批量调整数值不方便。
- 中文容易出现编码问题。
- 引用 id 写错无法及时发现。
- 权重、等级、价格、掉落等很难横向比较。
- 策划、美术或非程序参与配置时门槛高。

## 3. 配置编辑工具选择

### 3.1 单人开发阶段

推荐：

```text
WPS / Excel 本地 xlsx
```

优点：

- 编辑体验好。
- 支持筛选、排序、公式、批量填充。
- 不需要联网。
- 文件可以直接跟项目一起版本管理。

缺点：

- 多人同时编辑容易冲突。
- Git diff 不友好。

### 3.2 多人协作阶段

推荐：

```text
腾讯文档 / 飞书表格 / 钉钉表格
→ 定期导出 xlsx/csv
→ 脚本生成 JSON
```

优点：

- 多人协作方便。
- 可以加权限。
- 策划、美术也能参与。

缺点：

- 要制定导出规范。
- 在线表格字段名不能随便改。

### 3.3 后台配置阶段

暂不建议现在做。只有当项目需要运营后台、活动配置、灰度发布、远程开关时，再考虑：

```text
管理后台
→ 数据库
→ 导出远程 JSON
→ 客户端缓存
```

当前阶段不需要把怪物、装备、技能这种静态配置放数据库。

## 4. 配置表拆分方案

### 4.1 怪物表

对应输出：

```text
assets/resources/config/monsters.json
```

推荐工作簿：

```text
回到地面_怪物表.xlsx
├─ monsters
├─ monster_scale
└─ notes
```

`monsters` sheet 字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | string | 是 | 怪物唯一 id |
| zone | string | 是 | 所属区域 |
| name | string | 是 | 显示名 |
| hp | int | 是 | 生命值 |
| atk | int | 是 | 攻击 |
| def | int | 是 | 防御 |
| speed | int | 是 | 移动速度 |
| ai | enum | 是 | AI 类型 |
| exp | int | 是 | 经验 |
| appearanceWeight | int | 是 | 出现权重 |
| assetKey | string | 否 | 资源路径或资源 key |
| note | string | 否 | 备注，不导出或仅导出到 debug |

示例：

| id | zone | name | hp | atk | def | speed | ai | exp | appearanceWeight | assetKey |
|---|---|---|---:|---:|---:|---:|---|---:|---:|---|
| slime | forest | 森林史莱姆 | 12 | 3 | 0 | 50 | charger | 3 | 30 | monsters/forest/slime |
| elfArcher | forest | 精灵射手 | 8 | 5 | 0 | 60 | ranged | 6 | 10 | monsters/forest/elf_archer |

导出 JSON 结构保持当前兼容：

```json
{
  "metadata": {
    "version": "1.0.0",
    "lastUpdated": "2026-06-27",
    "description": "怪物属性表"
  },
  "forest": {
    "slime": {
      "name": "森林史莱姆",
      "hp": 12,
      "atk": 3,
      "def": 0,
      "speed": 50,
      "ai": "charger",
      "exp": 3,
      "appearanceWeight": 30
    }
  }
}
```

校验规则：

```text
id 不能为空
同 zone 下 id 不能重复
hp > 0
atk >= 0
def >= 0
speed > 0
exp >= 0
appearanceWeight > 0
ai 必须在允许列表中
zone 必须存在于 zones 表
```

### 4.2 装备表

对应输出：

```text
assets/resources/config/equipment.json
```

推荐工作簿：

```text
回到地面_装备表.xlsx
├─ slot_base_stats
├─ rarity_weights
├─ prefixes
├─ suffixes
├─ sets
├─ weapon_types
└─ notes
```

#### `slot_base_stats`

| slot | atk | def | maxHP | attackInterval | range | moveSpeedMod | dodgeDistance | dodgeCdReduction | cdReduction |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| weapon | 4 | 0 | 0 | 1.0 | 1.5 | 0 | 0 | 0 | 0 |
| chest | 0 | 2 | 0 | 0 | 0 | -0.05 | 0 | 0 | 0 |

#### `prefixes` / `suffixes`

| id | name | slotType | effect.atk | effect.def | effect.element | effect.critChance | effect.lifeSteal |
|---|---|---|---:|---:|---|---:|---:|
| sharp | 锋利的 | weapon,gloves | 2 | 0 |  | 0 | 0 |
| blazing | 燃烧的 | weapon,gloves | 0 | 0 | fire | 0 | 0 |

说明：

- `slotType` 用英文逗号分隔。
- `effect.*` 字段导出时自动合并为 `effect` 对象。
- 空值不导出，避免 JSON 里出现大量无意义字段。

校验规则：

```text
slotType 必须是合法装备槽位或 all
effect 至少有一个字段
rarity 权重总和建议为 100
套装 id 不能重复
weapon type id 不能重复
数值字段不能填中文
```

## 5. 表格通用规范

### 5.1 字段命名

推荐：

```text
id
name
desc
type
zone
rarity
weight
icon
assetKey
note
```

规则：

- 字段名使用英文。
- 不使用空格。
- 不使用中文括号。
- 不随意改字段名。
- `note`、`comment`、`备注` 这类字段默认不导出。

### 5.2 id 命名

推荐：

```text
forest_slime
forest_wolf
skill_fireball
item_small_potion
equip_prefix_sharp
zone_forest
```

规则：

- 只用小写英文、数字、下划线。
- 不用中文。
- 不用空格。
- 不用特殊符号。
- 一旦上线，不随意改 id。

### 5.3 空值规则

```text
数字字段空值：按 schema 默认值处理，或报错
字符串字段空值：如果必填则报错
数组字段空值：导出为空数组或不导出
对象字段空值：不导出
备注字段：不导出
```

### 5.4 数组写法

表格中用英文逗号：

```text
weapon,gloves
fire,frost,lightning
forest,catacombs,volcano
```

导出为：

```json
["weapon", "gloves"]
```

### 5.5 对象写法

简单对象建议拆列：

```text
effect.atk
effect.def
effect.element
effect.critChance
```

导出为：

```json
{
  "effect": {
    "atk": 2,
    "element": "fire"
  }
}
```

不要让人工在单元格里写复杂 JSON，除非必要。

## 6. 导出脚本方案

### 6.1 脚本职责

`export_config.py` 负责：

```text
读取 xlsx
解析 sheet
转换数据类型
处理数组/对象
校验字段
生成 JSON
格式化 JSON
写入 assets/resources/config
输出导出报告
```

`validate_config.py` 负责：

```text
检查 JSON 是否可解析
检查 schema
检查跨表引用
检查资源路径
检查权重和数值范围
检查中文编码
```

### 6.2 导出命令

建议在 `package.json` 增加：

```json
{
  "scripts": {
    "config:export": "python tools/config_pipeline/export_config.py",
    "config:check": "python tools/config_pipeline/validate_config.py",
    "config:all": "npm run config:export && npm run config:check",
    "prebuild": "npm run config:check"
  }
}
```

### 6.3 导出报告

每次导出输出：

```text
[config] monsters.xlsx -> monsters.json
  rows: 36
  zones: 6
  warnings: 0

[config] equipment.xlsx -> equipment.json
  prefixes: 12
  suffixes: 12
  sets: 6
  warnings: 1

[config] done.
```

如果失败：

```text
ERROR monsters.xlsx / monsters row 12:
  field hp must be > 0, got -5

ERROR skills.xlsx / skills row 8:
  icon path not found: textures/icons/skills/fireball
```

错误信息必须指向：

```text
文件
sheet
行号
字段
错误原因
```

## 7. 校验规则

### 7.1 通用校验

```text
id 不能为空
id 不能重复
必填字段不能为空
数字字段必须是数字
百分比字段范围合理
权重字段 > 0
价格字段 >= 0
描述字段不能乱码
JSON 必须 UTF-8
```

### 7.2 跨表校验

```text
monster.zone 必须存在于 zones
skill.element 必须存在于 elements
equipment.element 必须存在于 elements
item.effectType 必须存在于允许列表
zone.bossMonsterId 必须存在于 monsters
shop.itemId 必须存在于 items/equipment/skills
text key 被代码引用时必须存在
```

### 7.3 资源路径校验

配置中如果填了资源路径：

```text
textures/icons/skills/fireball
textures/monsters/forest/slime_idle
textures/ui/main/start_button
```

脚本需要检查：

```text
assets/resources/textures/icons/skills/fireball.png 是否存在
对应 .meta 是否存在
PNG 是否为 RGB/RGBA
```

### 7.4 微信小游戏合规校验

配置中避免出现明显敏感词，尤其是面向审核可见文本：

```text
text.json
monsters.json 的 name
items.json 的 name/desc
skills.json 的 name/desc
```

建议导出时加一份简单敏感词扫描。发现后给 warning，不一定直接失败。

## 8. 编码与乱码处理

当前配置文件里已经能看到中文乱码迹象，例如中文显示为类似：

```text
妫灄
鎬墿
榄傜煶
```

这通常来自编码不一致：

```text
GBK/ANSI 文件被当成 UTF-8 读取
终端编码不一致
脚本写入时未指定 encoding
复制粘贴来源编码异常
```

强制规则：

```text
xlsx 源文件保留正常中文
导出 JSON 必须 UTF-8
Python 读写必须 encoding="utf-8"
JSON 导出使用 ensure_ascii=False
不要用 Windows 记事本手动改 JSON
```

Python 写 JSON 示例：

```python
import json

with open(output_path, "w", encoding="utf-8", newline="\n") as f:
    json.dump(data, f, ensure_ascii=False, indent=2)
    f.write("\n")
```

验收标准：

```text
VS Code 打开 JSON 中文正常
Cocos 运行中文正常
git diff 中文正常
导出脚本重复执行不会把中文变乱码
```

## 9. 与 ConfigManager 的接入方式

### 9.1 保持运行时 JSON 格式

第一阶段不要改运行时代码期望的 JSON 结构。导出脚本应该生成和当前 `ConfigManager` 兼容的结构。

例如怪物表导出仍保持：

```json
{
  "forest": {
    "slime": {}
  }
}
```

而不是贸然改成：

```json
{
  "monsters": [
    {}
  ]
}
```

这样可以降低改造风险。

### 9.2 后续再统一结构

稳定后可以考虑把所有配置统一成：

```json
{
  "metadata": {},
  "data": {}
}
```

但这需要同步修改 `ConfigManager`，不建议和配置表导出第一阶段混在一起做。

### 9.3 启动时输出配置版本

建议 `ConfigManager` 加载后输出：

```text
[Config] monsters v1.0.0 rows=36
[Config] equipment v1.0.0 prefixes=12 suffixes=12 sets=6
[Config] skills v1.0.0 rows=...
```

这样你能立刻知道当前游戏读到的是不是刚导出的配置。

## 10. 配置修改工作流

### 10.1 日常修改流程

```text
1. 打开 design/config_sheets/对应表格
2. 修改数值或新增行
3. 保存 xlsx
4. 运行 npm run config:all
5. 检查导出报告
6. 打开 Cocos / 浏览器预览验证
7. 提交 xlsx 和生成的 json
```

### 10.2 新增怪物流程

```text
1. 在 怪物表 monsters sheet 新增一行
2. 填 id、zone、name、hp、atk、ai、appearanceWeight
3. 配置 assetKey
4. 确认资源 PNG 已存在
5. 运行 config:all
6. 如果校验通过，进入游戏测试该区域刷怪
```

### 10.3 新增技能流程

```text
1. 在 技能表 skills sheet 新增技能
2. 填 id、name、desc、element、cooldown、damage
3. 配 icon
4. 如果是新 element，先确认 elements 表存在
5. 运行 config:all
6. 测试升级/装备/技能 UI 是否显示
```

### 10.4 调平衡流程

推荐先复制一份对比列：

```text
hp_old
hp
atk_old
atk
note
```

导出时只读 `hp`、`atk`，`*_old` 和 `note` 不导出。

这样能保留调参记录。

## 11. Git 与协作规则

### 11.1 哪些文件提交

建议提交：

```text
design/config_sheets/*.xlsx
assets/resources/config/*.json
tools/config_pipeline/*
```

不建议提交：

```text
~$*.xlsx
*.tmp
导出日志临时文件
```

### 11.2 修改冲突处理

如果多人协作，优先在线表格，避免 xlsx 冲突。

如果必须本地 xlsx：

```text
一次只让一个人改同一张表
改之前在群里说
改完立即导出 JSON 并提交
```

### 11.3 JSON 不手改

约定：

```text
assets/resources/config/*.json 是生成产物
除紧急修复外，不直接手改
```

如果手改 JSON，下一次导出会被覆盖。

## 13. 第一批表格优先级

建议先做这三张：

```text
1. 怪物表 monsters
2. 装备表 equipment
3. 技能表 skills
```

原因：

- 数值最多。
- 最容易频繁调整。
- 最不适合手写 JSON。
- 对玩法体验影响最大。

第二批：

```text
4. 区域表 zones
5. 道具表 items
6. 经济表 economy
```

第三批：

```text
7. 战斗参数 battle
8. 玩家成长 player
9. 元素反应 elements
10. 文本 text
```

## 14. 最终目标

配置系统完成后，你的日常工作应该变成：

```text
打开表格
改数值
运行导出
看校验结果
进游戏验证
```

而不是：

```text
打开 JSON
找括号
改逗号
担心乱码
运行后才知道错在哪里
```

最终标准：

- 人编辑表格，不手写 JSON。
- 程序只读 JSON，不读 xlsx。
- 导出脚本统一编码、格式和结构。
- 校验脚本阻止坏配置进入游戏。
- 构建前自动检查配置。
- 配置错误能定位到表格文件、sheet、行、字段。
