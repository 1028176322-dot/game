## 二、⚠️ 需整改项（2 项）

### 整改 1：召唤型怪物可能导致同屏超 10 只

**问题**：
- 战斗系统规格中"召唤型"AI 每 4 秒召 1 个小怪
- 一场战斗最长 90 秒 → 理论最多召唤 22 只小怪
- 加上原有的 2~3 只怪 + 玩家 → 同屏可能达 25+ 单位
- ❌ **违反微信小游戏性能红线：同屏 ≤ 10 个单位**

**修复方案**（选一）：
- 方案 A（建议）：召唤型怪物**最多召唤 3 只小怪**，达到上限后改为其他行为
- 方案 B：同屏怪物总数 > 8 时，不再触发召唤
- 方案 C：小怪 10 秒后自动死亡（限时存活）

**影响文件**：
- `docs/战斗系统规格.md` — 第 7.2 节 召唤型 AI
- `design/组合性随机性系统设计.md` — 第 3.3 节 召唤 Summon
- `design/玩法与平衡性设计.md` — 第 1.4 节 怪物 AI

### 整改 2：缺失平台特有规则的设计

**问题**：
以下微信小游戏特有的要求在设计文档中**完全没有提及**：

| 缺失项 | 说明 | 影响 |
|--------|------|------|
| 启动屏设计 | 微信要求必须有合规启动屏（含版本号/版权信息）| Phase 3 需预留设计时间 |
| 网络异常处理 | 广告加载失败时当前 fallback 机制未设计 | 觉悟战广告失败→直接结算（已做），但遗漏了每日宝箱/跑马灯广告失败的处理 |
| 数据埋点方案 | 微信小游戏需使用 wx.reportAnalytics，日志 1KB/条限制 | 完全未设计，需新增 `docs/数据埋点方案.md` |
| 本地存储限制 | 微信小游戏本地存储上限 10MB | 魂石/角色解锁数据需设计存储预算 |
| 更新机制 | 微信审核通过后灰度→全量，不支持热更新 | 大版本更新需设计版本号检查逻辑 |
| 用户拒接广告 | 微信允许用户关闭个性化广告推荐 | 需设计"无广告推荐模式"的 fallback 体验 |

**修复方案**：
1. **启动屏** → 在 Phase 0 设计阶段新增启动屏线框图，控制在 1 张图 < 100KB
2. **网络异常** → 所有广告位增加 fallback：加载失败 → 自动跳过，不阻塞玩家
3. **数据埋点** → 新增 `docs/数据埋点方案.md`，设计 10~15 个核心事件，每条 < 1KB
4. **本地存储** → 设计存储结构：`{soulStones, unlockedCharacters, unlockedRelics, settings, lastPlayTime}` 约 10KB 足够
5. **版本更新** → 在游戏启动时检测本地版本号 vs 远程版本号，不一致时提示重启
6. **广告关闭** → 如果用户关闭个性化广告，仍显示广告但内容通用
