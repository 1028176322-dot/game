## 1. 目标与范围

### 1.1 设计目标
把现有"物理连续穿越 6×6 房间"的地牢,重做为《酋长你别跑》式的**节点路线图肉鸽**:进层看见整张节点地图 → 选路 → 点节点进入独立 3D 俯视场景打完/交互 → 回地图选下一节点。战斗本身不动(手动 ARPG 手感 + 元素反应 + 闪白/震屏/弹道)。

### 1.2 设计支柱(不可妥协)
1. **战略选择感** — 看全图、做路线赌注。
2. **单层清晰** — 路线一眼可见,不迷路。
3. **战斗零改动** — 复用现有 ARPG 战斗层。
4. **移动端稳健** — 断线可恢复,性能不崩。

### 1.3 与 ai-tasks 基座的关系(2D→3D 已落地部分)
`docs/ai-tasks/`(Demo0–6 + §5.5–5.13) 已落地:`GameContext`/`LifecycleManager`/`Logger`/`ConfigDatabase`、`CameraBrain`(3D 相机)、`ICollisionService`/`KinematicMover`、`SaveManager`(`ISaveManager`)、`SaveService`(`core/save`)、`ReplayRecorder`(FNV-1a 种子确定性)、`SkillGraph`、`DungeonGenerator`(多房间连通图)、`GameBootstrap`(服务注册 + lifecycle)。本重做**复用**这些基座,不重复实现。

> ⚠️ 但 Demo0–6 本质是**技术验证链**,`GameBootstrap` 当前在注册完服务后立即 `destroyAll()`(见 §8.2),**不是正式运行时容器**。本版 §8.2 明确将其与正式 `startup()` 切开。

### 1.4 3D 主线 / 2D 兜底(更正 v0.3 表述)
- **正式目标路径 = 3D encounter**:节点进入后由 `SingleRoomEncounterBuilder` 造单房 → 3D 俯视场景战斗。
- **2D 仅开发兜底**:当 3D 底座(相机/碰撞/动画)未就绪时,用 2D fallback 场景调试节点流程与状态机;**不作为发布形态**。
- 此更正针对 v0.3"先 2D 复用、后 3D 升级"——该表述与"正在做 3D 改造"主线矛盾,现统一为"3D 正式、2D 兜底"。

## 2. 核心循环 (Core Loop)

### Moment-to-Moment (0–30 秒)
- **Action**: 在节点地图上点一个**可达**节点(金色首领/Boss 徽记、战斗/事件图标)。
- **Feedback**: 节点高亮脉冲 + 镜头推近 + 进入加载。
- **Reward**: 进入场景即"选择被兑现"的掌控感。

### Session Loop (5–30 分钟,单层)
- **Goal**: 从 `start` 走到 `boss`,沿途吃掉 Elite/Event/Shop/Rest 节点。
- **Tension**: 路径分支的"赌注"——选高风险 Elite 还是稳妥 Rest。
- **Resolution**: 击杀 Boss → `floor_cleared` → 进入下一层或结算。

### Long-Term Loop (小时–周)
- **Progression**: 多层累积 build / 解锁。
- **Retention Hook**: 每日种子挑战(同种子同图,复用 `ReplayRecorder` 确定性)。

## 3. 节点路线图数据结构

> **v0.4 关键修正**: v0.3 把 `reachable`/`currentNodeId`/`resolvedHistory` 与节点 `state` 混在一个 `NodeRouteMap` 里,形成**双状态源**。现分裂为三件:

### 3.1 静态定义(生成一次,不可变)
```ts
// ASCII-only comments in code; Chinese notes live in this Markdown doc.
export interface NodeRouteNodeDef {
    id: string;
    row: number;
    col: number;
    type: RouteNodeType;
    parents: string[];   // edges from upstream
    children: string[];  // edges to downstream
}

export interface NodeRouteMapDefinition {
    floorIndex: number;
    rows: number;
    nodes: Record<string, NodeRouteNodeDef>;
    edges: Array<{ from: string; to: string }>;
    startNodeId: string;
    bossNodeId: string;
    seed: number;
}
```

### 3.2 运行时状态(随流程推进,单点真相)
```ts
export interface NodeRouteRuntimeState {
    currentNodeId: string;
    reachableNodeIds: string[];
    completedNodeIds: string[];
    phase: RouteRunPhase;
}
```

### 3.3 UI 视图状态(渲染时派生,不存储)
```ts
export type RouteNodeViewState = 'current' | 'completed' | 'reachable' | 'locked';

export function getNodeViewState(
    nodeId: string,
    state: NodeRouteRuntimeState,
): RouteNodeViewState {
    if (state.currentNodeId === nodeId) return 'current';
    if (state.completedNodeIds.includes(nodeId)) return 'completed';
    if (state.reachableNodeIds.includes(nodeId)) return 'reachable';
    return 'locked';
}
```

> **规则**: 节点 `NodeRouteNodeDef` 上**不存**任何运行时状态;`reachable`/`completed`/`current` 全部由 `NodeRouteRuntimeState` 经 `getNodeViewState` 派生。状态机(§5)是唯一写入 `NodeRouteRuntimeState` 的入口。

## 4. 生成算法与验证规则

### 4.1 算法
- `rows = 10 + floorIndex`(每深一层多一行,`[PLACEHOLDER]` 调)。
- 每行 2–4 节点,`start` 独占第 0 行,`boss` 独占末行。
- 边生成: 每个节点连 1–2 个下一行节点,保证**至少 2 条**从 start 到 boss 的不重复路径。
- 节点类型分布(按权重 `[PLACEHOLDER]`): combat / elite / event / treasure / shop / rest / upgrade / boss。
- 全部随机来自 `routeSeed`(§10),禁 `Math.random`。

### 4.2 验证规则 (`NodeRouteValidator`)
- **连通性**: 从 `start` BFS 必达 `boss`,且可达集 ≥ 2 条路径。
- **无环**: 边的 `to.row > from.row` 恒成立(禁止回头边)。
- **保底**: 每行至少有一个节点被前一行覆盖(无孤立行)。
- **Boss 可信**: `bossNodeId` 在末行且入度 ≥ 1。

### 4.3 "Broken" 定义(先于 playtest)
- 全单节点路径(无选择) → broken。
- Boss 不可达 → broken。
- 任意节点无出边且非 Boss → broken。

## 5. RouteRunController 状态机

### 5.1 七态
```ts
export type RouteRunPhase =
    | 'map_select'        // 地图选路中
    | 'node_entering'     // 正在进入场景(加载)
    | 'encounter_running' // 场景内战斗中
    | 'reward_pending'    // 战斗完成,待结算
    | 'node_resolved'     // 节点已结算
    | 'floor_cleared'     // 本层 Boss 已杀
    | 'run_settled';      // 整局结算
```

### 5.2 推进规则(唯一入口)
- **所有**节点流程只通过 `RouteRunController` 推进,UI 点击只调 `requestEnter(nodeId)`。
- 防呆: `node_entering`/`encounter_running`/`reward_pending` 期间**忽略一切节点点击**(杜绝重复进入/重复结算)。
- Boss 节点仅在其上游路径节点全部 `completed` 且处于 `reachable` 时方可进入,**禁止跳层提前解锁**。
- `RouteRunController` 不依赖地图 UI、不依赖 3D 场景,只消费/产出 `NodeRouteRuntimeState` 与 `RouteNodeEncounterConfig`(§8.3)。

## 8. 节点执行层:SingleRoomEncounterBuilder + 3D 主线

### 8.1 ❌ 禁止误用 DungeonGenerator roomCount:1
`DungeonGenerator.generate(seed, zone, { roomCount: 1 })` **不可用**:其 `line 54` 将 `roomCount` clamp 到 `[3,12]`,传入 1 实际变 3,且它是"多房间连通图"生成器(撑 Demo5 整层),职责不匹配单节点。

### 8.2 ✅ 新增 SingleRoomEncounterBuilder(专门入口)
```ts
export interface SingleRoomEncounterOptions {
    roomId: string;
    type: RoomType;          // Constants.RoomType
    zone: string;
    seed: number;
    minSize?: number;        // default 7
    maxSize?: number;        // default 11
}

export class SingleRoomEncounterBuilder {
    build(opts: SingleRoomEncounterOptions): RoomLayout {
        const rng = new Rng(opts.seed);
        return {
            id: opts.roomId,
            type: opts.type,
            gridX: 0,
            gridY: 0,
            width: oddSize(rng, opts.minSize ?? 7, opts.maxSize ?? 11),
            height: oddSize(rng, opts.minSize ?? 7, opts.maxSize ?? 11),
            connections: [],
        };
    }
}
```
- 不破坏 `DungeonGenerator` 多房间职责,符合 ai-tasks 契约边界(Demo5 卡外不改它)。
- 文件位置:`assets/scripts/dungeon/SingleRoomEncounterBuilder.ts`(非 `route/`,见 §14 卡清单)。

### 8.3 RouteNodeEncounterConfig(解耦契约)
```ts
export interface RouteNodeEncounterConfig {
    runId: string;
    floorIndex: number;
    nodeId: string;
    nodeType: RouteNodeType;
    zoneId: string;
    difficulty: number;
    seed: number;
    encounterSceneId: string;
    rewardProfileId: string;
}
```
- 地图层(`RouteRunController`)只**产** config;3D 场景层只**消费** config。
- `NodeRouteGenerator` 不依赖 3D;`RouteRunController` 不依赖地图 UI。

### 8.4 正式运行时硬规则(GameContext / Lifecycle)
> **v0.4 新增硬规则** — 当前 `GameBootstrap.ts:346` 注册完服务即 `destroyAll()`,属 Demo probe 行为。**正式运行时禁止**。

```ts
// formal runtime entry (NO destroyAll after startup)
async startup(): Promise<void> {
    await this._loadConfigAndAssets();
    this._wireInfra();          // register services into GameContext
    this._lifecycle.enterAll(); // services now live for the whole session
}

// demo probe ONLY (separated from formal startup)
runLifecycleSmokeTestOnly(): void {
    this._lifecycle.enterAll();
    this._lifecycle.pauseAll();
    this._lifecycle.resumeAll();
    this._lifecycle.exitAll();
    this._lifecycle.destroyAll();
}
```
- `ISaveManager`/`IRoomRuntime`/`IAssetCache` 等注册后**状态必须存活**至整局结束。
- Demo probe 拆到独立测试/调试入口,不混入 `startup()`。

## 9. 奖励预览与结算(防双发)

### 9.1 v0.3 双发风险(已确认)
`RoomFlowController.onBattleVictory()`(`line 44`)直接调 `grantRoomClearRewards`;若 route 模式战斗胜利也走此方法,再由 `NodeRewardResolver` 发一次 → **双发**。

### 9.2 修正:事件化,奖励永远由上层决定
```ts
// RoomFlowController: only emits completion, never grants in route mode
onBattleVictory(mode: 'legacy' | 'route' = 'legacy'): void {
    if (mode === 'route') {
        eventBus.emit('route:encounter_complete');
        return;
    }
    this._rewardService.grantRoomClearRewards(roomType);
}
```
或更干净:`RoomFlowController` 在 route 模式**只抛战斗完成事件**,奖励统一由 `RouteRunController` 在 `reward_pending` → `NodeRewardResolver` 结算。

### 9.3 NodeRewardResolver(节点统一结算)
- 输入:`nodeType + floor + difficulty + rewardSeed` → 输出 `rewardPreview`(地图显示倾向) + `resolvedReward`(完成后实际发)。
- 地图 UI 显示"奖励倾向"(如 combat→经验/装备, shop→金币),完成后发实际奖励,**只发一次**(状态机保证)。

## 10. 种子与存档恢复

### 10.1 种子 key 链(复用 ReplayRecorder)
```
runSeed       = user / daily / random seed
floorSeed     = FNV1a(runSeed, floorIndex)        // via ReplayRecorder.computeConfigHash
routeSeed     = FNV1a(floorSeed, "route")         // NodeRouteMapDefinition.seed
nodeSeed      = FNV1a(floorSeed, nodeId)          // per-node variance
rewardSeed    = FNV1a(nodeSeed, "reward")         // NodeRewardResolver
encounterSeed = FNV1a(nodeSeed, "encounter")      // -> config.seed (same seed, same layout)
```
- 禁 `Math.random`;复用 `ReplayRecorder.computeConfigHash`(FNV-1a)+ `Rng`。

### 10.2 ❌ 两套存档现状(已确认)
- `assets/scripts/save/SaveManager.ts`(`ISaveManager`,§5.6)
- `assets/scripts/core/save/SaveService.ts` + `SaveTypes.ts`(正式业务大量使用)

### 10.3 ✅ RouteSavePort(统一边界,业务不直接选层)
```ts
export interface RouteRunSnapshot {
    schemaVersion: number;
    runId: string;
    zoneId: string;
    floorIndex: number;
    routeMap: NodeRouteMapDefinition;       // static
    runtime: NodeRouteRuntimeState;         // dynamic (current/reachable/completed/phase)
    activeEncounter?: RouteNodeEncounterConfig;
    seedState: {
        runSeed: number;
        floorSeed: number;
        routeSeed: number;
    };
}

export interface RouteSavePort {
    saveRoute(snapshot: RouteRunSnapshot): boolean;
    loadRoute(): RouteRunSnapshot | null;
    clearRoute(): void;
}
```
- 实现:`RouteSaveAdapter` 底层先适配 `SaveService.saveRun()`,再桥接 `ISaveManager`(扩展 `RunState`/`DungeonState` 字段)。
- **对上层只暴露 `RouteSavePort`**;route 业务禁止直接 `new SaveManager()` / `new SaveService()`。
- 存盘点:进层即存、节点结算即存、进入场景即存。
- 恢复:从 `phase` 重建 `RouteRunController`;若 `phase==='encounter_running'`,用 `activeEncounter` 同 `encounterSeed` 重进同场景(幂等)。
