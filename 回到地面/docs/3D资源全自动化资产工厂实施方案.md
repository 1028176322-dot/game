# 3D Resource Automation Factory Implementation Plan

> Purpose: build a fully automated 3D asset production pipeline for the game.
> Target users: another AI implementation session, art-pipeline maintainer, and project reviewer.
> Encoding rule: source code comments and generated tool logs should use ASCII only. Chinese player-facing text must stay in config tables.
> Current scope source: `E:/game/docs/ART_ASSET_GAP_CHECKLIST.md`.

## 0. Executive Decision

The project should not use raw AI-generated GLB files directly as runtime assets.

Use this final production structure:

| Pipeline | Role | When to use |
| --- | --- | --- |
| Pipeline C/D: controlled low-poly asset factory | Main production path | All new formal 3D characters, bosses, monsters, props, backdrops |
| Pipeline A: voxel remesh rescue path | Fallback only | Existing broken AI prototype models that already exist and must be salvaged |
| Direct AI GLB import | Forbidden | Never enter `assets/resources/models/**` directly |

Current production gap from `ART_ASSET_GAP_CHECKLIST.md`:

| Category | Current 3D state | Formal 3D gap |
| --- | --- | --- |
| characters | 1/5 done: archer body + weapon | 4 roles: warrior, assassin, mage, berserker |
| monsters | 0/36 | 36 monster models |
| bosses | 0/42 | 12 final bosses + 30 mini bosses |
| tiles | 0/24 | 24 3D tile modules |
| effects | 0/27 | 27 3D VFX prefabs |
| UI 3D backdrops | 0/2 | `BG_MainMenu.glb`, `BG_Splash.glb` |

Existing 2D resources are fallback/review assets. They do not close the formal 3D gap.

The automation goal is:

```text
config job -> generate references/prototype -> normalize -> repair/retopo
-> bake textures -> rig -> bind animation -> validate -> import to Cocos
-> runtime smoke test -> approval report
```

Every step must be script-driven. Failed assets must be isolated and must not overwrite approved runtime assets.

```json
      "full body visible",

      "symmetrical body proportions",
      "clear arms and legs separation",
      "clean outline",
      "no text in image"
    ]
  }
```

Positive-only rule:

1. Describe only desired safe content.
2. Do not list dangerous forbidden objects in the positive prompt.

4. Avoid terms that trigger horror, gore, text, symbols, badges with letters, posters, signs.

If a generator supports negative prompts, put banned words in a separate `negative` field consumed only by the API layer. Do not mix negative words into the positive prompt.

## 4.1 AI 3D Generator Input Contract

This section is mandatory for Tencent Hunyuan3D or any other AI 3D generator.

The generator must receive a structured input package, not only a short prompt. The package must describe visual style, object scope, pose, camera, separated parts, forbidden runtime assumptions, and expected output usage.

The AI output is a draft source. It is never a final runtime asset.

### 4.1.1 Input Package Files

For each generated asset, create:

```text
art_source/3d/jobs/<type>/<id>/
+-- job.json
+-- prompt_positive.txt
+-- generator_request.json
+-- references/
    +-- front.png
    +-- side.png
    +-- back.png
    +-- three_quarter.png
    +-- color_palette.png
```

Required inputs:

| File | Required | Purpose |
| --- | --- | --- |
| `job.json` | yes | Machine-readable asset contract |
| `prompt_positive.txt` | yes | Positive prompt sent to AI |
| `generator_request.json` | yes | Generator adapter parameters |
| `front.png` | character/monster/boss preferred | Main shape reference |
| `side.png` | character/monster/boss preferred | Depth and silhouette reference |
| `back.png` | character/boss preferred | Back items, tail, cape, quiver |
| `three_quarter.png` | yes | Style and final visual reference |
| `color_palette.png` | optional | Keep category color consistency |

If the generator supports only text-to-3D, still create reference files when possible. They are used by local review and later repair steps.

### 4.1.4 Prompt Fields By Category

#### Characters

Required prompt content:

1. Species and role.
2. Full body visible.
3. Neutral rig-friendly pose.
4. Clear separated limbs.
5. Friendly cartoon animal fantasy style.
6. Weapon/shield policy.
7. Back item policy, if any.
8. Shared skeleton usage.

Character-specific notes:

| Role | Required identity | Equipment policy |
| --- | --- | --- |
| warrior | bear or lion shield warrior | sword and shield separate socket props preferred |
| archer | deer archer | bow and quiver allowed; quiver is archer-only |
| assassin | cat rogue | twin daggers separate props preferred |
| mage | rabbit or sheep mage | staff separate prop preferred |
| berserker | bear or boar heavy fighter | axe separate prop preferred |

#### Monsters

Required prompt content:

1. Region identity.
2. Creature body type.
3. Gameplay size.
4. Simple readable attack silhouette.
5. Rig profile hint: humanoid, quadruped, blob, flying, static.
6. Safe cartoon style.

Do not request scary, horror, corpse, bone, blood, realistic injury, or dark grotesque features. Use safe creature language such as forest sprite, mushroom critter, lava turtle, snow puff, swamp plant creature, crystal wisp.

#### Bosses

Required prompt content:

1. Boss role and region.
2. Larger scale than common monsters.
3. Strong readable silhouette.
4. Main attack appendage or magic focus.
5. Socket needs for skill effects.
6. Friendly fantasy boss style, not horror style.

Boss output can exceed common monster budgets but must stay within `bosses_final` or `bosses_mini` profile.

#### Tiles

Required prompt content:

1. Modular tile module name.
2. Region.
3. Top-down or isometric gameplay readability.
4. Seamable edges if tile repeats.
5. Low height profile unless it is wall/highground.
6. Collision shape expectation.

Tiles must not be generated as decorative illustrations. They must be modular scene geometry.

#### Effects

Required prompt content:

1. Effect type.
2. Shape volume: ring, burst, trail, projectile, aura, impact.
3. Color language.
4. Transparent/isolatable runtime use.
5. Short animation or prefab particle use.

Effects may be prefab-based instead of GLB-only. The pipeline must validate both model and prefab references.

#### UI 3D Backdrops

Required prompt content:

1. Scene name: main menu or splash.
2. Fullscreen background composition.
3. Center-safe area for UI or logo.
4. Low-motion environment.
5. `forceUnlit` friendly material.
6. No readable text built into the model.

`BG_MainMenu` and `BG_Splash` are optional enhancement layers. 2D fallback must remain.

```python
class ModelGenerator:
```

Hunyuan3D adapter rules:

1. Read API key from environment variable only.
2. Do not write API key into repo files, logs, reports, or prompts.
3. Save exact request payload to `generator_request.redacted.json`.
4. Save exact response metadata to `generator_response.json`.
5. Save raw model to `art_source/3d/prototypes/**`.
6. Never save raw generator output into `assets/resources/models/**`.
7. If the API supports seed, record seed.
8. If the API supports image input, use the reference package.
9. If the API supports quality/mesh/texture params, map them from `modelProfile`.
10. If the API returns multiple candidates, save all candidates and run prototype gate on each.

```bash
set HUNYUAN3D_API_KEY=your_key_here
```

The exact API endpoint and payload fields must live in `tools/3d_pipeline/generators/hunyuan3d.py`, not in gameplay code or config tables.

### 4.1.6 Prototype Acceptance Gate

Immediately after AI generation, run prototype checks before any repair:

| Check | Pass rule |
| --- | --- |
| Importable | Blender can import the file |
| Main object visible | at least one valid mesh |
| Completeness | full body or full module visible |
| Object scope | no unrelated extra characters/items |
| Text mesh | no text-like mesh objects |
| Fragment count | below profile threshold or routed to Pipeline A |
| Scale | roughly within category scale |
| Orientation | can be normalized to project forward |
| Style | cartoon animal adventure, not realistic dark/horror |
| Runtime suitability | can be retopologized/rigged |

If the prototype fails content checks, regenerate. Do not try to repair a wrong concept.

### 4.1.7 Candidate Selection

If the generator produces multiple models:

1. Run technical prototype gate for all candidates.
2. Generate preview images for all candidates.
3. Score each candidate:
   - content match: 40
   - silhouette readability: 20
   - geometry cleanliness: 20
   - style consistency: 10
   - budget closeness: 10
4. Select the highest passing candidate.
5. Keep rejected candidates under `failed/` or `prototypes/rejected/`.

Do not ask AI to "fix" a failed candidate unless the failure is purely style or minor shape mismatch. For topology/fragment failures, use local repair or regenerate.

### 4.1.7A 2D Mother Image To Hunyuan3D Prototype Contract

Current practical flow:

```text
AI generates 2D mother image
-> user uploads mother image to Tencent Hunyuan3D image-to-3D
-> user downloads GLB
-> factory processes GLB
```

To make the downloaded GLB bindable, skinnable, animatable, and visually consistent with the mother image, this boundary must be formalized. The factory cannot guarantee success from an arbitrary 2D image.

#### 4.1.7A.1 Mother Image Requirements

The 2D mother image must be created for 3D reconstruction, not just for illustration.

Required image format:

| Field | Requirement |
| --- | --- |
| size | 1024x1024 or 1536x1536 square |
| background | plain light neutral or clean transparent-looking background |
| crop | full body, no cropping |
| view | front three-quarter or clean front view |
| pose | neutral A-pose or relaxed rig-friendly pose |
| lighting | soft even lighting, no hard shadows hiding limbs |
| silhouette | clear separation of head, torso, arms, legs, hands, feet |
| equipment | weapon/shield visible but separable; body remains complete |
| style | same cartoon animal adventure style |
| text | no text, no logo, no watermark |
| occlusion | minimal overlap; arms not glued to torso; weapon not crossing face |

Mother image prompt addition:

```text
Full-body character reference for 3D reconstruction, neutral rig-friendly A-pose, arms slightly away from torso, legs separated, hands and feet visible, clear front three-quarter view, simple clean background, soft even lighting, complete readable silhouette, body complete without equipment, equipment shown as separable prop shapes.
```

Reject mother image before Hunyuan3D if:

1. Character is cropped.
2. Body is in attack/action pose.
3. Arms touch or merge with torso.
4. Legs merge into one shape.
5. Feet are hidden.
6. Weapon covers face/body.
7. Shield covers most of torso.
8. Cape hides legs.
9. Background is complex and touches silhouette.
10. The image includes text, logo, UI frame, or decorative border.

#### 4.1.7A.4 Prototype Gate Before Rigging

Before `normalize -> repair -> rig`, run:

```text
tools/3d_pipeline/prototype_gate.py
```

Checks:

| Check | Pass rule |
| --- | --- |
| importable | Blender can import GLB |
| main mesh exists | at least one valid main mesh |
| default junk | no default cube/camera/light included after import cleanup |
| body completeness | head, torso, arms, legs, hands, feet visible for humanoid |
| bindability | neutral pose, limbs separated |
| equipment separability | weapon/shield can be separated or ignored as prop |
| fragment count | below threshold or route to Pipeline A rescue |
| material sanity | textures linked and readable |
| scale sanity | model can normalize to target height without extreme scale |
| visual match | silhouette and colors close to mother image |
| unsafe content | no text/unsafe motifs |

Report:


If `bindability=fail`, regenerate from a better mother image. Do not proceed to auto-rig.

#### 4.1.7A.5 Visual Consistency With Mother Image

The final runtime model must remain visually consistent with the mother image, but not necessarily identical topology.

Add a visual consistency check:

```text

-> compare with mother image thumbnail
-> generate side-by-side contact sheet
-> score silhouette/color/proportion
```

Suggested scoring:

| Score item | Weight |
| --- | --- |
| silhouette match | 35 |
| species/role identity | 25 |
| color palette | 15 |
| equipment identity | 10 |
| face/body proportions | 10 |
| style consistency | 5 |

Pass:

```text
total >= 75
no critical identity mismatch
```

Critical mismatch examples:

1. Bear becomes cat/dog.
2. Warrior loses shield identity.
3. Mage loses staff/robe identity.
4. Archer loses bow/quiver identity.
5. Main palette changes completely.
6. Body shape becomes unreadable at gameplay scale.

Output:

```text
art_source/3d/reports/visual_match/warrior/

+-- model_front.png
+-- model_3q.png
```

#### 4.1.7A.6 Guarantee Boundaries

The factory can guarantee final compliance only if all gates pass:

```text
mother_image_gate pass
prototype_gate pass
normalize_transform_gate pass
repair_gate pass
skin_binding_gate pass
animation_gate pass
visual_match_gate pass
runtime_validate pass
```

If the Hunyuan3D GLB is structurally bad, the factory should not pretend it can always repair it. It should fail early with a clear reason:

```text
REGENERATE_MOTHER_IMAGE
REGENERATE_HUNYUAN_MODEL
ROUTE_TO_PIPELINE_A_RESCUE
MANUAL_RETOPO_REQUIRED
```

This is the only realistic way to guarantee final resources meet spec without endless manual fixing.

### 4.1.8 Downstream Runtime Usage Contract

The generator request must consider how the asset will be used later in Cocos. A visually nice model is not acceptable if it breaks rigging, animation, equipment binding, collision, UI preview, or performance.

Every `job.json` must include a `downstreamUse` block:

```json
{
  "downstreamUse": {
    "runtimeMode": "battle_actor",
    "cameraUse": ["battle_topdown", "ui_preview"],
    "animationUse": ["idle", "walk", "attack", "skill", "hit", "dodge", "death"],
    "equipmentUse": {
      "weapon": "socket_prop",
      "shield": "socket_prop",
      "backItem": "socket_prop_optional"
    },
    "effectSockets": ["Weapon", "SkillOrigin", "Chest", "Foot"],
    "collision": {
      "type": "capsule",
      "source": "config",
      "doNotGenerateComplexColliderMesh": true
    },
    "lodUse": true,
```

### 12.1 Master Texture vs Runtime Texture

Do not treat a 4096 texture as a permanent runtime exception.

The correct rule is:

```text
4096 master texture = allowed source quality
4096 runtime texture = validation fail unless asset profile explicitly allows it
```

The pipeline must maintain two stages:

```text
art_source/3d/master/**       high quality source, 2048/4096 allowed
assets/resources/models/**    runtime asset, platform budget enforced
```

Validation stages:

| Stage | Purpose | 4096 character texture |
| --- | --- | --- |
| `master_validate` | source quality and editability | warning/pass |
| `runtime_validate` | Cocos/TapTap runtime budget | fail for characters |

Authorized exceptions are allowed only for master source files unless explicitly approved for runtime release.

Exception example:

```json
{
  "assetId": "models/characters/warrior/CHR_Warrior_A",
  "field": "sourceTextureSize",
  "allowed": 4096,
  "scope": "master_only",
  "reason": "high quality source retained for future rebake",
  "expires": "2026-08-31"
}
```

Forbidden exception:

```json
{
  "field": "runtimeTextureSize",
  "allowed": 4096
}
```

Runtime 4096 textures increase package size, memory use, load time, and mobile GPU pressure. They must not be hidden behind a long-term `authorized-exception`.

### 12.4 Quality-Safe Downscale Rules

Automatic downscale must protect appearance. Do not use a naive resize for all texture types.

#### BaseColor

Rules:

1. Convert to linear or use high-quality sRGB-aware resize if available.
2. Resize with Lanczos or area filter.
3. Apply mild unsharp mask after resize.
4. Do not over-sharpen outlines.
5. Preserve color palette and saturation.
6. Compare average color delta against source thumbnail.

Recommended parameters:

```json
{
  "baseColor": {
    "resizeFilter": "lanczos",
    "sharpen": {
      "enabled": true,
      "radius": 0.6,
      "amount": 0.35,
      "threshold": 4
    },
    "maxColorDelta": 8.0
  }
}
```

#### Normal Map

Rules:

1. Resize RGB vector data carefully.
2. Renormalize normal vectors after resize.
3. Do not apply color sharpening.
4. Keep tangent-space normal orientation unchanged.

Pseudo:

```python
def renormalize_normal_map(img):
    # Decode RGB [0,255] to normal [-1,1], normalize, encode back.
    pass
```

#### ORM / MRA Map

Rules:

1. Preserve channels independently.
2. Use area filter to avoid roughness/metallic noise.
3. Do not sharpen.
4. Validate channel min/max does not collapse.

#### Alpha Map

Rules:

1. Use premultiplied-alpha-aware resize for color+alpha images.
2. Preserve crisp cutout edges where needed.
3. Validate transparent border if asset depends on transparency.

### 12.6 How To Avoid Visual Loss Without Raising Texture Size

Use these before increasing runtime resolution:

1. **Better UV packing**: allocate larger UV islands to face, hands, weapon, torso emblem.
2. **Texture atlas discipline**: do not waste UV space on hidden backfaces or tiny decorations.
3. **Style simplification**: cartoon style should use clear color blocks, not noisy micro-details.
4. **Material separation when needed**: separate weapon/backdrop material only if it improves budget and clarity.
5. **Mip bias control**: in Cocos, tune sampler/mipmap settings instead of oversized textures.
6. **ASTC platform compression**: use ASTC 6x6 or 8x8 depending on asset importance.
7. **LOD texture sets**: hero/boss close views may use 1024, distant enemies use 512.
8. **Preview-specific override**: UI preview can use a higher preview texture only if it is not loaded in battle bundles.

Decision table:

| Problem after downscale | First fix | Second fix | Last resort |
| --- | --- | --- | --- |
| face becomes blurry | larger face UV island | simplify face texture | character runtime 1024 |
| armor details disappear | remove micro-detail | paint larger shapes | 1024 for hero only |
| normal map looks wrong | renormalize normal | lower normal intensity | rebake normal |
| colors shift | sRGB-aware resize | palette correction | manual texture touch-up |
| UI backdrop looks soft | 2048 backdrop runtime | split foreground/detail | profile exception |

### 14.5 AI Model Format Requirements For Auto-Rigging

If the character will be automatically fitted to `humanoid_animal_v1`, the AI-generated 3D model must follow stricter format rules.

These requirements must be included in `job.json`, `prompt_positive.txt`, and prototype validation.

#### Required Character Model Format

| Requirement | Rule |
| --- | --- |
| Pose | neutral A-pose or relaxed T-pose, arms away from torso |
| Full body | head, torso, arms, hands, legs, feet all visible |
| Limb separation | arms must not touch body; legs must not merge |
| Hands | visible hands, not hidden inside weapon/shield |
| Feet | visible feet at ground level |
| Body symmetry | roughly symmetrical for rig fitting |
| Clothing | attached to body, not floating unrelated pieces |
| Weapon | separate prop or easily separable visual shape |
| Shield | separate prop or easily separable visual shape |
| Back item | separate prop if large; archer quiver only for archer |
| Cape | separate back piece or simple attached cloth area |
| Tail | visible and not fused with legs |
| Ears | visible and not fused with weapons/hat |
| Mesh | one main character body mesh preferred |
| Floating objects | not allowed except planned separate props |
| Scale | full body centered, feet at bottom |
| Texture | clean stylized colors, no tiny unreadable noise |

Prompt addition for auto-rigging:

```text
The character stands in a neutral rig-friendly A-pose, full body visible, arms slightly away from torso, legs separated, hands visible, feet on the ground, clear head torso arms legs and feet. Equipment is readable as separate prop shapes for later socket attachment. The body remains complete without equipment.
```

#### Auto-Rig Rejection Cases

Reject and regenerate if:

1. Arms are glued to the torso.
2. Legs are merged into a single shape.
3. Feet are missing or hidden.
4. Weapon is fused through the hand/body.
5. Shield covers most of the body.
6. Character is sitting, crouching, jumping, or in action pose.
7. Body is cropped.
8. Model is only a bust/portrait.
9. Cape hides the full back/legs.
10. Too many floating decorative parts exist.

Do not try to fix these with weights. These are generation-format failures.

### 14.6 Automatic Skeleton Fitting Steps

Script:

```text
tools/3d_pipeline/blender/fit_standard_skeleton.py
```

Input:

```text
art_source/3d/working/<type>/<id>/02_repaired.blend
art_source/3d/rig_templates/humanoid_animal_v1/humanoid_animal_v1.blend
```

Output:

```text
art_source/3d/working/<type>/<id>/03_skeleton_fitted.blend
art_source/3d/working/<type>/<id>/skeleton_fit_report.json
```

Automatic fitting steps:

1. Import repaired character mesh.
2. Calculate bounding box.
3. Detect approximate feet bottom and body center.
4. Scale character to profile height.
5. Import `humanoid_animal_v1.blend` armature.
6. Scale armature to character height.
7. Align `Root` to feet center.
8. Estimate hip, chest, head positions by height ratios.
9. Estimate shoulder width from mesh bounds.
10. Place arm bones along visible arm span.
11. Place leg bones along visible leg span.
12. Place optional tail/ear bones if job config enables them.
13. Create socket empties from socket map.
14. Parent sockets to target bones.
15. Save fitted blend.

Default height ratios:

| Point | Ratio from feet |
| --- | --- |
| Hips | 0.47 |
| Chest | 0.68 |
| Neck | 0.82 |
| Head | 0.90 |
| Shoulder | 0.72 |
| Hand | 0.47 |
| Knee | 0.25 |

These ratios are starting values. The script may adjust using mesh landmarks if available.

Fit report:

```json
{
  "jobId": "warrior",
  "status": "pass",
  "height": 1.6,
  "scaleApplied": 0.82,
  "rootAtFeet": true,
  "missingBones": [],
  "missingSockets": [],
  "warnings": []
}
```

Failure conditions:

| Code | Meaning |
| --- | --- |
| `FIT_NO_MESH` | no valid character mesh |
| `FIT_BAD_POSE` | model pose unsuitable |
| `FIT_MISSING_LIMBS` | limb detection failed |
| `FIT_ARMS_MERGED` | arms too close to body |
| `FIT_LEGS_MERGED` | legs too close together |
| `FIT_SCALE_INVALID` | scale cannot be normalized |
| `FIT_SOCKET_MISSING` | required socket not created |

### 14.7 Automatic Skin Weight Steps

Script:

```text
tools/3d_pipeline/blender/auto_weight.py
```

Input:

```text
03_skeleton_fitted.blend
```

Output:

```text
03_rigged.blend
skin_weight_report.json
```

Steps:

1. Select character mesh.
2. Select armature.
3. Apply automatic weights.
4. Normalize all vertex groups.
5. Limit max influences per vertex according to profile.
6. Smooth problematic weight areas.
7. Assign rigid weights for props if they are included as child meshes.
8. Test deform with basic pose samples.
9. Save report.

Weight rules:

| Area | Expected behavior |
| --- | --- |
| Head | follows Head mostly rigidly |
| Torso | smooth Spine/Chest deformation |
| Arms | no torso pulling during attack |
| Hands | stable for weapon sockets |
| Legs | no skirt/body collapse during walk |
| Feet | stable ground contact |
| Tail | tail bone or body fallback |
| Ears | optional ear bones or head rigid weight |

Failure conditions:

| Code | Meaning |
| --- | --- |
| `WEIGHT_AUTO_FAIL` | Blender automatic weights failed |
| `WEIGHT_EMPTY_GROUP` | required bone has no weights |
| `WEIGHT_TOO_MANY_INFLUENCES` | vertex influence count above budget |
| `WEIGHT_BAD_DEFORM` | pose sample deforms badly |
| `WEIGHT_HAND_DRIFT` | hand moves incorrectly with weapon socket |

If automatic weight fails, route to manual review. Do not export a broken rig.

### 14.10 Skeleton Profile Matrix For Monsters And Bosses

`humanoid_animal_v1` is only for player characters and humanoid animal actors. It must not be forced onto all bosses and monsters.

The factory must resolve skeleton profile from `type + bodyType`, then load the matching rig template, animation set, socket rules, and validation rules.

Required job fields for animated 3D assets:

```json
{
  "id": "forest_boar",
  "type": "monster",
  "bodyType": "quadruped",
  "skeletonProfile": "quadruped_v1",
  "animationSet": "enemy_quadruped_v1",
  "modelProfile": "monsters"
}
```

Resolver:

```python
def resolve_skeleton_profile(job):
    asset_type = job.get("type")
    body_type = job.get("bodyType")

    if asset_type in ("tile", "backdrop", "prop") and body_type in ("static", None):
        return "static_v1"

    mapping = {
        "humanoid": "humanoid_animal_v1",
        "quadruped": "quadruped_v1",
        "blob": "blob_v1",
        "flying": "flying_v1",
        "plant": "plant_v1",
        "serpent": "serpent_v1",
        "golem": "golem_v1",
        "floating_caster": "floating_v1",
        "boss_custom": "boss_custom_v1"
    }

    if body_type not in mapping:
        raise ValueError("unknown bodyType: {0}".format(body_type))
    return mapping[body_type]
```

Recommended profile matrix:

| skeletonProfile | Applies to | Examples | P0 clips |
| --- | --- | --- | --- |
| `humanoid_animal_v1` | player roles, humanoid enemies | warrior, mage, guard | idle, walk, attack, skill, hit, death |
| `quadruped_v1` | four-legged beasts | boar, wolf, fox beast | idle, walk, attack, hit, death |
| `blob_v1` | slime/soft body | slime, poison jelly | idle, move, attack, hit, death |
| `flying_v1` | flying enemies | bat, ghost, floating crystal | idle, fly, attack, hit, death |
| `plant_v1` | rooted plants/turrets | vine, mushroom, poison flower | idle, attack, hit, death |
| `serpent_v1` | long body creatures | snake, worm, tentacle | idle, slither, attack, hit, death |
| `golem_v1` | bulky rock/elemental | golem, lava giant | idle, walk, attack, hit, death |
| `floating_v1` | floating caster body | abyss wraith, void caster | idle, float, attack, skill, hit, death |
| `boss_custom_v1` | complex bosses | multi-arm, multi-phase boss | idle, move, attack, skill, phase_change, hit, death |
| `static_v1` | no skeleton | tile, backdrop, static prop | none or optional idle |

Each profile must have a directory:

```text
art_source/3d/rig_templates/<profile>/
+-- <profile>.blend
+-- <profile>_skeleton.json
+-- <profile>_socket_map.json
+-- <profile>_animation_set.json
+-- <profile>_fit_rules.json
```

Generation prompt must match the profile:

| bodyType | Required AI pose wording |
| --- | --- |
| `humanoid` | neutral A-pose, arms away from torso, legs separated |
| `quadruped` | neutral four-legged standing pose, side-readable body, four legs separated |
| `blob` | centered soft body, simple base, no limbs required |
| `flying` | floating centered pose, wings or floating body visible, no ground contact required |
| `plant` | rooted base, flexible stem/head, fixed bottom |
| `serpent` | long curved body, head and tail visible, segments readable |
| `golem` | bulky upright stance, clear arms and legs, stable feet |
| `floating_caster` | floating torso/core, readable casting appendages, no walking legs required |
| `boss_custom` | profile-specific neutral boss pose, separated attack appendages |

Pre-rig gate:

1. `bodyType` must exist for animated monster/boss jobs.
2. `skeletonProfile` must resolve from `bodyType`.
3. Rig template must exist.
4. Animation set must exist.
5. AI model pose must match profile fit rules.
6. If profile mismatch is detected, fail before auto-rig.

Never use `humanoid_animal_v1` as fallback for an unknown monster or boss body. Unknown body type means the job config is incomplete.

### 14.11 Profile-Specific Skinning Strategy

Each profile needs its own auto-weight fallback. The nearest-bone-segment method is useful, but the body-region rules must differ by profile.

| profile | Fallback skinning rule |
| --- | --- |
| `humanoid_animal_v1` | head/torso/arms/legs/feet nearest bone segments |
| `quadruped_v1` | spine chain plus four leg chains, head/tail weighted by region |
| `blob_v1` | mostly root/body bones, optional squash/stretch bones |
| `flying_v1` | body core + wing bones; no leg expectation |
| `plant_v1` | root/base rigid, stem/head flexible chain |
| `serpent_v1` | body vertices weighted along spline/segment chain |
| `golem_v1` | rigid chunk weighting, less smooth blending |
| `floating_v1` | core body + appendage/tendril chains |
| `boss_custom_v1` | custom fit and weight script per boss family |

Validation must be profile-aware. Example:

```json
{
  "profile": "quadruped_v1",
  "requiredWeightedBones": ["Root", "Spine_01", "Spine_02", "Head", "Leg_FL_Upper", "Leg_FR_Upper", "Leg_BL_Upper", "Leg_BR_Upper"],
  "requiredSockets": ["Head", "Chest", "SkillOrigin"],
  "requiredClips": ["idle", "walk", "attack", "hit", "death"]
}
```

If a boss cannot fit any general profile, create a new `boss_<family>_v1` profile instead of weakening the validator.

## 26. Minimum Core Code

### 26.1 Job loader

```python
from pathlib import Path
import json

PROJECT_ROOT = Path(__file__).resolve().parents[2]
JOB_PATH = PROJECT_ROOT / "assets/resources/config/3d_jobs.json"

def load_jobs():
    data = json.loads(JOB_PATH.read_text(encoding="utf-8"))
    jobs = data.get("jobs", [])
    seen_ids = set()
    seen_assets = set()
    for job in jobs:
        job_id = job["id"]
        asset_id = job["assetId"]
        if job_id in seen_ids:
            raise ValueError(f"duplicate job id: {job_id}")
        if asset_id in seen_assets:
            raise ValueError(f"duplicate asset id: {asset_id}")
        seen_ids.add(job_id)
        seen_assets.add(asset_id)
    return jobs
```

### 26.2 Safe copy to runtime

```python
from pathlib import Path
import shutil
import time

def safe_copy_approved(src: Path, dst: Path, project_root: Path):
    src = src.resolve()
    dst = dst.resolve()
    runtime_root = (project_root / "assets").resolve()
    if runtime_root not in dst.parents:
        raise ValueError(f"destination outside assets: {dst}")
    if not src.exists():
        raise FileNotFoundError(src)
    if dst.exists():
        backup = dst.with_suffix(dst.suffix + f".bak_{int(time.time())}")
        shutil.copy2(dst, backup)
    dst.parent.mkdir(parents=True, exist_ok=True)
    shutil.copy2(src, dst)
```

### 26.3 State update

```python
import json
from datetime import datetime, timezone

def set_status(state_path, job_id, status, **extra):
    state = {}
    if state_path.exists():
        state = json.loads(state_path.read_text(encoding="utf-8"))
    item = state.setdefault(job_id, {})
    item["status"] = status
    item["updatedAt"] = datetime.now(timezone.utc).isoformat()
    item.update(extra)
    state_path.parent.mkdir(parents=True, exist_ok=True)
    state_path.write_text(json.dumps(state, ensure_ascii=False, indent=2), encoding="utf-8")
```

### 26.4 Validation wrapper

```python
import subprocess
import sys

def run_validator(asset_path, budget_path, report_path):
    cmd = [
        sys.executable,
        "tools/asset_validate.py",
        str(asset_path),
        "--budget",
        str(budget_path),
        "--report",
        str(report_path),
    ]
    result = subprocess.run(cmd, text=True, capture_output=True)
    if result.returncode != 0:
        raise RuntimeError(result.stderr or result.stdout)
```

## 27. Risks And Fixes

| Risk | Root cause | Fix |
| --- | --- | --- |
| AI model generates fragmented mesh | image-to-3D creates many loose parts | use template low-poly mainline; Pipeline A only as rescue |
| Model looks different from 2D art | prompt drift | reference sheet + style locked in job config |
| Triangle count too high | remesh without retopo | budget gate + template topology |
| Animation breaks | inconsistent skeleton | shared skeleton profiles |
| Weapon floats | no stable socket | socket validation and separate weapon props |
| Runtime import path wrong | two project roots | path resolver must print project root and reject outside-root writes |
| Old approved asset overwritten | batch overwrite | default no overwrite; backup before copy |
| Cocos loads source asset not prefab | wrong runtime abstraction | runtime always loads prefab/registry id |
| Manual fixes disappear | no state tracking | state file and reports are authoritative |
