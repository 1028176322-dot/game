## 项目约定

- **目标平台**: 微信小游戏 — 所有设计/美术/代码均以此为目标
  - 主包 < 4MB（目标 3.5MB），美术风格为**卡通动物风**（高清卡通动物游戏美术，非像素块风，非暗黑风）
  - 游戏名称/描述/怪物名称已全部适配微信内容审核规范
  - 内容红线：不得出现"血/杀/死/尸/亡/骷髅/恶魔/地狱"等敏感词
  - 美术红线：不得有真实血液/碎骨/尸体/恐怖元素，死亡=化光消散，受击=闪白
  - 60FPS，同屏 ≤ 10 单位，Draw Call < 100，内存 < 200MB
  - IAA 纯广告变现，使用微信官方广告组件
  - 碎片化场景 5-15 分钟，单场战斗 30-90 秒
  - 本地存储 < 500 字节（上限 10MB）
  - 使用 wx.reportAnalytics 上报数据，单条 < 1KB
  - 所有广告位必须有网络异常 fallback
  - 必须含合规启动屏（<100KB），点击 2 秒跳过
  - 需预留软著申请 + 自审报告时间
  - 提审前必须过一遍第9章《微信小游戏提审规则与开发注意事项》
- **文件编码**: 所有文件强制 UTF-8（已确认全部文件均 UTF-8）
- **引擎**: Cocos Creator 3.x + TypeScript
- **远程仓库**: git@github.com:1028176322-dot/game.git（main/develop/feature 分支策略）
- **配置表规范**: 
  - 所有数值从 `GameConfig.ts` + `assets/resources/config/*.json` 读取，禁止代码硬编码
  - 每条配置需标注用途/合法范围/默认值（`$comment` 字段）
  - JSON 表行键用 `小写单数`，字段用 `小驼峰`
  - 环境隔离通过 `env/dev.json` 覆盖层，不维护多套完整配置
  - 新增配置需 commit 写明：旧值→新值 + 原因
- **版权规避**: 
  - 系统和机制名称必须使用通用术语，不得使用已注册商标/版权保护的专有名词
  - 怪物名称必须为通用幻想类型（骷髅/魔像/史莱姆等），禁用特定 IP 专属怪物名
  - 避免直接复制其他游戏的技能/能力实现方式——同效果不同实现路径
  - 美术风格最终确定前需要进行版权筛查（美术风格本身不版权，具体形象版权）
  - 元素体系名称已选用通用元素名（Fire/Frost/Lightning/Poison/Shadow/Holy），无侵权风险
  - 参考来源需在开发说明中标注"灵感来源"，而非"设计参考"
  - 与提灯与地下城的差异：回合制≠实时、有宠物≠无宠物、亮度系统≠无亮度系统，已书面记录
- **已执行的版权改名**: 暗影步→穿影 / 战吼→怒吼 / 时间扭曲→时序加速 / Blink→瞬移
- **已执行的内容审核合规改名(微信小游戏)**: 吸血光环→生命吸取 / 浴火→火行者 / 血月→绯红之月 / 亡灵君主→白骨君主 / 深渊魔王→深渊霸主 / 剧毒沼泽→翠毒沼泽 / 之亡者→之幽影 / 吸血(词缀)→汲取 / 腐朽→霜蚀 / 击杀→击败 / 死亡→战败 / 骷髅兵→骨兵 / 亡魂射手→暗影射手 / 食尸鬼→掘地行者 / 死亡骑士→暗影骑士 / 恶魔步兵→炎魔步兵 / 冰骷髅→冰晶兵 / 腐烂树精→枯木精 / 暗影恶魔→暗影魔 / 自爆魔像→爆裂魔像

## 2026-06-29 根因治理记忆（强制遵守）

- **后续方案标准**：用户已明确要求“给能从根源解决问题的方案，允许大改代码”。后续遇到架构、资源、配置、构建、审核、数据、性能问题时，必须优先分析根因并给系统性改造方案，不得把“调阈值、重试 AI、局部补丁”作为第一方案。
- **架构治理执行文件**：新的执行交接方案为 `docs/项目架构根因治理执行方案.md`。后续另一个 AI 或开发者进行架构改造时，应优先阅读并按该文件分阶段执行。
- **架构改造优先级**：
  1. 配置系统根治：新增 `ConfigService`，启动等待 `assets/resources/config/*.json` 加载完成，JSON 成为运行时唯一权威数据源。
  2. 随机数根治：新增 `Rng/RunRng`，除初始 seed 生成外，业务代码禁止直接使用 `Math.random()`。
  3. 事件系统根治：新增类型化事件总线，拆 `runEvents/uiEvents`，禁止新增裸字符串事件；战斗暂停不得调用全局 `eventBus.pause()`。
  4. 拆分 `DungeonSceneController`：场景控制器只负责节点装配；房间推进、奖励、广告、埋点、角色初始能力、变异应用必须迁移到服务层。
  5. 资源加载根治：新增 `AssetBundleService` 和 `assets.json` 资源映射；微信小游戏必须按首包/分包/区域 bundle 管理资源，不能把全部纹理放进首包。
  6. 存档和平台服务统一：新增 `StorageService/AdService/AnalyticsService`，禁止各模块直接散落访问 `wx/localStorage`。
- **资源生成根因原则**：
  - 透明主体类资源（characters / monsters / bosses / icons / ui）不再以 chroma magenta 作为主方案，必须使用“浅色中性分割背景 + 主体分离 + 结构验收”。
  - effects 仍可使用黑底转 alpha，因为其本质是发光特效；tiles 使用程序化生成；backgrounds 使用不透明母版 + 运行时压缩候选。
  - AI 只负责画主体，透明、裁切、拼接、压缩、验收由脚本接管。
  - 不合格资源必须进入 `textures_review/rejected` 或 review CSV，不得进入 `runtime_replace` 或正式 `textures`。
- **资源替换门禁**：
  - 禁止直接把 `runtime_replace` 全量复制进 `assets/resources/textures`。
  - 必须经过 `master -> runtime_candidates -> validated_runtime -> staging_textures -> Cocos import check -> replace`。
  - 只要存在 `missing_png`、`dimension_mismatch`、`alpha_opaque`、`visual_safety_rework_required`，就不得执行正式替换。
- **当前资源审计记忆**：
  - 最近审计显示资源管线仍未收敛：`runtime_replace_final_summary.json` 曾出现 `missing: 105`、`alpha_opaque: 182`、`dimension_mismatch: 89`、`issue_count: 643`。
  - 因此在修复完成前，不能宣称全部美术资源已可安全替换。
- **微信小游戏包体记忆**：

  - 必须做 Asset Bundle / 分包 / 按区域预加载释放，首包只保留 splash、main、core_ui、基础配置和极少量必需资源。

  - 当前已把 350 条透明主体资源提示词从 chroma magenta 改为浅色中性分割背景，并加入 `POSTPROCESS CONTRACT` 或 `BACKGROUND CONTRACT`。

## 2026-06-30 运行时自动装配与渲染接入记忆（强制遵守）

- **正式装配原则**：后续不得再依赖“在 Cocos 编辑器里手动拖 SpriteFrame/组件引用”作为主方案。正式上线标准必须走运行时自动装配：入口脚本负责创建必要节点，资源由 `assets/resources/config/assets.json` 统一映射，业务代码通过服务层加载。
- **已建立的运行时资源链**：
  1. `assets/scripts/assets/AssetBundleService.ts`：唯一资源映射入口，负责加载 `config/assets.json`，支持 `resources` bundle 与后续分包 bundle。
  2. `assets/scripts/assets/ArtResourceResolver.ts`：按角色、怪物、地块、背景等规则生成资源 ID，不允许业务代码到处拼散乱路径。
  3. `assets/scripts/assets/RenderAssetService.ts`：统一把资源 ID 应用到 `Sprite`，加载成功后替换调试占位，失败时只警告不阻断主流程。
  4. `assets/scripts/scene/DungeonSceneInstaller.ts`：负责创建地牢运行时节点结构，`DungeonSceneController` 只负责接线，不再堆节点创建细节。
- **当前已自动接入的主渲染链**：
  - 地牢战斗背景：`textures/backgrounds/bg_combat_{zone}`。
  - 玩家角色 idle 图：`textures/characters/{character}/{character}_idle`。
  - 普通怪物 idle 图：`textures/monsters/{zone}/monster_{zone}_{monsterId}_idle`，`monsterId` 来自 `assets/resources/config/monsters.json`。
  - 6 个区域 tiles：`textures/tiles/{zone}/tile_{zone}_{floor|wall|thorn|highground}`。
  - 已用脚本检查：角色 idle、普通怪 idle、6 区域 tile、森林背景均能命中 `assets.json`。
- **仍需继续正式化的渲染范围**：
  - UI、effects、icons、bosses 虽已在资源表中，但还需要各自 UI/技能/战斗表现系统调用 `RenderAssetService` 才算真正接入渲染系统。
  - 不得宣称 418 个资源都已“完整接到画面上”，当前完成的是主战斗渲染链和自动装配底座。
- **启动链记忆**：
  - `GameBootstrap.ensure(root)` 不能把组件直接挂到 Scene 本体上，Cocos 不允许 `Scene.addComponent`；必须创建 `GameBootstrap` 子节点后 `node.addComponent(GameBootstrap)`。
  - `EventBus.emit` 必须使用 `entry.callback.apply(entry.target, args)`，否则事件回调里的 `this` 会丢失，曾导致 `this._loadDungeonScene is not a function`。
  - Cocos 浏览器预览可能继续跑旧的 `temp/programming/packer-driver` chunk；源码修复后若浏览器仍报旧错误，需等待编辑器重新编译、停止预览重开，必要时重启编辑器。
- **代码质量记忆**：
  - 发现乱码注释可能把真实代码吞进 `//` 后面，不能只看表面运行结果。已清理 `GridManager`、`BattleManager`、`DungeonManager`、`GameBootstrap` 中的关键乱码/断行问题。
  - 后续任何触碰核心脚本后必须扫乱码风险，尤其是 `? if`、`? const`、中文乱码后同一行出现代码的情况。
- **验证记忆**：
  - 每次资源/装配调整后必须运行 `npm run validate:all`。
  - 需要额外验证资源 ID 命中：从 `assets.json` 检查角色、怪物、tiles、背景是否存在。
  - 当前 `npx.cmd -p typescript tsc -p temp/tsconfig.cocos.json --noEmit` 不能完整运行，原因是缺少 Cocos 生成的 `temp/declarations/cc*` 声明文件；需要在 Cocos Creator 重新生成声明后再跑 TS 检查。

## 工程规范（自开发规范提取 & 适配）

> 以下规则提取自 `../参考资料/游戏代码开发规范与流程.md`(参考文档)，已适配本项目的 Cocos Creator + TypeScript + 微信小游戏 + 单人开发场景。

### 1. 分层职责

| 层 | 职责 | 禁止 |
|----|------|------|
| **业务逻辑** | 玩法规则、状态流转、数值结算、条件判断 | 不直接依赖 UI 节点名/资源路径/Prefab 层级 |
| **UI 层** | 展示、输入、动画、转发操作意图 | UI 点击不直接修改战斗/背包/魂石等核心数据 |
| **数据配置** | 配置表有默认值/类型/范围/枚举说明；加载后完整校验 | 配置缺失不可崩，新增字段必须兼容旧存档 |
| **资源层** | 通过统一资源管理接口引用，禁止硬编码路径 | Prefab/图片/音效/字体/特效必须有缺失兜底 |

### 2. 编码硬性规则

- **状态字段集中定义**：每个状态字段必须有可选值列表、初始状态、允许转移、终止状态、回退规则
- **单一写入口**：存档/背包/魂石/天赋各自只有一个写入口，禁止多个模块直接写同一字段
- **副作用方法显式命名**：`useItem`、`grantReward`、`saveProgress`，不写模糊的 `doSomething`
- **异步回调处理**：UI 关闭/场景切换后，异步回调必须可取消或跳过已销毁对象
- **配置加载校验**：主键重复/外键缺失/数值越界/空字符串/非法枚举 → 加载时拦截
- **资源引用**：通过统一接口加载，不得散落硬编码路径；资源名变更同步检查引用

### 3. 状态变更规范

- 所有状态切换走统一方法（`setState`/`applyResult`/`commitReward`），禁止散落赋值
- 禁止状态降级：已确认的状态（已装备/已购买）不允许被刷新或重置逻辑覆盖
- 如必须回退，显式标注原因

### 4. 命名规范

- 类名 = 对象，方法名 = 动作，变量名 = 含义
- 布尔值：`is/has/can/should` 开头
- 异步接口在命名上体现（如 `loadAsync`、`fetchData`）
- 不使用 `data/info/temp/new/old/obj` 作为有意义的变量名

### 5. 开发流程（单人适配版）

**开发前**：
1. 明确需求来源和验收标准
2. 列出影响模块：业务/UI/资源/配置
3. 查现有实现，优先复用

**开发中**：
1. 小步提交，小步验证（每完成一个功能点就 commit）
2. 先改核心逻辑 → 再改 UI 展示
3. 每次新增状态/字段/配置，同步补校验
4. 每次修 bug，补至少一个防复发机制（断言/校验/检查）
5. 不混入无关重构

**自测（最低清单）**：
- 主流程可走通
- 失败流程可恢复（魂石不足、HP=0、资源加载失败）
- 重复点击不重复发奖/扣费/请求
- 快速打开关闭 UI 不报错
- 微信小游戏环境：网络慢/数据为空/资源缺失时不崩
- 旧存档数据可正常进入

### 6. 提交前检查（单人版）

1. `git status` → 确认只包含本次相关文件
2. `git diff` → 复核核心逻辑
3. 运行最小回归
4. 确认未误提临时文件（截图/.bak/日志）
5. 写清 commit message：改了什么 + 为什么改 + 怎么验证

### 7. 开发完成定义

以下全部满足才算完成：
- 功能符合规格
- 主流程通过
- 失败流程可恢复
- 状态机无非法转移
- 配置和资源校验通过
- 最小回归通过
- 无无关文件进入提交
- 对这次 bug 有防复发机制（如有）

### 8. 关键风险自查（提交前过一遍）

- [ ] 是否有空引用风险？（所有 getComponent/find 判空）
- [ ] 是否有异步回调访问已销毁对象？
- [ ] 是否有配置缺失兜底？
- [ ] 是否有资源缺失兜底？
- [ ] 是否会重复发奖/扣费/扣魂石？
- [ ] 是否影响旧存档数据？
- [ ] 是否跑过最小回归？
- [ ] 是否有防复发检查？
