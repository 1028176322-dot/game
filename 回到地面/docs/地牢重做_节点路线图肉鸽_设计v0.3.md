## 1. 目标与范围

### 1.1 设计目标
把现有"物理连续穿越 6×6 房间"重做为《酋长你别跑》式的**节点路线图肉鸽**:进层看见整张节点地图 → 战略选路 → 点节点进入独立 3D(或开发期 2D fallback)场景执行战斗/交互 → 结算返回地图 → 选下一节点 → 抵达层顶 Boss。

### 1.2 范围边界(明确不做什么)
| 类别 | 范围 | 说明 |
|------|------|------|
| ✅ 做 | 节点路线图生成 / 地图 UI / 节点选路状态机 / 节点执行层 / 奖励预览结算 / 种子复现 / 存档恢复 | 本重做核心 |
| ❌ 不做 | 群体随从 / 部族养成 | 参考游戏无,用户确认排除 |
| ❌ 不做 | 后方威胁追逐 / 撤离张力 | 参考游戏无,用户确认排除 |
| ➰ 复用 | 单人 ARPG 战斗(虚拟摇杆+攻击+受击闪白+震屏+弹道+元素反应) | 搬入节点场景,不改规则 |
| ➰ 复用 | 现有 `DAGGenerator`(冻结保留,作 legacy 房间链兜底) | 不改造,新增 `NodeRouteGenerator` 作规范来源 |

### 1.3 3D 与节点路线图的依赖关系(本版最重要原则)
**节点路线图与 3D 不是强绑定。** 节点地图只负责"生成选路数据 + 产出 `RouteNodeEncounterConfig`";3D 场景只负责"消费 config 执行战斗"。两者经中间契约解耦(见 §8.3),使:
- `NodeRouteGenerator` 不依赖任何 3D 场景;
- 3D 战斗逻辑不依赖地图 UI;
  - 若 3D 底座未完成,可用 2D fallback 场景调试节点流程,**但正式目标路径以 3D encounter 为准**。

### 1.4 与既有 2D→3D 基座(`docs/ai-tasks/`,另一 Agent 已实现)的关系
> 经核对 `docs/ai-tasks/` 实际代码:Demo0–6 + §5.5–§5.13 已落地 3D 底座与基础设施。**本重做必须复用而非重建**,否则违反 `_agent_contract.md`(禁止重复实现)并产生双系统。

- **`DungeonGenerator`(Demo5,已存在 `assets/scripts/dungeon/DungeonGenerator.ts`)**:输出 `DungeonLayout{seed,zone,rooms[],startRoomId,bossRoomId}`,`RoomType='start'|'battle'|'elite'|'reward'|'boss'`,按 seed+zone 生成连通房间布局。**本重做将其定位为「单个节点的 3D 房间建造器」**:进入某 combat/elite/boss 节点时,`RouteRunController` 以 `encounterSeed+zone` 调 `DungeonGenerator.generate(..., {roomCount:1})` 取得单个 `RoomLayout`,再经 `RoomBuilder→TileMap→NavigationGrid→RoomRuntime` 渲染。其自带 `connections` 连通图在路线模式下**不充当地图**(地图连通性归 `NodeRouteMap`)。
  - ⚠️ **协调项**:`DungeonGenerator.generate` 当前总是产出多房间连通图;路线模式需其支持 `roomCount:1`(或 `singleRoom`)产出单房。此改动须作为 **Demo7(路线系统)** 卡的一部分,或由本会话单独提交,**不得**由 ai-tasks Agent 在现有 Demo5 卡外擅自改动(契约 #6)。
- **`CameraBrain`(Demo2)/`ICollisionService`+`KinematicMover`(Demo3)/`SkillGraph`(Demo4)**:直接作为 P0 3D 底座复用,路线执行层消费之,不重建。
- **`SaveManager`(§5.6,已存在)`ISaveManager`**:`saveRun/savePlayer/saveDungeon/saveEnemy/saveSeed/loadRun` + `RunState{floor,seed,items}`/`DungeonState{zone,clearedRooms,currentRoom}`。**本重做的 `RouteRunSave` 不是新存档层,而是序列化进 `ISaveManager.saveRun`/`saveDungeon` 的字段扩展**(见 §10.2)。
- **`ReplayRecorder`(§5.7)+`RunRng`**:提供 `computeConfigHash`(FNV-1a)与种子确定性。**本重做种子 key 链(§10.1)复用其 FNV-1a 哈希与 `Rng`,不另造哈希。**
- **`GameContext`/`LifecycleManager`/`Logger`(Demo0)**:所有新 `route/` 模块经 `ctx.get` 取依赖、实现 `ILifecycle`,与基座一致。

## 2. 核心循环 (Core Loop)

### Moment-to-Moment (0–30 秒)
- **Action**: 在节点地图上点击一个**当前可达**节点(战斗/事件/宝藏)。
- **Feedback**: 地图高亮选中 → 过场(淡入/传送)→ 进入执行场景(3D 或 2D fallback)。
- **Reward**: 立刻知道"我要打什么/拿什么",决策感满足 + 预览奖励倾向。

### Session Loop (5–30 分钟)
- **Goal**: 从层底节点一路选路抵达层顶 Boss 并击败。
- **Tension**: 路径资源有限(血/金币/强化次数),"精英高风险高回报" vs "绕路保平安"。
- **Resolution**: 击败 Boss → 进入下一层(地图重新生成,更深更难)→ 或失败结算。

### Long-Term Loop (小时–周)
- **Progression**: 多层推进解锁 meta(角色/装备/元素 Build);种子挑战与每日/周常。
- **Retention Hook**: 每层地图不同带来的重开新鲜感 + Build 成长 + 深层 Boss 目标。

**Fun Hypothesis(必须最先验证的单一爽点)**: "看全图 → 为下一节点做有意义的取舍 → 进场景打赢" 这一连串的**决策→执行**节奏感。若玩家进层 3 秒内不能理解"我在哪、能去哪、Boss 在哪",本系统即失败。

## 3. 节点路线图数据结构

> 旧 `DAGGenerator` 产出 `RoomNode`/`DungeonDAG`(房间链: `connections/parent/entryRoomId/bossRoomId`),偏"逐房推进",**不满足**节点路线图所需的"行集合 + 可达集 + 当前节点 + 已解历史"。故**新增** `NodeRouteMap`,不改造旧结构。

### 3.1 类型定义(`assets/scripts/dungeon/route/NodeRouteTypes.ts`)
```ts
/** 路由节点类型(本重做规范来源,与旧 RoomType 解耦,见 §6) */
export type RouteNodeType =
    | 'start'      // 层底入口,非战斗安全点
    | 'combat'     // 普通战斗
    | 'elite'      // 精英战斗(高风险高回报)
    | 'event'      // 2 选 1 变量事件
    | 'treasure'   // 宝箱/金币
    | 'shop'       // 商店
    | 'rest'       // 回血 或 强化 3 选 1(合并旧 Healing)
    | 'upgrade'    // 强化房(若与 rest 分开;多数层并入 rest)
    | 'boss';      // 层顶 Boss

/** 单个节点 */
export interface NodeRouteNode {
    id: string;            // 稳定 ID(用于 seed 派生,见 §10)
    type: RouteNodeType;
    row: number;           // 行号(0=层底入口行, maxRow=层顶 Boss 行)
    col: number;           // 同行内的列序号
    x: number;             // 地图 UI 显示坐标(派生,不参与逻辑)
    y: number;
    next: string[];        // 连向下一行节点的 ID(仅 r→r+1)
    state: RouteNodeState; // 见 §5 NodeRouteState
}

/** 一层节点路线图(规范来源) */
export interface NodeRouteMap {
    floorIndex: number;
    rows: number;                       // 总行数 = R_BASE + floorIndex [PLACEHOLDER]
    nodes: Record<string, NodeRouteNode>;
    edges: Array<{ from: string; to: string }>;
    startNodeId: string;
    bossNodeId: string;
    reachable: Set<string>;            // 当前可达节点 ID 集合
    currentNodeId: string;             // 玩家当前所在节点
    resolvedHistory: string[];         // 已结算节点 ID(按序)
    seed: number;                      // routeSeed(见 §10)
}
```

### 3.2 文件落位(新增模块)
| 文件 | 职责 |
|------|------|
| `assets/scripts/dungeon/route/NodeRouteTypes.ts` | 上述类型定义 |
| `assets/scripts/dungeon/route/NodeRouteGenerator.ts` | 纯逻辑生成 `NodeRouteMap`(消费 routeSeed) |
| `assets/scripts/dungeon/route/NodeRouteState.ts` | 节点/层运行状态(可达集、当前节点、历史、run stats) |
| `assets/scripts/dungeon/route/NodeRouteValidator.ts` | 生成后校验(可达/Boss 可达/无环/≥2 路径/无孤点) |
| `assets/scripts/dungeon/route/RouteRunController.ts` | **唯一**推进节点流程的状态机(见 §5) |
| `assets/scripts/dungeon/route/NodeRewardResolver.ts` | 节点统一结算(见 §9) |
| `assets/scripts/dungeon/route/RouteNodeTypeAdapter.ts` | `RouteNodeType` ↔ 旧 `RoomType` 映射(见 §6) |

> 旧 `assets/scripts/dungeon/DAGGenerator.ts` **冻结保留**,不被本系统读取;如有 legacy 房间链消费者(如旧 `GridManager` 种子)继续用,不强制迁移。
> **注意**:`DungeonGenerator.ts`(Demo5,**已存在**)是**房间建造器**而非地图生成器;本项目存在两套"地图"概念——`DAGGenerator`(legacy 房间链,冻结)、`DungeonGenerator`(3D 房间布局,已建)、`NodeRouteGenerator`(本重做,战略元地图)。三者职责不同:`NodeRouteMap` 管"选哪条路",`DungeonGenerator` 管"那条路上的 3D 房间内部"。路线模式启用后,游戏主地图以 `NodeRouteMap` 为准,`DAGGenerator` 退为 legacy 兜底。

## 4. 生成算法与验证规则

### 4.1 生成算法(`NodeRouteGenerator.generate(routeSeed, floorIndex, difficulty)`)
```
1. 行数 rows = R_BASE + floorIndex          [PLACEHOLDER] R_BASE=10, 上限 15
   理由: 每层比上层长 1 行,制造"越深越长"压迫感;>15 超 30min 单局。

2. 每行节点数
   - row 0(层底): 1 个固定 start 节点
   - row maxRow(层顶): 1 个固定 boss 节点
   - 中间行: 2~4 个节点(randInt,消费 nodeCountSeed)
   理由: 入口/Boss 唯一保证路径收束。

3. 生成边(仅 r→r+1)
   - 每个 row r 节点连到 row r+1 的 1~3 个节点
   - 约束 A: row r+1 每个节点至少被 1 个 row r 节点连接(无死点)
   - 约束 B: 禁止跨行连接(保证无环 DAG)
   - 约束 C: 至少 2 条从 start 到 boss 的不同路径(战略选择前提)

4. 节点类型分配(按 §11 配置表权重,消费 typeSeed)
   - start 行: 固定 start
   - boss 行: 固定 boss
   - 其余: 权重抽取,但保底:
       · 每层 ≥1 商店 / ≥1 休息(保命经济)
       · 每层 ≥1 事件(叙事/变量)
       · combat + elite 占比 ≥ 55%(保证战斗密度)
   约束: 避免连续 2 行单节点(失去选路、视觉疲劳)。

5. 坐标派生: x/y 由 row/col 派生(地图 UI 用,逻辑无关)。

6. 验证(NodeRouteValidator,失败重试 3 次,仍失败用预设模板):
   - ✅ 所有节点从 start BFS 可达
   - ✅ boss 节点可达
   - ✅ 无环、无孤点(0 入度且非 start)
   - ✅ ≥2 条不同路径
   - ✅ 保底节点类型齐全
```

### 4.2 "Broken" 定义(验收前必须识别)
- 任一层玩家 **0 选择路径**(全单节点链) → broken。
- Boss **不可达** → broken。
- 某非 start 节点 **0 入度** → 生成失败,触发重试。
- 连续 ≥2 行单节点 → 体验降级(告警,非硬 broken)。

## 5. RouteRunController 状态机

> 没有状态机,本系统极易出现:重复点击进两次场景、结算两次、返回错误节点、Boss 提前解锁。**所有节点流程只能通过 `RouteRunController` 推进**,UI/场景不直接改状态。

### 5.1 状态枚举
```ts
export type RouteRunPhase =
    | 'map_select'        // 地图选路中(可点可达节点)
    | 'node_entering'     // 已点节点,加载场景过场中(锁输入)
    | 'encounter_running' // 场景内执行战斗/交互
    | 'reward_pending'    // 场景完成,等待结算
    | 'node_resolved'     // 已结算,标记 cleared,回到 map_select
    | 'floor_cleared'     // 本层所有可达路径耗尽 / Boss 已解
    | 'run_settled';      // 整局结束(胜利或失败)
```

### 5.2 唯一推进入口(伪代码)
```ts
class RouteRunController {
    private phase: RouteRunPhase = 'map_select';

    /** UI 点击节点(仅 map_select + 节点在 reachable 内才生效) */
    selectNode(nodeId: string) {
        if (this.phase !== 'map_select') return;          // 防重复点击
        if (!this.state.reachable.has(nodeId)) return;    // 防越权
        this.phase = 'node_entering';
        const cfg = this.buildEncounterConfig(nodeId);    // 见 §8.3
        this.sceneLoader.enterEncounter(cfg, () => {
            this.phase = 'encounter_running';
        });
    }

    /** 场景通知:执行完成 */
    onEncounterComplete() {
        if (this.phase !== 'encounter_running') return;
        this.phase = 'reward_pending';
        this.rewardResolver.resolve(nodeId, ...);         // 见 §9,只发一次
        this.state.markResolved(nodeId);                  // 标记 cleared
        this.state.advanceReachable(nodeId);              // 解锁下一行可达
        this.phase = 'node_resolved';
        if (nodeId === bossNodeId) this.phase = 'floor_cleared';
        else this.phase = 'map_select';
    }
}
```
**防呆规则**: `node_entering`/`encounter_running`/`reward_pending` 期间忽略一切节点点击;Boss 节点仅在"前一行全部可达且玩家处于其上游路径"时进入 `reachable`,**禁止**因跳层而提前解锁。

### 5.3 与既有 `RoomFlowController` 的边界(防双控制器打架)
- `RouteRunController` = **节点间元流程**(选路→进入→结算→回地图→层清)。
- `assets/scripts/run/RoomFlowController.ts`(已存在)= **节点内微流程**(商店/宝箱/回血/区域 Boss/复活等房间事件处理)。
- **委托关系**: 进入 shop/rest/event/treasure 等非纯战斗节点时,`RouteRunController` 在 `encounter_running` 阶段**委托**既有 `RoomFlowController.onEnterShopRoom/onEnterTreasureRoom/onEnterHealingRoom` 等处理(这些已存在),节点自身不重建 UI;`onEncounterComplete` 仍由 `RouteRunController` 统一收口结算。两者不互相写状态,杜绝 Boss 提前解锁/重复结算。

## 6. 节点类型与旧 RoomType 映射

> 文档类型(`RouteNodeType`)与代码枚举(`RoomType`)**不直接混用**,经 adapter 单向映射,避免全项目改枚举。

### 6.1 映射表(`RouteNodeTypeAdapter.ts`)
| RouteNodeType | 旧 RoomType | 进入场景 | 备注 |
|---------------|-------------|----------|------|
| start | —(无对应,安全点) | 无(或安全营地) | 层入口,不触发战斗 |
| combat | `RoomType.Normal` | 战斗场景 | |
| elite | `RoomType.Elite` | 强化战斗场景 | |
| boss | `RoomType.Boss` | Boss 场景 | |
| treasure | `RoomType.Treasure` | 宝箱场景 | |
| shop | `RoomType.Shop` | 商店场景 | |
| event | `RoomType.Event` | 事件场景 | |
| rest | `RoomType.Healing` | 营地场景 | **合并旧 Healing→rest** |
| upgrade | `RoomType.Upgrade` | 强化场景 | 多数层并入 rest(配置控制) |

> 注意:项目内存在**三套类型系统**,adapter 负责桥接,禁止互相直接混用:
> 1. `RouteNodeType`(本重做 9 类,路由元层规范来源)
> 2. `DungeonGenerator.RoomType = 'start'|'battle'|'elite'|'reward'|'boss'`(Demo5,**已存在**,用于 3D 房间建造)
> 3. `Constants.RoomType = Normal|Elite|Boss|Treasure|Healing|Shop|Upgrade|Event|Rest`(旧,9 类,用于奖励/事件结算)
>
> **映射规则**:
> - combat/elite/boss/start → 同时映射 `DungeonGenerator.RoomType`(建 3D 房间)与 `Constants.RoomType`(结算奖励)。
> - treasure/shop/event/rest/upgrade → **不调 `DungeonGenerator`**(无 3D 战斗房间),改由 `RoomFlowController` 既有房间事件处理(走 `Constants.RoomType`)。
> - `Healing` 与 `Rest` 语义重复,路由层统一为 `rest`,adapter 将二者都映射到 `rest`(legacy 战斗侧 `RoomType.Healing` 仍可正常结算,不破坏旧逻辑)。

## 7. 节点地图 UI 规范

### 7.1 总体
- 地图 UI 为 **2D 覆盖层**(类 Slay the Spire 竖向节点图),叠加在场景之上或独立全屏面板。**不强制 3D**(信息 UI)。
- 方向直觉: 层底(起点)在下,层顶(Boss)在上,或反之,但须一致且标注。
- **地图层与战斗 HUD 层互斥**:进入 `encounter_running` 时地图面板销毁/隐藏,战斗 HUD 才显示;返回 `map_select` 时反之。杜绝两层重叠误触。

### 7.2 移动端规范(当前游戏主要横屏)
| 规范项 | 要求 | 理由 |
|--------|------|------|
| 节点点击热区 | ≥ 72×72 px(逻辑像素) | 触屏防误触 |
| 连线(边) | **不拦截触摸事件**(仅视觉) | 避免连线盖住节点/误触 |
| 当前可达节点 | **必须有动画**(脉冲/呼吸/箭头指引) | 明确"能去哪" |
| Boss 节点 | 常驻可见 **或** 提供快速定位按钮 | 目标感不丢失 |
| 滚动/缩放 | 行数 >10 可拖拽缩放,节点间距 ≥ 50px | 触屏友好 |
| 已通过节点 | 灰色勾选态,可回顾不可重入 | 进度反馈 |

### 7.3 节点状态样式

## 8. 节点执行层:先 2D 复用,后 3D 升级

### 8.1 执行层职责
每个 `RouteNodeType` 对应一个执行场景(战斗/事件/宝箱/商店/营地/Boss)。玩家在场景内执行,完成 → `RouteRunController.onEncounterComplete()` → 结算 → 回地图。

### 8.2 两种执行载体
| 载体 | 用途 | 状态 |
|------|------|------|
| **3D 俯视场景**(目标) | 正式玩法,复用 `../docs/2D转3D全面升级方案.md` v3(倾斜相机 35°–50°、`ICollisionService`、先 Grid 寻路) | 主线目标 |
| **2D fallback 场景**(开发期) | 3D 底座未完成时,复用现有 2D dungeon 战斗调试节点流程 | 仅开发兜底,不进正式包 |

> 切换不改动 `NodeRouteGenerator`/`RouteRunController`/地图 UI——只换 `sceneLoader` 注入的实现。

### 8.3 解耦契约 `RouteNodeEncounterConfig`(核心)
```ts
/** 地图层产出,执行层消费。双方唯一通信面。 */
export interface RouteNodeEncounterConfig {
    runId: string;
    floorIndex: number;
    nodeId: string;
    nodeType: RouteNodeType;     // 决定进哪个场景
    zoneId: string;              // 区域(决定怪物/Boss 池)
    difficulty: number;          // 难度系数
    seed: number;                // encounterSeed(见 §10),保证战斗可复现
    encounterSceneId: string;    // 场景资源 ID(3D 或 2D fallback)
    rewardProfileId: string;     // 奖励档案 ID(见 §9)
}
```
- `RouteRunController.buildEncounterConfig(nodeId)` 生成此对象 → 传给 `sceneLoader.enterEncounter(cfg)`。
- 执行层**只认 config**,不认识地图 UI;地图 UI**只产 config**,不认识 3D。

### 8.4 复用现有战斗(不重写)
- 战斗规则不变: 虚拟摇杆移动、近身攻击、受击闪白、震屏、弹道、`CombatEffectService`、元素反应、Boss HP 条。
- 现有 `assets/scripts/run/RoomFlowController.ts` 的 `onBattleVictory → RewardService.grantRoomClearRewards` 链路 **保留**,由 `NodeRewardResolver`(§9)在 `reward_pending` 阶段统一接管调用,确保奖励只发一次。
- 对于 shop/rest/event/treasure/upgrade 等非纯战斗节点,**不进 3D 战斗场景**:`RouteRunController` 在 `encounter_running` 阶段直接委托 `RoomFlowController` 既有 `onEnterShopRoom`/`onEnterTreasureRoom`/`onEnterHealingRoom` 等处理(这些已存在),节点自身不重建 UI;完成后仍由 `RouteRunController.onEncounterComplete` 统一收口 → `reward_pending` → 标记 cleared。确保"一个节点一种收口",避免双控制器写状态。
- 3D 化时按 2D转3D 方案 v3 将 `RuntimeLayerService`→3D 渲染组、`SpriteAnimationService`→骨骼动画、`ICollisionService` 接入 3D 重叠检测;事件总线(`attack:performed` 等)**复用**。

## 9. 奖励预览与结算

> 节点肉鸽必须统一"节点结算奖励",否则地图选路的价值不清晰(玩家不知道选这条路图什么)。

### 9.1 `NodeRewardResolver`(`assets/scripts/dungeon/route/NodeRewardResolver.ts`)
```
输入:  nodeType + floorIndex + difficulty + rewardSeed(§10)
输出:  rewardPreview   // 地图上显示的"奖励倾向"(图标类别,如:经验/装备/金币/强化)
       resolvedReward  // 实际发放(具体数值/物品,战斗/交互完成后确定)
```
- **地图阶段**: 显示 `rewardPreview`(如 combat=经验+装备倾向, treasure=金币倾向, rest=回血/强化倾向)。
- **结算阶段**(`reward_pending`): 用 `rewardSeed` 算 `resolvedReward` 并**只发放一次**(状态机保证),委托 `RewardService.grantRoomClearRewards`(按 `RouteNodeType→RoomType` 映射后的类型)。

### 9.2 奖励来源映射(复用现有系统)
| 节点 | 奖励 | 复用 |
|------|------|------|
| combat/elite | 经验 + 装备掉落 | 装备系统设计 |
| treasure | 金币/道具 | 道具系统 |
| shop | 购买 | 现有商店 |
| rest | 回血 或 强化 3 选 1 | 现有强化房 |
| event | 2 选 1 变量 | 现有事件房 |
| boss | 通关 + 进深层 | 现有 Boss 结算 |

## 10. 种子与存档恢复

### 10.1 种子 key 链(复现/每日挑战/录像回放基础)
```
runSeed      = 用户种子 / 每日挑战种子 / 随机种子
floorSeed    = hash(runSeed, floorIndex)
routeSeed    = hash(floorSeed, "route")        // 生成 NodeRouteMap
nodeSeed     = hash(floorSeed, nodeId)          // 节点内变化(如事件选项)
rewardSeed   = hash(nodeSeed, "reward")         // 节点结算奖励
encounterSeed= hash(nodeSeed, "encounter")      // 战斗布阵/怪物(写入 config.seed)

> **`hash(x, salt)` 定义(复用既有,不另造)**:`FNV-1a(x, salt)`,直接复用 `assets/scripts/replay/ReplayRecorder.ts` 的 `computeConfigHash`(已用 FNV-1a,无 `Math.random`,符合红线 5)。所有随机走 `core/rng/Rng`(xorshift32)或 `RunRng`,**禁止**在生成/战斗内调用无 key 的 `Math.random()`,否则破坏复现与回放。
```
> 所有随机必须走此链,**禁止**在生成/战斗内调用无 key 的 `Math.random()`,否则破坏复现与回放。

### 10.2 存档恢复(复用既有 `ISaveManager`,不新建存档层)
> **禁止**新建存档系统(违反 `_agent_contract.md` 禁止重复实现 + 与既有 `SaveManager` 重复)。`RouteRunController` 经 `ctx.get(ISaveManager)` 持久化。

- **扩展既有状态形状**(在 `assets/scripts/save/SaveManager.ts` 的 `RunState`/`DungeonState` **增加字段,不新增接口方法**,保持 §5.6 签名 1:1):
```ts
// RunState 扩展(在原 {version,seed,floor,items} 基础上加)
routeMap: SerializedRouteMap;       // NodeRouteMap(Set→数组序列化)
currentNodeId: string;
completedNodeIds: string[];
reachableNodeIds: string[];
phase: RouteRunPhase;
activeEncounter?: RouteNodeEncounterConfig;

// DungeonState 已含 {version,zone,clearedRooms,currentRoom}
// 路线模式复用 clearedRooms = completedNodeIds,currentRoom = currentNodeId(可选同步)
```
- **触发点**: 进层即 `saveRun`;节点结算(`node_resolved`)即 `saveRun`;进入场景(`node_entering`,含 `activeEncounter`+`phase`)即 `saveRun`。
- **恢复**: `loadRun()` → 重建 `RouteRunController`;若 `phase==='encounter_running'`,用 `activeEncounter` **重新进入同一场景**(同 `encounterSeed` → 同布阵,幂等)。
- 参考 `../docs/2D转3D全面升级方案.md` v3 §5 存档+崩溃恢复(既有的 `SaveManager`/`ReplayRecorder` 已落地该能力)。
