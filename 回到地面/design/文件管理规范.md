# 文件管理规范

> **版本**: v1.0 | **生效日期**: 2026-06-24
> **适用范围**: `E:\game\回到地面\` 及后续所有子目录
> **必读对象**: 所有参与本项目的设计、开发、美术人员

---

## 0. 设计原则

本规范遵循 5 个原则，每项规则都用它们来检验：

| 原则 | 含义 | 检查标准 |
|------|------|----------|
| **见名知意** | 看文件名就知道里面是什么 | 不看内容，5 秒内能判断是否要找的文件 |
| **唯一性** | 每个文件有且仅有一个正确位置 | 不出现"这文件该放哪"的困惑 |
| **一致性** | 同类文件用同种命名和结构 | 新人加入后能立即找到文件 |
| **可回溯** | 每个文件都有版本/日期/作者 | 能准确回答"这是什么时候的版本" |
| **低摩擦** | 创建和维护的开销要最小 | 不因为规范太复杂而绕过规范 |

- 只有**顶层目录**和**明确列出的二级目录**是预定义的

- 禁止在根目录直接堆放文件（唯一的例外是 `GDD_回到地面.md` 和 `README.md`）
- 空目录需包含 `.gitkeep` 文件以保留结构（供 Cocos Creator 识别）

## 2. 文件命名规则

### 2.1 通用规则

| 规则 | 说明 | 正确示例 | 错误示例 |
|------|------|----------|----------|
| 语言 | 统一使用简体中文 | `战斗系统设计.md` | `battle-system-design.md` |
| 分隔 | 使用下划线 `_` 或连字符 `-` | `角色_战士_设计.md` | `角色战士设计.md` |
| 长度 | 不超过 40 个字符 | `AP战斗系统数值框架_v1.xlsx` | `关于回到地面回合制游戏AP战斗系统的完整数值设计框架.xlsx` |
| 特殊字符 | 禁止使用 `\ / : * ? " < > \|` | `元素反应表.xlsx` | `元素/反应表.xlsx` |
| 空格 | 禁止包含空格 | `新手引导流程.md` | `新手引导流程 .md` |
| 大小写 | 保持标题式大小写（首字大写） | `GDD_回到地面.md` | `gdd_回到地面.md` |

### 2.2 文档类命名格式

```
[系统/模块]_[内容描述]_[版本].md
```

| 部分 | 规则 | 示例 |
|------|------|------|
| 系统/模块 | 所属系统名，首字母大写 | `AP战斗`、`元素反应`、`地牢生成` |
| 内容描述 | 精准概括，不超过 20 字 | `系统详设`、`数值框架`、`玩家流程` |
| 版本 | `v数字`，可选 | `v1`、`v0.5` |

**实际例子**：
- `GDD_回到地面_v1.md` ✅
- `AP战斗_系统详设_v0.5.md` ✅
- `元素反应_反应表.xlsx` ✅
- `回到地面设计稿.md` ❌（缺少模块前缀）
- `最终版最终版_战斗设计.md` ❌（不要"最终版"）

### 2.3 美术资源命名格式

```
[类型]_[名称]_[尺寸或用途]_[状态].[扩展名]
```

| 部分 | 取值 | 示例 |
|------|------|------|
| 类型 | `char`/`monster`/`ui`/`tile`/`fx`/`bg` | `char` |
| 名称 | 英文或拼音，小写 | `warrior`/`skeleton_fire` |
| 尺寸/用途 | 可选：`idle`/`walk`/`attack`/`64px` | `idle` |
| 状态 | 可选：`wip`(制作中)/`final`(终版) | `final` |

**实际例子**：
- `char_warrior_idle_final.png` ✅
- `monster_skeleton_fire_attack_wip.png` ✅
- `ui_btn_start.png` ✅
- `我画的超帅的剑士.png` ❌

### 2.4 数据文件命名格式

```
[类型]_[内容描述]_[日期/版本].[扩展名]
```

| 类型 | 说明 | 示例 |
|------|------|------|
| `balance` | 数值平衡表 | `balance_AP战斗数值_v1.xlsx` |
| `map` | 地图/关卡数据 | `map_第一层种子测试.json` |
| `config` | 配置 | `config_元素反应表.json` |
| `log` | 日志/记录 | `log_playtest_2026-06-24.csv` |

### 2.5 命名禁区

| 禁止模式 | 原因 | 替代方案 |
|----------|------|----------|
| 最终版、完结版、再也不改版 | 下一次修改会打脸 | 使用语义化版本号 |
| 副本、copy、备份 | 无法区分哪个最新 | `文件名.bak.日期` 临时备份，或使用 Git |
| 未命名、新建文件、New | 毫无描述性 | 花 10 秒起个好名字 |
| 张某某的稿子、我做的 | 多人协作看不懂 | 用系统/模块前缀 |
| v2、v3、v4（无上下文） | 不知道改了啥 | 建议用语义化版本或语义版本名 |

## 3. 文件创建流程

### 3.1 新文件创建标准流程

```
步骤 1: 确定文件类型
  ├── 设计文档 → design/ 下
  ├── 技术文档 → docs/ 下
  ├── 美术资源 → arts/ 下
  ├── 数值表   → design/数值/ 下
  └── 脚本工具 → tools/ 下

步骤 2: 按命名规则起名
  └── 核对: 是否符合"见名知意 + 唯一性 + 一致性"

步骤 3: 检查是否存在同类文件
  ├── 是追加内容 → 编辑现有文件 (不要新建)
  └── 是全新内容 → 创建新文件

步骤 4: 文件头模板
  └── 见 3.2

步骤 5: 如果是数据文件（xlsx/json/csv）
  └── 在对应的 目录索引.md 中注册新文件及其用途

步骤 6: 提交 Git（至少每天一次）
  └── commit message: [模块] 做了什么事
```

### 3.2 文件头模板

#### Markdown 文档

```markdown
# 标题

> **版本**: v0.1 | **作者**: [名字] | **日期**: YYYY-MM-DD
> **编码**: UTF-8
> **版权**: 本文件内容为独立原创

---
```

#### JSON 配置文件

```json
{
  "$schema": "optional_参考schema",
  "version": "0.1",
  "author": "名字",
  "date": "2026-06-24",
  "encoding": "UTF-8",
  "description": "本文件的用途说明"
}
```

#### 数值表格 (.xlsx)

首行预留一行元数据（隐藏或注释）：

| version | author | date | encoding |
|---------|--------|------|----------|
| v0.1 | 名字 | 2026-06-24 | UTF-8 |

### 3.3 "先查后建"原则

**在创建任何新文件之前，必须执行以下检查：**

```
查目录:  同类文件放在哪个目录？
查内容:  现有文件是否已包含我要写的内容？
查命名:  我的文件名是否和现有文件冲突？
```

如果发现已有文件包含相关内容 → 编辑该文件，不新建。
如果发现文件名冲突 → 确认是否真的不同内容，若是则调整命名。

## 4. 版本控制方案

### 4.1 Git 分支策略

```
main                      # 稳定版（仅从 develop 合并，且测试通过后）
  └── develop             # 开发主线（日常开发合并至此）
       ├── feature/xxx    # 功能分支（从 develop 拉出，开发完后合并回 develop）
       ├── fix/xxx        # 修复分支
       └── prototype/xxx  # 原型/实验分支（不会合并到 develop）
```

| 分支 | 用途 | 谁可以合并 | 是否可删除 |
|------|------|-----------|-----------|
| `main` | 只包含稳定版本，每个里程碑打 tag | 项目经理 | ❌ |
| `develop` | 日常开发汇总 | 所有开发者 | ❌ |
| `feature/*` | 单个功能开发 | 功能负责人 | ✅ 合并后删除 |
| `fix/*` | 单个问题修复 | 修复人 | ✅ 合并后删除 |
| `prototype/*` | 实验性原型 | 发起人 | ✅ 不再需要时删除 |

### 4.2 语义化版本号

格式：`v主版本.次版本.修订号`

| 层级 | 什么时候加 | 示例 |
|------|-----------|------|
| **主版本** | 重大重构、系统推翻重做 | v1.0.0 → v2.0.0 |
| **次版本** | 新增系统、重大功能添加 | v0.4.0 → v0.5.0 |
| **修订号** | 修改/修正/补充，不改变系统边界 | v0.5.0 → v0.5.1 |

**当前项目版本**: v0.5（设计阶段的次版本计数）

### 4.3 Commit Message 规范

```
[模块] 操作/变更的简要描述

可选: 详细说明（如果需要）
```

| 模块前缀 | 对应范围 |
|----------|----------|
| `[GDD]` | 主设计文档 |
| `[战斗]` | 战斗系统相关 |
| `[Build]` | 构筑/强化系统 |
| `[经济]` | 数值/经济/平衡 |
| `[广告]` | 广告变现 |
| `[地牢]` | 地牢生成/地图 |
| `[事件]` | 事件/随机内容 |
| `[美术]` | 美术资源 |
| `[工具]` | 辅助工具/脚本 |
| `[规范]` | 项目规范/约定 |

**示例**：
```
[战斗] 新增元素反应链式触发逻辑
[Build] 调整核心能力平衡：穿影 AP 消耗从 2 降为 1
[规范] 统一命名：暗影步→穿影（版权规避）
```

### 4.4 版本标记 (Tag)

- 每个里程碑（Phase 完成）在 `main` 分支打 tag
- tag 格式：`release/vX.Y.Z`
- 示例：`release/v0.5.0`（Phase 0 设计阶段完成）

## 5. 备份与恢复机制

### 5.1 备份层级

| 层级 | 频率 | 工具 | 内容 | 恢复方式 |
|------|------|------|------|----------|
| L0: 日常备份 | 每次修改前 | 手动 `.bak.日期` | 单个文件 | 删除 `.bak` 后缀 |
| L1: 本地 Git | 每次 commit | Git | 全部已跟踪文件 | `git checkout` |
| L2: 远程仓库 | 每天结束 | Git push | 全部已跟踪文件 | `git clone / pull` |
| L3: 云盘存档 | 每个 Phase 完成 | 手动打包 | 完整项目 + Git 历史 | 解压到本地 |

### 5.2 日常备份规则（L0）

修改现有文件前，使用以下命令：

```bash
# 方式 1（推荐——单个文件）
cp 源文件 源文件.bak.2026-06-24

# 方式 2（Git 方式——不需要手动备份）
git commit -m "提交当前版本再修改"
```

**什么时候必须做 L0**：
- 修改核心设计文档（GDD、方案确定记录）
- 修改数值表（balancing 文件）
- 不确定修改方向，需要留一个回退点

**不需要做 L0 的情况**：
- 创建新文件（Git 会自动追踪）
- 修改草稿/注释/个人笔记

### 5.3 Git 日常操作流程（L1）

```bash
# 开始工作前
git pull                      # 拉取最新
git checkout -b feature/xxx   # 新建功能分支

# 工作中的保存点
git add 文件
git commit -m "[模块] 描述"

# 完成功能后
git checkout develop
git merge feature/xxx         # 合并到开发主线
git branch -d feature/xxx     # 删除功能分支
```

### 5.4 远程仓库设置（L2）

**待实施**——当前尚无远程仓库。建议：

```bash
# 在 GitHub / Gitee / 其他平台创建空仓库后
git remote add origin 仓库地址
git push -u origin main
git push -u origin develop
```

后续每天结束时执行：

```bash
git push --all          # 推送全部分支
git push --tags         # 推送所有 tag
```

### 5.5 云盘存档规则（L3）

每个 Phase 完成后，打包整个项目（含 `.git` 目录）上传至云盘：

```bash
# Windows 打包命令
tar -czf 回到地面_phase0_2026-06-24.tar.gz 回到地面/
```

存档命名：`项目名_Phase名_日期.tar.gz`

存档后上传至团队共享的云盘/网盘目录。

### 5.6 恢复流程

| 场景 | 恢复方式 |
|------|----------|
| 文件修改后想撤回 | `git checkout -- 文件名` |
| 想找回删除的文件 | `git rev-list -n 1 HEAD -- 文件名` → `git checkout 哈希^ -- 文件名` |
| 整个分支搞砸了 | `git reset --hard 上一个正常commit的哈希` |
| 本地文件全部丢失 | `git clone 远程仓库地址` |
| 远程仓库也丢了 | 从云盘存档解压 |

## 7. 安全与权限

### 7.1 敏感信息隔离

| 数据类型 | 处理方式 | 禁止行为 |
|----------|----------|----------|
| API 密钥 / Token | 存储在环境变量或 `.env` 文件（已加入 `.gitignore`）| 硬编码到代码中 |
| 微信 AppID / AppSecret | 通过 Cocos Creator 构建参数注入 | 提交到 Git |
| 个人账号密码 | 永远不入库 | 写在任何文件中 |
| 内网地址 / 测试服务器 | 写在 `.gitignore` 保护的 `config.local` 中 | 提交到公开仓库 |
| 用户数据 / 日志 | 仅保留脱敏后的统计数据 | 包含真实用户信息 |

# 系统文件
