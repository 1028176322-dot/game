```

风格定位:      卡通像素风 (Cartoon Pixel Art) — 明亮色调、圆润造型、可爱化处理
              非暗黑风格，非写实风格，无恐怖/血腥元素

## 0.1 微信小游戏资源策略

> 微信小游戏不能只按“单张图尽量小”来做美术。正确目标是：首包足够小、首屏足够快、运行时资源足够清晰、分包按需加载。

### 0.1.1 包体与加载分层

| 层级 | 内容 | 资源策略 | 是否允许高清 |
|------|------|----------|--------------|
| 首包 | 启动屏、主界面最小资源、loading、必要字体、基础配置 | 极小化，只放首屏必需资源 | 谨慎，优先 1x/低清 |
| 基础分包 | 地牢基础 UI、通用 HUD、首个角色、首个区域怪物 | 默认清晰度，保证首次游玩体验 | 允许 mid 档 |
| 区域分包 | forest/catacombs/volcano/tundra/swamp/abyss 等区域资源 | 进入区域前加载 | 允许 mid/high 档 |
| Boss/活动分包 | Boss、特殊特效、活动图 | 用到前加载，用完释放 | 允许 high 档 |
| 非运行时源文件 | AI 母版、PSD、Aseprite、1024 原图 | 放 `art_source/` 或外部素材库 | 不进入构建 |

包体规则：

```text
首包目标：< 4MB，越小越好
分包总量预算：<= 30MB
单个运行时 PNG 不再一刀切要求极小，而是按资源类型设预算
AI 母版、raw 图、备份图、.bak 文件禁止放入 assets/resources
```

参考依据：Cocos Creator 3.8 微信小游戏分包文档说明，微信小游戏主包大小不能超过 4M，所有分包总大小不超过 30M；构建时可将 Asset Bundle 配置为小游戏分包，启动时只下载必要主包，后续由开发者手动加载分包。

关键修正：

- 首包限制不等于所有图片都只能 48px/64px。
- 主战斗角色、怪物、Boss 应放分包或按需资源里，不能为了首包把正式资源压糊。
- `resources` 目录会被 Cocos 作为动态资源参与构建，任何母版、备份、raw 图都应移出。
- 微信小游戏首屏只需要能进入 loading 和主界面，不需要把全部怪物、Boss、区域背景放进首包。

### 0.1.2 微信小游戏下的图片格式建议

| 资源类型 | 首选格式 | 说明 |
|----------|----------|------|
| 透明角色/怪物/Boss | PNG RGBA | 保留透明边缘，不用 JPG/TinyJPG |
| 像素风 Tile | PNG RGBA/RGB | 使用 nearest，不做有损降色 |
| UI 小件/按钮/图标 | PNG RGBA | 可无损压缩，尺寸按显示大小导出 |
| 大背景 | JPG/WebP/PNG 三选一 | 无透明背景优先 JPG/WebP；首包背景可降清 |
| AI 母版 | PNG/PSD/Aseprite | 不进入 `assets/resources` |

> 若目标微信基础库和 Cocos 构建链确认支持 WebP，可将大背景和插画类资源导出 WebP；透明战斗单位第一阶段仍建议 PNG，避免兼容和透明边缘问题。

### 0.1.3 资源目录治理

运行时目录只放游戏会加载的最终资源：

```text
assets/resources/textures/
```

非运行时文件必须移出：

```text
art_source/textures_master/      # AI 母版、1024 原图、PSD/Aseprite
art_source/textures_export/      # 导出中间产物
art_source/textures_backup/      # 替换前备份
art_source/textures_review/      # 前后对比图
```

禁止长期留在 `assets/resources/textures`：

```text
*_raw.png
*.bak.*
1024x1024 的图标母版
AI 原始大图
临时对比图
未压缩导出图
```

## 0.2 AI 出图与导出总规范

### 0.2.1 AI 不直接产最终小图

AI 生成时不要直接要求 48x48、64x64。小尺寸会让 AI 丢结构，后续再放大会糊。正确流程：

```text
AI 高清母版 -> 统一画布/透明边 -> 人工或工具修边 -> 导出运行时尺寸 -> 无损压缩 -> Cocos 验收
```

推荐母版尺寸：

| 资源类型 | AI 母版尺寸 | 运行时首包/低清 | 运行时默认 | 运行时高清 |
|----------|-------------|----------------|------------|------------|
| 小图标 | 512x512 | 96/128 | 256 | 512 |
| 技能/遗物图标 | 1024x1024 | 128 | 256 | 512 |
| 玩家角色单帧 | 512x512 起 | 128 | 192/256 | 384 |
| 普通怪物 | 512x512 起 | 96/128 | 192 | 256 |
| Boss 单帧 | 1024x1024 起 | 192 | 256/384 | 512 |
| 特效单帧 | 512x512 起 | 96/128 | 192/256 | 384 |
| 横向背景 | 2048px 宽起 | 750/1000 宽 | 1500 宽 | 2000 宽 |
| 竖屏背景 | 2048x4096 起 | 750x1334 | 1125x2001 | 1500x2668 |

### 0.2.2 Prompt 必填字段

每次生成必须把以下字段写清楚：

```text
asset_name: 文件名
asset_type: character / monster / boss / icon / ui / background / effect / tile
style: bright cartoon pixel art, rounded cute shapes, clean silhouette
canvas: 母版尺寸，例如 1024x1024
runtime_export: 最终导出尺寸，例如 256x256 或 每帧 192x192
background: transparent 或 no transparency
frame_count: 单帧或具体帧数
sheet_layout: vertical / horizontal / single
safe_margin: 8%-12% transparent padding
anchor: feet centered / center / bottom center
wechat_minigame: optimized for small package, no baked text, no watermark
forbidden: no text, no watermark, no signature, no logo, no gore, no realistic horror
```

### 0.2.3 可复制 Prompt 模板

角色/怪物/Boss：

```text
Create a bright cartoon pixel-art game sprite for WeChat Mini Game.
Asset name: {asset_name}.
Character type: {role_or_monster_description}.
Master canvas: 1024x1024, transparent background.
Runtime export target: each frame {runtime_frame_width}x{runtime_frame_height}.
Pose/action: {idle/attack/walk/hit/death/skill}, {frame_count} frames.
Sprite sheet layout: {vertical_or_horizontal}; keep same frame count and same frame canvas for every frame.
Style: cute rounded shapes, clean silhouette, readable at small mobile size, bright colors, no realistic horror.
Anchor: feet centered, consistent ground contact point across frames.
No text, no watermark, no signature, no writing of any kind.
No blood, no gore, no severed body parts; death is light particles or poof disappearance.
```

图标：

```text
Create a bright cartoon pixel-art icon for WeChat Mini Game.
Asset name: {asset_name}.
Master canvas: 1024x1024, transparent background.
Runtime export target: 256x256 PNG RGBA, also keep 512x512 high version outside resources.
Icon subject: {subject}.
Readable at 64px display size, strong silhouette, centered object, 8%-12% transparent safe margin.
No text, no watermark, no signature, no writing of any kind.
```

背景：

```text
Create a bright cartoon pixel-art background for WeChat Mini Game.
Asset name: {asset_name}.
Master canvas: at least 2048px wide.
Runtime export target: {runtime_width}x{runtime_height}.
No UI elements, no text, no logo, no watermark.
Composition must support mobile crop/cover; keep important subject inside center safe area.
Bright adventurous tone, no realistic horror, no gore.
```

### 0.2.4 导出与压缩顺序

```text
1. 保留高清母版到 art_source/textures_master
2. 从母版导出 runtime 目标尺寸
3. 检查透明边、锚点、帧数、排列方向
4. 使用无损 PNG 压缩
5. 必要时为背景导出 JPG/WebP
6. 同名覆盖运行时 PNG，保留 .meta
7. 在微信开发者工具和真机上验收
```

## A. 附录: AI 出图严格格式规范

> **每次使用 AI 生成资源时，必须将以下约束写入 prompt。** 否则 AI 默认输出 1024×1024 RGB 大图，会导致主包大小爆炸。

### A.1 通用约束（所有资源共用）

```
- 文件格式: 透明资源用 PNG RGBA, 无透明背景可用 JPG/WebP/PNG RGB
- 图片缩放: 像素风使用 NEAREST 最近邻插值并保持整数倍缩放
- 风格: 卡通像素风 (Cartoon Pixel Art), 明亮色调, 圆润造型
- 文字: 所有图片中不得包含任何文字 (no text, no English text, no Chinese text)
- 文件名: 全小写 + 下划线, 禁止中文/空格/大写字母
- 微信小游戏: AI 母版不得直接进入 assets/resources；运行时图必须按首包/分包预算导出
- 替换旧资源: 必须保留原文件名、原目录、原 .meta、原帧数、原 Sprite Sheet 排列方向
```

### A.2 各类资源尺寸与大小上限

### B.1 内容红线（一票否决）

| 红线 | 具体要求 |
|------|---------|
| **无血腥** | 不得出现红色血液/溅射/喷血效果，受击统一为闪白+火花 |

| **无断肢** | 不得有斩首/肢解/切割/穿刺等身体破坏表现 |
| **无敏感词** | 标语/描述中不得出现"血/杀/死/尸/亡/骷髅/恶魔/地狱" |

### B.2 命名规范（所有美术文件）

```
✅ 可用: 骨兵/炎魔/暗影/翠毒/冰晶/爆裂/枯木精/掘地行者
❌ 禁止: 骷髅/恶魔/死亡/亡灵/地狱/腐烂/吸血/自爆/食尸

✅ 可用: 击败/战败/驱散
❌ 禁止: 击杀/斩杀/死亡/处决/吸血
```

### B.3 图片文件技术规范

| 项目 | 要求 | 否则后果 |
|------|------|---------|
| **文件大小** | 首包从严，分包按资源类型预算；母版不得进运行时目录 | 首包膨胀、加载慢、可能无法上传/提审 |

## C. 附录: 文本渲染规范

### C.1 核心原则

```
PNG = 纯图形（壳）
Label 组件 = 文本（灵魂）

所有玩家可见的文字必须通过 Cocos Creator 的 Label 组件渲染，
禁止将文字直接画在 PNG 图片中。
```

### C.2 为什么不能把文字烤在 PNG 里

| 问题 | 烤图文字 | Label 组件渲染 |
|------|---------|---------------|
| **修改成本** | 重新出图 + 重新导入 | 改一行代码 |
| **本地化** | 每个语言重画一套 PNG | 改 text.json 配置 |
| **微信审核** | 改敏感词要重新出图 | 搜 T('ui.xxx') 改配置 |
| **字体统一** | 每张图字体可能不一致 | 统一使用 fonts/ 下的 TTF |
| **加载速度** | 每张都是独立大图 | 字体只需加载一次 |

### C.3 正确做法

```
编辑器里:
  Button (节点)
    ├── Background (SpriteFrame)  ← 纯图形 PNG, 无文字
    └── Label (子节点)            ← 代码控制文字内容

代码里:
  label.string = T('ui.startButton');  // 从 text.json 读取

配置里 (assets/resources/config/text.json):
  { "ui": { "startButton": "开始冒险" } }
```

### C.4 代码中设置文本的方式

```typescript
import { T } from '../core/TextManager';

// 静态文本 — 从 text.json 读取
this.titleLabel.string = T('ui.equipment');          // → "装备"

// 动态文本 — 模板变量替换
this.hpLabel.string = T('ui.hp', { cur: 80, max: 100 });  // → "生命: 80/100"
this.floorLabel.string = T('ui.floor', { floor: 5 });     // → "第 5 层"

// 所有文本集中在 text.json 中管理，方便审核检查和修改
```

### C.5 新增文本时的流程

```
1. 在 text.json 的对应分类下添加键值对
2. 在 UI 代码中用 T('分类.键名') 引用
3. 过一遍敏感词检查（血/杀/死/尸/亡/骷髅/恶魔/地狱）
4. 提交时 commit 写明: 新增了哪些文本键
```
