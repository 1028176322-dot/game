## 0. 目标平台约束：微信小游戏

**本项目所有产出——设计、美术、代码——均以微信小游戏为最终发布平台。** 以下约束对每一项决策都有效。

### 0.1 包体大小红线

| 指标 | 上限 | 说明 |
|------|------|------|
| 主包大小 | **< 4 MB** | 微信小游戏 4MB 红线，超出需分包（但分包有额外成本）|
| 首包大小目标 | **< 3 MB** | 留余量给必要库和基础资源 |
| 总包大小（含分包）| **< 8 MB** | 超过后可能导致审核/加载问题 |
| 单张图集 | < 512 KB | 压缩后，含多张小图 |

**对设计的约束**：
- 不依赖大量高清贴图——美术风格倾向于**像素**或**低面数矢量**
- 所有 UI 素材使用图集（Sprite Atlas）合并，单张图集 < 512 KB
- 动画使用骨骼动画或序列帧（每帧 < 20 KB），避免逐帧大图
- 音频使用低比特率（96kbps）MP3/Ogg，单文件 < 200 KB

### 0.2 性能红线

| 指标 | 目标 | 说明 |
|------|------|------|
| 帧率 | **60 FPS** | 战斗场景不能掉帧 |
| 同屏敌人上限 | **10 个** | 超过需做性能优化（LOD/简化AI）|
| 网格复杂度 | ≤ 8×8 | 目前 6×6 安全，预留扩展空间 |
| Draw Call | < 100 | 降低渲染批次 |
| 内存占用 | < 200 MB | iOS/低端安卓机限制 |
| 代码体积 | < 500 KB | 压缩后 |

**对设计的约束**：
- 不设计"满屏粒子特效"的技能——元素反应的表现用简单 Sprite 代替粒子
- 不设计大量同屏单位（超过 10 个敌人不出现）
- 地形效果使用 Tile 底色变化，不用独立高模

### 0.3 用户规模约束

微信小游戏的典型用户特征：

| 维度 | 特征 | 设计影响 |
|------|------|----------|
| 网络环境 | 4G/5G，**可能不稳定** | 广告加载失败需有 fallback |
| 设备性能 | 中低端安卓为主 | 特效不能太复杂 |
| 游戏时长 | **碎片化，5~15 分钟** | 单局时长锁定 |
| 付费意愿 | IAA 为主 | 无内购设计，纯广告变现 |
| 屏幕尺寸 | 全面屏/刘海屏 | UI 需适配安全区域 |
| 冷启动 | 从微信→小游戏 | 首包加载 < 5 秒 |

### 0.4 微信小游戏特有的规则

| 规则 | 对设计的影响 |
|------|-------------|
| **启动屏**：必须有符合规范的开屏图 | 预留设计空间 |
| **版权要求**：软著+自审报告 | Phase 5 预留时间 |
| **广告组件**：必须使用微信官方的广告组件 | 广告位对接微信广告（Banner/激励/插屏）|
| **更新机制**：审核通过后灰度→全量 | 大版本更新需预留灰度测试期 |
| **数据上报**：日志长度限制 1KB/条 | 埋点设计需精简字段 |

### 0.5 Cocos Creator 3.x 的微信小游戏约束

| 事项 | 说明 |
|------|------|
| 引擎版本 | Cocos Creator 3.x LTS |
| 构建目标 | WeChat Mini Game |
| 分包策略 | 将场景、预制体、音频分包 |

## 1. 文件编码规范

### 1.1 强制规定

**本仓库所有文件必须使用 UTF-8 编码。** 包括但不限于：

- `.md` 设计文档
- `.ts` / `.js` 源代码
- `.json` / `.yaml` 配置文件
- `.xlsx` 数值表格（导出时选择 UTF-8 CSV）
- `.psd` / `.png` 元数据字段
- 任何包含中文文本的文件

### 1.2 检查方式

```bash
# Git Bash / Linux
file --mime-encoding 文件名

# PowerShell
[System.Text.Encoding]::Default.GetString([System.IO.File]::ReadAllBytes("文件名"))
```

### 1.3 违反后果

非 UTF-8 文件在跨平台协作（Windows + macOS + 微信开发者工具）时会出现乱码，导致：
- 设计文档中文不可读
- Cocos Creator 中文字符串渲染为乱码
- 微信小游戏审核因乱码被拒绝

**每次提交前需检查新文件的编码。** 推荐在 `.gitattributes` 中设置：

```
*.md text working-tree-encoding=UTF-8
*.ts  text working-tree-encoding=UTF-8
*.json text working-tree-encoding=UTF-8
```

## 2. 版权风险规避指南

### 2.1 基本原则

| 原则 | 说明 |
|------|------|
| **机制不版权** | 游戏机制/玩法规则不受版权保护，但具体表达（代码、美术、文案）受保护 |
| **名称需通用** | 系统/怪物/技能名称必须使用通用幻想类型术语 |
| **实现不同路** | 即使功能相似，实现代码和表现方式必须独立原创 |
| **灵感≠抄袭** | 标注"灵感来源"而非"设计参考" |
| **美术零借鉴** | 不得直接复制/描图/改编任何现有游戏的美术资源 |

### 2.5 重点对比：提灯与地下城

> 本项目的类型和题材可能与《提灯与地下城》存在"地牢肉鸽"的类别重叠。以下是**核心差异点**的完整记录，确保在玩法设计层面有足够区分度。

| 维度 | 提灯与地下城 | 回到地面 (本项目) | 差异等级 |
|------|-------------|-----------------|----------|
| **战斗方式** | 实时战斗，自动攻击，玩家控制走位 | **AP回合制**，每回合3AP手动分配 | 🔴 根本差异 |
| **核心玩法** | 刷装备 + 宠物养成 | **战术决策 + 元素反应组合** | 🔴 根本差异 |
| **操作方式** | 竖版单手 | 横版网格战术 | 🟡 显著差异 |
| **战斗节奏** | 快速刷刷刷 | 慢节奏战略规划 | 🔴 根本差异 |
| **棋盘系统** | 自由移动，无网格 | **6×6网格，6种地形战术** | 🔴 根本差异 |
| **宠物系统** | **核心系统**：数百只契约兽，进化融合 | **不设宠物系统** | 🔴 完全不存在 |
| **亮度系统** | **标志性创新**：灯芯亮度影响难度/掉落 | **不设亮度系统** | 🔴 完全不存在 |
| **元素系统** | 水火地三系简单克制 | **六元素两两反应**（11种反应+链式） | 🔴 根本差异 |
| **怪物生成** | 预设怪物 | **程序化装配**（身体×元素×技能×变异） | 🟡 显著差异 |
| **怪物 AI** | 简单AI（自动战斗型） | **6种AI行为模式**（冲锋/远程/召唤/自爆等） | 🟡 显著差异 |
| **变现模式** | 内购+抽卡 | **纯广告 IAA** | 🟡 商业模型不同 |
| **核心乐趣** | 刷刷刷的多巴胺 | 发现combo的"啊哈时刻" | 🔴 根本差异 |
| **Roguelike类型** | 地牢Roguelike | 地牢Roguelike | ✅ 类型相同，类型不版权 |

#### 核心结论

两个游戏**相同的只有"地牢Roguelike这个大类型"**——而游戏类型本身不受版权保护。

**根本差异点**（4个）：
- 实时 vs 回合制——战斗逻辑完全不同
- 刷装备 vs 战术规划——核心乐趣不同
- 有宠物 vs 无宠物——核心系统不存在
- 亮度系统 vs 无亮度——标志性系统不存在

#### 后续避嫌措施

- [ ] 在宣传语中主动强调"回合制战术肉鸽"而非"地牢刷宝"
- [ ] UI布局避免竖版单手操作布局
- [ ] 不出现"提灯"、"亮度调节"、"契约兽"等要素
- [ ] 在项目说明中标注"灵感来源包括杀戮尖塔、陷阵之志等回合制战术游戏"
