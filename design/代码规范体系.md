# 代码规范体系 · 完整方案

> 版本: v1.0 | 日期: 2026-06-26
> 用途: 供 AI 按此规范生成项目代码，确保风格统一、结构清晰、质量可控
> 适用: 所有 TypeScript + Cocos Creator 项目文件

---

## 规范结构模板

每条规范使用统一的 Markdown 结构：

```
## [编号] [规范名称]

### 适用场景
什么场景下需要遵守此规范

### 格式要求
具体的代码格式规则

### 字段说明
如果涉及结构化字段，列出字段名、类型、含义

### ✅ 正确示例
符合规范的示例代码

### ❌ 错误示例
违反规范的示例代码

### 🤖 AI 生成指令
直接复制给 AI 的 prompt 片段，用于生成符合此规范的代码
```

## 一、命名规范

### N1 文件命名

**适用场景**: 所有 .ts / .json / .scene / .prefab 文件

**格式要求**:
- TypeScript 文件: `PascalCase.ts` (类文件) / `camelCase.ts` (工具函数)
- JSON 文件: `snake_case.json`
- Cocos 场景/预制体: `camelCase.scene` / `camelCase.prefab`

**字段说明**: 无结构化字段，纯文件名规则

**✅ 正确示例**:
```
PlayerController.ts       // 类文件 → PascalCase
mathUtils.ts              // 工具函数 → camelCase
battle_config.json        // JSON → snake_case
dungeonScene.scene        // 场景文件 → camelCase
skillSlot.prefab          // 预制体 → camelCase
```

**❌ 错误示例**:
```
player-controller.ts      // 禁止连字符
playercontroller.ts       // 禁止全小写
BattleConfig.json         // 禁止 PascalCase JSON
dungeon_scene.scene       // 禁止 snake_case 场景名
```

**🤖 AI 生成指令**:
```
生成 TypeScript 文件时：
- 如果该文件导出一个类 → 文件名使用 PascalCase
- 如果该文件导出一组工具函数/常量 → 文件名使用 camelCase
- JSON 配置文件 → snake_case
- Cocos Creator 场景/预制体 → camelCase
不允许使用连字符、不允许全小写无分隔、不允许混用大小写风格。
```

---

### N2 类/接口/类型命名

**适用场景**: TypeScript 中的 class / interface / type / enum 定义

**格式要求**:
| 范畴 | 风格 | 示例 |
|------|------|------|
| 类 class | PascalCase | `class PlayerController` |
| 接口 interface | PascalCase, I 前缀可选 | `interface RuntimeStats` |
| 类型 type | PascalCase | `type AttackResult` |
| 枚举 enum | PascalCase | `enum PlayerState` |
| 枚举值 | PascalCase | `Idle`, `Moving` |

**✅ 正确示例**:
```typescript
export interface AttackResult {
    target: MonsterController;
    damage: number;
    isCrit: boolean;
}

export enum PlayerState {
    Idle = 'idle',
    Moving = 'moving',
    Dodging = 'dodging',
}
```

**❌ 错误示例**:
```typescript
export interface attackResult { }    // 接口用 camelCase
export enum player_states { }       // 枚举用 snake_case
type attack_result = {};             // 类型用 snake_case
```

**🤖 AI 生成指令**:
```
所有类名、接口名、类型名、枚举名使用 PascalCase。
枚举值使用 PascalCase，其字符串值使用 camelCase 或 kebab-case。
不允许使用 snake_case 命名类/接口/类型/枚举。
```

---

### N3 变量/函数/方法命名

**适用场景**: TypeScript 中的变量、函数、方法、参数名

**格式要求**:
| 范畴 | 风格 | 示例 |
|------|------|------|
| 变量 | camelCase | `const playerHp: number` |
| 函数 | camelCase | `function getNearestMonster()` |
| 方法 | camelCase | `this._setState()` |
| 私有成员 | camelCase + `_` 前缀 | `private _timer: number` |
| 布尔值 | 动词前缀 | `isLoaded`, `hasKey`, `canAttack` |
| 常量 | UPPER_SNAKE_CASE | `const MAX_RETRY_COUNT = 3` |
| 事件名 | `domain:action` | `'attack:performed'` |

**✅ 正确示例**:
```typescript
private _timer: number = 0;
public isActive: boolean = false;
const MAX_RETRY_COUNT = 3;
eventBus.emit('game:phaseChanged', newPhase, oldPhase);

function calculateDamage(atk: number, def: number): number {
    return Math.max(GameConfig.MIN_DAMAGE, atk - def);
}
```

**❌ 错误示例**:
```typescript
private timer: number = 0;       // 私有成员无 _ 前缀
public is_active: boolean = false; // 布尔值用 snake_case
const maxRetryCount = 3;          // 常量用 camelCase
eventBus.emit('gamePhaseChanged'); // 事件不用 domain:action 格式
```

**🤖 AI 生成指令**:
```
变量/函数/方法使用 camelCase。
私有成员变量/方法使用 _ 前缀 (camelCase)。
布尔值以 is/has/can/should 开头。
常量使用 UPPER_SNAKE_CASE。
事件名字符串使用 'domain:action' 格式 (冒号分隔)。
不允许 snake_case(除非 JSON 字段), 不允许匈牙利命名法。
```

### N4 JSON 配置表命名

**适用场景**: `assets/resources/config/` 下的 JSON 文件

**格式要求**:
| 层级 | 规则 | 示例 |
|------|------|------|
| 文件名 | `snake_case.json` | `battle.json`, `monsters.json` |
| 表名(顶层键) | 小写复数 | `weapons`, `monsters`, `zones` |
| 行键(数据 ID) | 小写单数 | `sword`, `slime`, `forestGuardian` |
| 字段 | 小驼峰 | `atk`, `maxHP`, `attackRange` |
| 注释 | `$comment` 字段 | `"$comment": "范围: 0.2~5.0"` |

**✅ 正确示例**:
```json
{
    "weapons": {
        "sword": {
            "name": "铁剑",
            "atk": 10,
            "range": 1.2,
            "$comment": "atk: 1~50 | range: 0.5~5"
        }
    }
}
```

**🤖 AI 生成指令**:
```
JSON 键名规则：
- 顶层表名 → 小写复数 (weapons, monsters)
- 数据 ID → 小写单数 (sword, slime)
- 字段 → 小驼峰 (maxHP, attackRange)
- 注释 → $comment 字段，格式为 "中文描述 | 范围: 最小值~最大值"
不允许 PascalCase 或 snake_case 的字段名。
```

## 二、注释规范

### C1 文件头注释

**适用场景**: 每个 .ts 文件的开头

**格式要求**:
```
/**
 * 文件名 - 简短描述
 * 
 * 职责: 该文件负责什么
 * 使用方式: 外部怎么调用
 * 依赖: 依赖的其他模块
 */
```

**✅ 正确示例**:
```typescript
/**
 * PlayerStats - 运行时属性叠加层
 * 
 * 职责: 统一管理玩家最终属性的计算和修饰符叠加
 * 使用方式: PlayerController 通过 stats.getFinalStats() 获取最终属性
 * 依赖: GameConfig (属性边界), EventBus (属性变更通知)
 */
```

**❌ 错误示例**:
```typescript
// PlayerStats.ts
// 管理玩家属性
```

**🤖 AI 生成指令**:
```
每个文件开头必须添加 JSDoc 注释，包含：
1. 文件名 + 短描述
2. 职责 (2~3 行说明)
3. 使用方式 (外部如何调用)
4. 依赖 (依赖的其他模块/文件)
单行 // 注释不够，必须使用 /** */。
```

---

### C2 函数/方法注释

**适用场景**: 所有公开方法 + 复杂私有方法

**格式要求**:
```typescript
/**
 * 做什么 (动词开头)
 * @param paramName - 参数说明 (类型+含义)
 * @returns 返回值说明
 */
```

**✅ 正确示例**:
```typescript
/**
 * 执行自动攻击（使用最终属性）
 * @param stats - 玩家的最终运行时属性 (含 ATK/暴击/射程等)
 * @returns AttackResult 包含目标/伤害/是否暴击/是否击杀
 */
private _performAutoAttack(stats: RuntimeStats): AttackResult { }
```

**❌ 错误示例**:
```typescript
// 攻击
function attack() { }
```

**🤖 AI 生成指令**:
```
公开方法和复杂私有方法必须加 JSDoc。
格式: 动词开头说明做什么 → @param 参数名 - 说明 → @returns 说明。
如果方法没有参数或返回值，省略对应的 @param/@returns。
不写多余的 @description 或 @deprecated(除非真的废弃了)。
```

---

### C3 配置项注释

**适用场景**: GameConfig.ts / JSON 配置表的每个字段

**格式要求**:
```typescript
/** 用途描述 | 范围: 最小值~最大值 | 默认: 值 */
```

**✅ 正确示例**:
```typescript
/** 自动攻击间隔（秒）| 范围: 0.2~5.0 | 默认: 1.0 */
AUTO_ATTACK_INTERVAL: 1.0,

// JSON 版本:
"baseInterval": 1.0,
"$comment": "baseInterval: 自动攻击基础间隔(秒), 范围0.2~5.0"
```

**❌ 错误示例**:
```typescript
AUTO_ATTACK_INTERVAL: 1.0,   // 攻击间隔
```

**🤖 AI 生成指令**:
```
GameConfig.ts 的每个配置项上方加 JSDoc 注释。
格式: "中文描述 | 范围: 最小值~最大值 | 默认: 当前值"
JSON 配置表的每个字段在相邻位置加 $comment 字段。
不允许无注释的数值配置项。
```

### C4 TODO / FIXME / HACK 标记

**适用场景**: 需要后续处理但暂时不阻塞的地方

**格式要求**:
```typescript
// TODO: [谁] 需要做什么 - 原因 (日期)
// FIXME: [谁] 已知问题描述 - 影响范围 (日期)
// HACK: [谁] 临时方案描述 - 为什么这么写 (日期)
```

**✅ 正确示例**:
```typescript
// TODO: [designer] 此处需要从 zones.json 读取房间数量 - 当前硬编码 (2026-06-25)
// FIXME: [engineer] 此处数组下标越界风险 - 影响: 怪物死亡后切换目标 (2026-06-25)
// HACK: [designer] 因事件系统未完成暂时直接给奖励 - 上线前需替换为真实逻辑 (2026-06-25)
```

**🤖 AI 生成指令**:
```
标记格式: TAG: [责任人] 问题描述 - 附加上下文 (日期)
TAG 只能是 TODO / FIXME / HACK 三种：
- TODO: 以后要做但不阻塞当前
- FIXME: 已知 Bug 但不阻塞
- HACK: 临时方案，需要替换
日期格式 YYYY-MM-DD。
```

## 三、接口规范

### I1 模块导出规则

**适用场景**: 每个 .ts 文件的 export 定义

**格式要求**:
| 导出方式 | 适用场景 |
|----------|----------|
| `export class` | 需要实例化的类 |
| `export function` | 工具函数 |
| `export const` | 常量 |
| `export enum` | 枚举 |
| `export interface` | 接口/类型 |

**✅ 正确示例**:
```typescript
// core/Constants.ts — 只导出枚举/常量/类型
export enum GamePhase { ... }
export const BATTLE_CONSTANTS = { ... } as const;

// battle/PlayerController.ts — 只导出类
export class PlayerController extends Component { ... }
```

**❌ 错误示例**:
```typescript
// 一个文件同时导出类和一堆工具函数
export class BattleManager { }
export function helperFunc() { }
export const SOME_CONST = 1;

// 应该是: 工具函数放到 utils/, 常量放到 Constants.ts
```

**🤖 AI 生成指令**:
```
每个文件职责单一：
- 类文件 → 只 export class，工具函数放 utils/
- 常量/枚举/类型 → 放 Constants.ts 一起 export
- 配置 → 放 GameConfig.ts 一起 export
不允许多个不相关的 export 在同一个文件中。
例外: 紧密相关的 interface + 实现它的 class 可以放在同一文件。
```

---

### I2 接口定义规范

**适用场景**: interface / type 定义

**格式要求**:
- 字段必须有 JSDoc 注释 (中文)
- 可选字段用 `?` 标记
- 只读字段用 `readonly` 标记
- 联合类型用 `|` 穷举所有可能值
- 避免 any，用泛型或 Record 替代

**✅ 正确示例**:
```typescript
/** 运行时属性接口 */
export interface RuntimeStats {
    /** 攻击力 | >=0 */
    atk: number;
    /** 暴击率 (0-1) */
    critChance: number;
    /** 是否启用 */
    isActive: boolean;
    /** 攻击时附着的元素类型 (默认 Physical) */
    attackElement?: ElementType;
    /** 只读标识 */
    readonly id: string;
}

/** 联合类型穷举 */
export type ModifierType = 'flat' | 'percent';
```

**❌ 错误示例**:
```typescript
export interface Data {
    a: any;       // 禁止 any
    b: string;    // 无注释
    c: number | null | undefined | string | boolean;  // 过度联合
}
```

**🤖 AI 生成指令**:
```
接口/类型定义规范：
1. 每个字段必须有中文 JSDoc 注释
2. 可选字段加 ? 标记
3. 只读字段加 readonly 标记 (数据对象/配置对象)
4. 联合类型穷举所有合法值，不用 string
5. 禁止 any，用 unknown 或 Record<string, T> 或泛型替代
6. 回调函数类型用 type 而非 interface
```

## 四、错误处理规范

### E1 配置缺失兜底

**适用场景**: 所有配置读取 (ConfigManager / GameConfig)

**格式要求**:
```typescript
// 三层兜底策略:
// 1. 运行时值 (PlayerStats 等)
// 2. GameConfig 默认值 (GameConfig.ts)
// 3. ConfigManager 兜底值 (|| defaultValue)
const config = this._configs.get(name) || {};  // 配置表缺失不崩
return config[key] !== undefined ? config[key] : defaultValue;  // 配置项缺失用默认值
```

**✅ 正确示例**:
```typescript
getWeaponConfig(weaponId: string): any {
    return this.get('weapons', weaponId, {
        name: '拳脚',
        atk: GameConfig.PLAYER_BASE_ATK,
        range: 1,
        speed: 1.0,
    });
}
```

**🤖 AI 生成指令**:
```
所有配置读取必须使用三层兜底：
1. this._configs.get(name) || {}，配置表不存在时返回空对象不崩溃
2. 配置项不存在时用 defaultValue
3. 确保 defaultValue 包含所有必需的子字段

不允许: config[key] 直接取值无兜底
不允许: 假设配置表一定不为空
```

---

### E2 空引用安全

**适用场景**: getComponent / find / 回调函数调用

**格式要求**:
```typescript
// 所有 getComponent / find 判空
const component = this.getComponent(SomeClass);
if (!component) return;  // 或 component?.method()

// 可选回调
callback?.({ success: true });

// 可选链访问
const value = obj?.nested?.property ?? defaultValue;
```

**✅ 正确示例**:
```typescript
// 在 update 中重新获取引用
if (!this._player) {
    this._player = this.getComponent(PlayerController);
    if (!this._player) return;  // 尚未就绪，跳过此帧
}
const result = this._player.stats.getFinalStats();

// 可选回调安全调用
callback?.({ success: true, rewarded: true });
```

**❌ 错误示例**:
```typescript
const hp = this._player.stats.hp;                // 假设 _player 不为空
callback({ success: true });                     // 假设 callback 不为空
this.node.getComponent(Comp).doSomething();       // 链式不判空
```

**🤖 AI 生成指令**:
```
以下场景必须判空或使用可选链：
1. getComponent 后 → if (!comp) 或 comp?.method()
2. 可选回调参数 → callback?.(result)
3. 深层对象访问 → obj?.nested?.value ?? default
4. update 中依赖的组件引用 → 每帧重新获取并判空

不允许假设引用不为空，不允许链式访问不判空。
```

---

### E3 异步回调生命周期安全

**适用场景**: 广告回调 / 定时器 / 网络请求

**格式要求**:
```typescript
// UI 关闭/场景切换后，异步回调必须可取消或跳过已销毁对象
if (!this.isValid) return;  // Cocos Creator 的节点有效性检查
```

**✅ 正确示例**:
```typescript
// 广告关闭回调中检查节点是否有效
ad.onClose((res: any) => {
    if (!this.isValid) return;  // 节点已销毁，跳过
    const rewarded = res?.isEnded ?? false;
    this.onAdComplete(rewarded);
});
```

**🤖 AI 生成指令**:
```
所有异步回调函数体内的第一行必须是节点有效性检查:
if (!this.isValid) return;

适用场景: 广告回调 / tween 完成回调 / setTimeout / 网络请求回调
不检查的后果: 场景切换后 -> 已销毁的节点收到回调 -> 控制台报错
```

---

### E4 网络异常 Fallback

**适用场景**: 微信广告 / 数据上报 / 远程资源加载

**格式要求**:
```typescript
// 网络异常时直接给奖励，不阻塞玩家
try {
    // ... 网络操作
} catch (err) {
    console.warn(`[module] 操作失败`, err);
    this._fallbackReward(callback);  // 直接给奖励
}

private _fallbackReward(callback?: AdCallback): void {
    callback?.({ success: true, rewarded: true });  // fallback
}
```

**✅ 正确示例**:
```typescript
playRewardedAd(placement: AdPlacement, callback?: AdCallback): void {
    try {
        const ad = wx.createRewardedVideoAd({ adUnitId });
        ad.show().catch(() => {
            ad.load().then(() => ad.show()).catch(() => {
                this._fallbackReward(callback);  // 加载+重试失败 → fallback
            });
        });
    } catch (err) {
        this._fallbackReward(callback);  // 异常 → fallback
    }
}
```

**🤖 AI 生成指令**:
```
所有网络操作（广告/上报/远程加载）必须：
1. 用 try-catch 包裹
2. catch 中执行 fallback 逻辑（给奖励/缓存/跳过）
3. fallback 不阻塞玩家正常游戏流程
4. 控制台 warn 记录失败原因

不允许: 网络失败时卡住界面
不允许: 网络失败时玩家损失奖励
```

---

### E5 断言与防复发

**适用场景**: 修复 Bug 时添加防复发检查

**格式要求**:
```typescript
// 每次修 Bug，补至少一个防复发机制
// 断言: 在可能再次出问题的地方加检查
// 校验: 在边界条件处拦截

// 例: 修复数组越界 → 加下标检查
if (index >= 0 && index < this._modifiers.length) {
    this._modifiers[index] = fullMod;
}

// 例: 修复重复点击 → 加状态锁
if (this._isProcessing) return;
this._isProcessing = true;
```

**✅ 正确示例**:
```typescript
// 修复: 同一 source 不重复叠加
const existingIdx = this._modifiers.findIndex(m => m.source === mod.source);
if (existingIdx >= 0) {
    this._modifiers[existingIdx] = fullMod;  // 替换旧值
} else {
    this._modifiers.push(fullMod);           // 新增
}
```

**🤖 AI 生成指令**:
```
每次修复 Bug 时，在修改处添加至少一个防复发机制：
1. 数组操作加下标边界检查
2. 数值操作加 min/max 钳制
3. 重复操作加状态锁 (isProcessing)
4. 配置读取加兜底值
5. 资源访问加判空
防复发机制写在修复代码的同一处或紧邻的位置。
```

## 五、状态管理规范

### S1 状态集中定义

**适用场景**: 所有 enum 状态定义

**格式要求**: 集中写在 Constants.ts 中，禁止散落在各个类里

**✅ 正确示例**:
```typescript
// 所有状态集中在 Constants.ts
export enum PlayerState {
    Idle = 'idle',
    Moving = 'moving',
    Dodging = 'dodging',
    Attacking = 'attacking',
    Dead = 'dead',
}

export enum MonsterState {
    Idle = 'idle',
    Chase = 'chase',
    Attack = 'attack',
    Defend = 'defend',
    Dead = 'dead',
}
```

**❌ 错误示例**:
```typescript
// 在 PlayerController.ts 里写:
const STATE_IDLE = 'idle';

```

### S2 统一状态变更入口

**适用场景**: 状态机使用统一方法切换状态

**格式要求**:
```typescript
private _setState(newState: PlayerState): void {
    if (!this._isValidTransition(this._state, newState)) {
        console.warn(`非法状态跳转: ${this._state} → ${newState}`);
        return;
    }
    const oldState = this._state;
    this._state = newState;
    this._onStateChanged(newState, oldState);
}
```

**✅ 正确示例**:
```typescript
// 统一方法，禁止散落赋值
this._setState(PlayerState.Moving);  // ✅
// this._state = PlayerState.Moving; // ❌ 禁止直接赋值
```

**🤖 AI 生成指令**:
```
状态切换必须走统一的 setState 方法，禁止直接对状态字段赋值。
setState 方法内部必须：
1. 校验状态转移是否合法 (_isValidTransition)
2. 记录旧状态
3. 赋值新状态
4. 触发状态变更回调/事件
```

## 六、分层职责规范

### L1 三层架构

**适用场景**: 所有 .ts 文件的代码组织

**格式要求**:
```
业务逻辑层 (battle/ dungeon/ core/):
  - 玩法规则、状态流转、数值结算
  - 禁止: 直接依赖 UI 节点名/资源路径/Prefab 层级

UI 层 (ui/):
  - 展示、输入、动画、转发操作意图
  - 禁止: UI 点击直接修改战斗/背包/魂石等核心数据

数据配置层 (core/GameConfig.ts, resources/config/):
  - 配置表有默认值/类型/范围/枚举说明
  - 禁止: 配置缺失不兜底
```

**✅ 正确示例**:
```typescript
// UI 层: 点击按钮 → 发送事件 → 业务层处理
// BattleHUD.ts (UI层)
btnShield.onClick(() => {
    eventBus.emit('skill:use', 'shield');  // ✅ 只发事件，不调用业务逻辑
});

// PlayerController.ts (业务层)  
eventBus.on('skill:use', (skillId: string) => {
    this._useSkill(skillId);  // ✅ 业务层处理逻辑
});
```

**❌ 错误示例**:
```typescript
// UI 层直接修改数据
btnShield.onClick(() => {
    BattleManager.getInstance().useSkill('shield');  // ❌ UI 层直接调用业务层
    playerStats.applyModifier(...);                  // ❌ UI 层修改战斗数据
});
```

**🤖 AI 生成指令**:
```
严格遵守三层职责分离：
- 业务逻辑层 (battle/ dungeon/ core/) → 玩法规则、状态流转、数值结算
- UI 层 (ui/) → 展示、输入，通过 eventBus 转发操作意图
- 数据配置层 (GameConfig.ts / config/*.json) → 配置读取、校验、兜底

UI 层不允许直接调用业务层方法或修改核心数据。
UI 层应该 emit 事件，业务层 on 事件处理逻辑。
```
