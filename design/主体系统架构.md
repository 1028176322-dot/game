# 回到地面 · 主体系统架构

> **版本**: v1.0 | **日期**: 2026-06-24 | **设计者**: GameDesigner
> **用途**: 本项目所有系统的完整地图。每个系统先在此确定存在性和位置，再逐步细化具体方案。
> **编码**: UTF-8
> **版权声明**: 所有系统/机制名称均为通用游戏术语

---

## 0. 系统总览

```
┌─────────────────────────────────────────────────────────────┐
│                       UX 层                                  │
│          新手引导 / 界面流程 / 玩家交互                         │
├─────────────────────────────────────────────────────────────┤
│                     变现层                                    │
│         广告系统 / 跑马灯 / 每日宝箱                            │
├─────────────────────────────────────────────────────────────┤
│                     元层                                      │
│        长线成长 / 经济系统 / 种子系统 / DDA                     │
├─────────────────────────────────────────────────────────────┤
│                    内容层                                     │
│   Build构筑 / 元素反应 / 词缀 / 事件 / 房间变异                  │
├─────────────────────────────────────────────────────────────┤
│                    核心层                                     │
│    战斗系统 / 地牢生成 / 资源系统 / 怪物系统 /　属性数值         │
└─────────────────────────────────────────────────────────────┘
```

**设计原则**：下层不依赖上层，上层依赖下层。
- 核心层 → 可以独立 playtest
- 内容层 → 在核心层之上增添多样性
- 元层 → 驱动长期留存
- 变现层 → 在完整循环之上做商业化

## 1. 核心层 (Core Layer) — 必须优先完成

这层是游戏的"骨架"，没有这层游戏不能玩。优先级最高。

### 1.1 战斗系统 (Combat System)

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 方案已确定 |
| **位置** | `design/玩法与平衡性设计.md → 第1章` |
| **核心机制** | AP 回合制 (3AP/回合)，6×6 网格，6 种地形 |
| **伤害模型** | `伤害 = max(ATK + d6 - DEF×0.5, 1)`，暴击 d6=6 时 ×1.5 |
| **回合流程** | 玩家分配 3AP → 怪物按 SPD AI 行动 → 回到玩家 |
| **行动清单** | 移动(1AP) / 普攻(1AP) / 技能(2AP) / 防御(1AP) / 道具(1AP) / 等待(0AP) |
| **关键边界** | AP可保留1点到下回合；移动和攻击可穿插进行 |
| **失败条件** | 玩家HP归零 → 进入觉悟战流程 |
| **依赖** | 无（独立设计，仅依赖属性数值）|

**待细化**：
- [ ] 每个地形的触发时机和判定逻辑
- [ ] 防御姿态的持续时间与叠层规则
- [ ] 玩家属性与成长的关系公式

### 1.2 地牢生成系统 (Dungeon Generation)

| 字段 | 内容 |
|------|------|
| **状态** | ⏳ 方案已确定，算法未详设 |
| **位置** | `docs/待创建 → 地牢生成算法.md` |
| **核心机制** | DAG 有向无环图，每层 5~7 个房间，2~3 分支汇聚至 Boss |
| **房间池** | 战斗(55%) / 宝箱(12%) / 回复(10%) / 商店(10%) / 强化(8%) / 事件(5%) |
| **层数结构** | 初始 3 层，预留扩展至 5 层 |
| **种子依赖** | 接受种子作为随机种子，确保可复现 |
| **关键边界** | 不存在死胡同房间；Boss 房一定可达；至少有一条路径经过强化房 |
| **依赖** | 无（独立算法，输出房间序列供后续系统消费）|

**待细化**：
- [ ] DAG 生成算法具体实现（随机回溯/拓扑排序/预置模板）
- [ ] 房间在网格上的物理布局
- [ ] 种子如何映射到房间序列
- [ ] 如何保证"至少一条路径经过强化房"

## 2. 内容层 (Content Layer) — 增添多样性

这层在核心层之上增添"每局都不一样"的内容。

### 2.1 Build 构筑系统 (Build System)

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 方案已确定 |
| **位置** | `design/玩法与平衡性设计.md → 第2章` |
| **层级结构** | 核心能力(A) / 属性强化(B) / 遗物(C) |
| **选择频率** | 每层 3 选 1，每局 3 次强化 |
| **已设计** | **12 种** A 能力（含 2 个被动触发绑定）/ **7 种** B 取舍 / **8 种** C 遗物 |
| **已定义流派** | **10 种**（已适配实时战斗）|
| **关键边界** | 核心能力必须**改变自动攻击行为**而非固定生效；取舍必须有痛感；遗物不能简单叠加 |
| **依赖** | 战斗系统（能力的实时效果）、主角属性|

**待细化**：
- [ ] 每种 A 能力的 **完整机制规格**（输入/输出/边界/失败状态）
- [ ] 流派之间的协同/克制关系表
- [ ] 强化房生成算法（选项池筛选逻辑，避免连续出现相同方向）

### 2.2 元素反应系统 (Element Reaction)

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 方案已确定 |
| **位置** | `design/组合性随机性系统设计.md → 第1章` |
| **元素** | Fire / Frost / Lightning / Poison / Shadow / Holy |
| **反应数** | 10 种两两反应 + 1 种三重奏 + 链式反应 |
| **反应触发** | 目标身上存在两种元素状态时自动触发 |
| **链式反应** | 反应效果本身带元素 → 可触发二次反应，无预设上限 |
| **关键边界** | 链式反应必须有深度限制（实战中 3 层合理）；不能触发死循环 |
| **依赖** | 战斗系统（元素是附加在攻击和技能上的）|

**待细化**：
- [ ] 每种元素的**持续时间和叠层规则**
- [ ] 链式反应的**深度计数器和终止条件**
- [ ] 元素状态的 UI 表现方案

## 3. 元层 (Meta Layer) — 驱动长期留存

### 3.1 长线成长系统 (Long-term Progression)

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 方案已确定 |
| **位置** | `design/玩法与平衡性设计.md → 第2.3节` |
| **核心货币** | 魂石（Soul Stone），每局结算获得 |
| **可解锁** | 4 角色 / 3 天赋 / 遗物池 / 永久升级 |

**待细化**：
- [ ] **魂石掉落增长曲线**（第1层=5、第3层=30……的数学公式）
- [ ] 各解锁项的**性价比排序**（哪个值得先解锁，避免玩家走弯路）
- [ ] 解锁的**前置条件**（是否需要先解锁角色A才能购买天赋B？）

### 3.4 动态难度系统 (DDA)

| 字段 | 内容 |
|------|------|
| **状态** | ⏳ 概念已确定，未规格化 |
| **位置** | `design/组合性随机性系统设计.md → 第8章` |
| **核心原则** | 防止卡死、防止反复受挫，但不能让玩家感觉"被系统操控" |
| **触发条件** | 连续死亡 / 低HP无回复 / 连续无反应 / Boss 前战力不足 / 同一怪连续死 |
| **软锁保护** | 商店特价药 / 无钥匙可开木箱 / Boss 前篝火 / 连续死5次送不朽石 |
| **关键边界** | DDA 不能让玩家觉得"我菜所以系统给我优待"或"我强所以系统惩罚我"|
| **依赖** | 战斗、经济、Build（DDA 调整这三者的参数）|

**待细化**：
- [ ] **DDA 参数计算公式**（连续死亡几次后，怪物 HP 下调多少%）
- [ ] **软锁保护的触发条件和持续时间**
- [ ] DDA 调整值对玩家是否可见（建议不可见）

## 4. 变现层 (Monetization Layer)

### 4.2 跑马灯系统 (Marquee)

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 概念已确定 |
| **位置** | `design/玩法与平衡性设计.md → 第5.3节` |
| **核心机制** | 战斗结束后 3 格进度条，每格满需激励广告点亮，满 3 格得 1 钥匙 |
| **重置规则** | 进入下一层时进度清零 |
| **关键边界** | 每个战斗房最多积累 1 格；进度不能累计到下一层 |
| **依赖** | 战斗系统（战斗结束事件触发进度）|

**待细化**：
- [ ] 跑马灯的**UI 交互设计**（何时弹出、如何选择、退出流程）
- [ ] 是否支持跳过（可以，但放弃该格奖励）
- [ ] 跑马灯与插屏广告的冲突处理（不能同时弹两个）

## 6. 系统依赖图

```
种子系统
  │
  ├─→ 地牢生成 ──→ [房间序列]
  │     │
  │     ├─→ 房间变异（挂载在楼层上）
  │     ├─→ 事件系统（事件房是房间的一种）
  │     └─→ 怪物系统（怪物装配并放置在战斗房中）
  │
  ├─→ 数值系统 ←── 资源系统
  │     │
  │     ├─→ 战斗系统 ──→ Build 系统 ──→ 元素反应 / 词缀系统
  │     │     │              │
  │     │     └── 跑马灯 ←── 经济系统 ←── DDA
  │     │
  │     └─→ 长线成长 ──→ 广告系统
  │
  └─→ 新手引导（依赖全部系统，但独立于运行时循环）
```

**拓扑排序（实现顺序）**：

```
Phase 1:  种子 → 地牢生成 → 数值系统 → 资源系统 → 战斗系统 → 怪物AI
Phase 2:  Build → 元素反应 → 词缀
Phase 3:  经济 → 长线成长 → DDA
Phase 4:  广告 → 跑马灯
Phase 5:  新手引导 → 完整循环测试
```
